状态：v1 设计母稿

## 1. 这套文档集解决什么问题

当仓库开发、维护、排障和知识交接越来越依赖 AI Agent 时，传统“Agent 直接读源码”的模式会同时暴露四个基础性问题：

1. 全量源码上下文昂贵，Token 消耗高，且随着仓库增长非线性恶化。
2. Agent 很难在第一次读取时快速定位真正相关的文件、模块、边界和风险点。
3. 普通 README、注释和 Wiki 不是面向模型的语义表示，既不稳定，也不具备强同步机制。
4. 文档与代码一旦分离，就会出现双真相源，反而降低 Agent 与人类的决策质量。

ANRSM 的目标不是再做一套普通文档系统，而是定义一种 **AI Native 的仓库语义镜像基础设施**：

- 源码保持唯一真相源；
- 机器优先的语义镜像层为 Agent 提供低 Token、高密度、可同步、可审计的上下文入口；
- 人类优先的知识层从同一语义基础派生，服务于架构理解、交接、排障和学习；
- 改动检测、漂移分析和 CI 门禁确保镜像层不会脱离代码演化。

## 4. v1 的最小一致结论

这套文档集对 v1 的结论是明确的：

- **唯一真相源** 必须是源码与仓库受管配置。
- **文件级语义镜像** 是 v1 的强制能力，而不是可选增强。
- **模块级镜像** 是 v1 推荐能力，用于任务路由和上下文聚合。
- **人类知识层** 在目标架构中成立，但不要求成为 v1 MVP 的阻塞项。
- **任何 Material Change** 都必须可被检测为镜像漂移，且必须具备门禁策略。
- **Agent 默认读取顺序** 必须是“任务 -> 索引 -> 模块镜像 -> 文件镜像 -> 按需展开源码”，而不是“默认直接吞源码”。
- **Prompt 不得直接替代结构化提取**。LLM 可参与摘要和知识编译，但必须锚定到结构化事实与可追溯来源。
- **镜像文档不是普通 prose**。它是受约束的、低冗余的、可验证的语义压缩工件。

## 5. 适用对象与推荐场景

ANRSM 最适合以下仓库形态：

- 中小型到中大型跨端仓库；
- 日常开发大量依赖 Agent；
- 代码、配置、测试、脚本和接口契约耦合较强；
- 人员流动高，交接成本高；
- 需要同时优化 Agent 开发效率和人类维护理解成本；
- 希望把“文档同步”从软约束升级为硬门禁。

## 6. 不应误解为

ANRSM 不是：

- 普通的代码注释生成器；
- README 聚合器；
- 任意文本索引系统；
- 单纯的代码搜索引擎；
- 只给人看的知识库；
- 允许镜像脱离源码独立演化的文档系统；
- 替代编译、测试、审查和回归验证的万能机制。

## 1. 文档目的

本文件定义 **AI Native Repo Semantic Mirror（ANRSM）** 的 v1 设计母稿。  
它的任务只有一个：在工程实现之前，先把系统的核心概念、目标、不变量、边界和落地原则压缩成一套可执行的共识。

本文件不追求覆盖所有未来能力，而是优先回答以下 P0 问题：

1. ANRSM 到底是什么，不是什么。
2. 它解决的根问题是什么。
3. 它的真相源、派生层、同步机制和门禁机制分别是什么。
4. v1 必须实现哪些能力，哪些能力可以后置。
5. 哪些设计原则一旦违反，系统就会在规模化后失效。

## 2. 问题定义

### 2.1 当前 Agent 开发模式的根问题

当仓库开发越来越依赖 Agent 时，默认让 Agent 直接读取源码，会触发四类成本：

1. **上下文成本过高**  
   仓库越大，直接吞源码越容易挤占上下文窗口，降低单次任务可容纳的信息密度。

2. **任务路由效率过低**  
   Agent 往往先读了很多不相关代码，之后才定位到真正需要修改的文件和调用链。

3. **认知面不稳定**  
   普通注释、README、Wiki、Issue 描述与代码常常不同步，Agent 无法判断它们是否仍然可信。

4. **知识交接成本高**  
   新工程师、Reviewer、维护者和运维人员需要重新建立对仓库的心智模型，且这个模型经常依赖口头知识。

### 2.2 传统文档思路为什么不足

传统文档体系通常有三种问题：

- 面向人类，而不是面向 Agent；
- 没有与源码的一一映射和强制同步关系；
- 没有“语义压缩”约束，导致要么过长，要么失真。

因此，问题不是“仓库里缺文档”，而是“缺少一个以源码为唯一真相源、面向机器消费、可增量维护、可强制同步的语义派生层”。

---

## 3. 系统定义

ANRSM 是一套 **以源码为唯一真相源的仓库语义镜像系统**。  
它通过结构化提取、语义压缩、漂移检测和 CI 门禁，为代码仓库生成并维护三类派生表示：

1. **文件级语义镜像（File Mirror）**  
   与源码文件一一映射，面向 Agent，低 Token，强约束，强同步。

2. **模块级语义镜像（Module Mirror）**  
   聚合多个文件镜像，表达模块责任、边界、依赖关系和变更影响。

3. **人类知识层（Knowledge Layer）**  
   面向人类的架构、功能、排障、交接文档，从同一语义基础派生。

在任何时刻，源码与受管配置是唯一真相源；镜像与知识层都只能作为派生工件存在。

---

## 4. 设计目标

### 4.1 一级目标

ANRSM 的一级目标必须同时满足：

- **G1. Token 经济性**  
  让 Agent 不必以“读源码全文”为默认入口。

- **G2. 路由效率**  
  让 Agent 能在任务开始阶段快速定位相关模块、文件、接口和风险点。

- **G3. 语义新鲜度**  
  让镜像文档与代码演化保持可验证同步，而不是事后补文档。

- **G4. 审计性**  
  让任何镜像工件都可追溯到来源路径、版本、指纹和生成器版本。

- **G5. 统一知识基座**  
  让 Agent 和人类不再分别依赖不同的、不一致的知识来源。

### 4.2 二级目标

- **G6. 增量更新优先**：默认使用变更集而非全量重扫。
- **G7. 语言可扩展**：语言适配器应可插拔。
- **G8. 策略可治理**：同步规则和门禁强度可配置。
- **G9. 回退安全**：当镜像不可信时，Agent 必须能有序回退到源码展开。
- **G10. 部署简单**：v1 优先支持仓库内落地，不依赖复杂外部服务。

## 5. 设计公理

以下公理是 ANRSM 的基础；违反其中任意一条，系统都会退化。

### 公理 A1：源码唯一真相源

源码、受管配置、接口契约、测试和必要脚本组成仓库真实行为的唯一来源。  
镜像和知识层不得成为独立真相源。

### 公理 A2：派生工件必须可追溯

每个镜像工件都必须声明来源路径、来源提交、语义指纹、生成时间、生成器版本和新鲜度状态。

### 公理 A3：默认读取最小充分上下文

Agent 必须先读取最便宜且足以支持当前决策的语义工件；只有在镜像不足以支撑决策时，才允许展开源码。

### 公理 A4：结构化事实优先于自由文本

系统必须先做结构化提取，再做语言摘要。  
纯 Prompt 生成且没有结构化锚点的镜像不可视为可信。

### 公理 A5：Material Change 必须可观测

只要源码发生对语义有影响的变更，系统就必须能将其识别为某一级别的漂移。

### 公理 A6：门禁必须针对漂移而不是针对“是否写了文档”

要求并不是“文档存在”，而是“文档对当前源码仍然有效”。

---

## 6. 核心不变量

v1 必须强制以下不变量：

1. 每个纳入范围的源码文件都必须有确定性的镜像路径映射。
2. 每个文件镜像都必须含有受约束的机器可读文档头。
3. 每个文件镜像都必须具有 Freshness State。
4. 每个 Material Change 都必须触发至少一个镜像工件进入 `stale` 或 `invalid`。
5. 任何 `blocking` 级漂移都必须可以阻塞合并。
6. Agent 在执行代码修改任务前，必须先进行镜像读取或显式说明为什么跳过。
7. 人类知识层必须从镜像或同一结构化事实派生，不能与镜像体系并行野生演化。

## 7. 系统形态

ANRSM 在逻辑上分为三层。

### 7.1 真相层

包括：

- 一方源码；
- 运行时关键配置；
- API/事件/Schema 契约；
- 数据库迁移；
- 关键测试；
- 发布/运维关键脚本。

### 7.2 机器语义镜像层

包括：

- 文件级镜像；
- 模块级镜像；
- 符号索引；
- 依赖图；
- 路由索引；
- 漂移报告。

这一层面向 Agent，要求：

- 低 Token；
- 可验证；
- 可增量更新；
- 结构稳定；
- 有严格字段定义。

### 7.3 人类知识层

包括：

- 架构总览；
- 功能边界说明；
- 排障 Runbook；
- 交接和 Onboarding 文档；
- 模块使用指南。

这一层面向人类，允许更强解释性，但不得脱离镜像层独立生长。

## 8. v1 的正式边界

### 8.1 v1 必须实现

以下能力属于 P0，必须落地：

1. 仓库范围定义与路径映射。
2. 文件级语义镜像生成。
3. 机器可读文档头与语义指纹。
4. 变更集驱动的增量更新。
5. Material Change 分类。
6. 漂移检测与报告。
7. pre-commit 或 CI 级门禁。
8. Agent 读取顺序与写回义务。
9. 至少一种主语言的 AST 级结构化提取。
10. 至少一种模块级聚合工件。

### 8.2 v1 推荐实现

以下能力属于 P1，建议在 v1 后半程完成：

1. 多语言适配器。
2. 模块级质量评分。
3. 低置信度人工复核队列。
4. 路由基准测试集。
5. 人类知识层自动编译。

### 8.3 v1 明确不要求

以下内容不应成为 v1 上线阻塞项：

1. 多仓库全局图；
2. 复杂 UI 平台；
3. 实时后台常驻服务；
4. 任意语言零成本支持；
5. 自动修复所有镜像质量问题；
6. 无需源码展开即可完成所有变更任务。

## 9. 为什么必须采用“结构化提取 -> 受约束摘要”的两段式架构

如果直接把源码丢给 LLM 生成文档，会出现三个系统性问题：

1. 摘要结果不可复现，且字段不稳定。
2. 无法做细粒度差异比较，因为没有结构化中间层。
3. 一旦模型幻觉或忽略细节，CI 很难识别。

因此，ANRSM 的推荐流水线是：

1. **结构化提取**  
   解析 AST、导入导出、符号、接口、路由、事件、状态对象、关键副作用。

2. **语义压缩**  
   将结构化事实压缩成规定格式的镜像正文。

3. **验证与写回**  
   计算语义指纹，更新头信息，生成索引，出具漂移报告。

这让 Prompt 的角色从“直接发明文档”变成“在结构化事实约束下进行表达压缩”。

## 10. v1 的参考架构

### 10.1 核心组件

1. **Scope Manager**  
   定义哪些目录、文件类型和工件在系统范围内。

2. **Language Adapter**  
   负责解析指定语言，提取 AST、符号、依赖、接口等。

3. **Semantic Extractor**  
   把语言层结果统一成中立语义模型。

4. **Fingerprint Engine**  
   计算源码指纹与语义指纹。

5. **Mirror Generator**  
   依据模板和压缩规则生成文件级/模块级镜像。

6. **Index Builder**  
   生成符号索引、依赖图、路由索引。

7. **Drift Analyzer**  
   对比新旧语义状态，判断是否发生 Material Change。

8. **Policy Gate**  
   根据门禁策略决定是否阻塞提交或合并。

9. **Knowledge Compiler**  
   从镜像和图谱派生人类知识层。

### 10.2 架构原则

- 文件系统优先；
- 增量计算优先；
- 文本工件可 diff；
- 状态可重建；
- 失败可回退；
- 规则可配置；
- 结果可审计。

## 11. 工作流

### 11.1 初始化工作流

初始化不是“生成文档”，而是“建立语义基线”。

步骤：

1. 确定仓库范围与排除规则。
2. 识别模块边界和主语言。
3. 对纳入范围的文件做结构化提取。
4. 生成初始文件镜像和索引。
5. 计算语义基线指纹。
6. 对低置信度工件做人工复核。
7. 启用门禁。

初始化完成后，仓库进入“可增量维护”状态。

### 11.2 增量更新工作流

代码发生变更后：

1. 用 `git diff` 获取候选变更集。
2. 对变更文件及其受影响邻域重新解析。
3. 重新计算语义指纹。
4. 对比旧状态与新状态。
5. 更新受影响镜像和索引。
6. 输出漂移报告。
7. 根据策略决定是否放行。

### 11.3 Agent 任务工作流

一个合规的 Agent 必须按以下顺序工作：

1. 读取任务描述；
2. 读取路由索引；
3. 读取相关模块镜像；
4. 读取相关文件镜像；
5. 只在必要时展开源码；
6. 修改代码与测试；
7. 更新镜像；
8. 通过门禁；
9. 输出包含漂移状态的提交说明。

---

## 12. v1 的质量属性

ANRSM v1 必须优先优化以下质量属性：

### 12.1 Fidelity（语义保真）

镜像必须保留完成工程决策所需的最小充分语义，而不是只保留粗略功能描述。

### 12.2 Freshness（新鲜度）

镜像的可信度取决于它与当前源码的同步关系，而不是它写得多漂亮。

### 12.3 Routeability（可路由性）

系统必须能把“任务文本”稳定映射到候选模块和文件集合。

### 12.4 Auditability（可审计性）

必须能回答“这个镜像由什么生成、基于哪个版本、为什么现在判定它失效”。

### 12.5 Compression Efficiency（压缩效率）

镜像必须显著低于源码 Token 体量，同时仍能支撑大部分定位和规划决策。

---

## 13. 设计中的硬边界

以下边界不可模糊：

1. **镜像不是源码替代物**  
   当决策依赖具体实现细节时，必须展开源码。

2. **镜像不是自由编辑文档**  
   机器可读头和受管段落不得被随意手改。

3. **人类知识层不是平行真相源**  
   它必须从镜像或同一结构化事实派生。

4. **CI 门禁不是格式检查**  
   其核心任务是阻止语义漂移进入主干。

5. **Prompt 不是解析器**  
   Prompt 可以压缩和解释，不应承担核心事实抽取职责。

---

## 14. 典型风险与控制措施

### 风险 R1：双真相源

表现：工程师习惯直接改镜像或人类文档，而不更新源码或结构化事实。  
控制：只允许编辑特定手工补充区域；镜像头和关键区块必须由生成器维护。

### 风险 R2：过度压缩导致误导

表现：镜像过短，只说“做什么”，不说关键约束、状态和副作用。  
控制：正文模板必须包含责任、输入输出、状态、副作用、风险、依赖和展开提示。

### 风险 R3：把非语义变更当成语义变更

表现：格式化、注释改动频繁触发阻塞。  
控制：语义指纹必须忽略注释、空白和纯格式变化；Material Change 分类要分级。

### 风险 R4：全量重扫导致系统不可用

表现：每次提交都重算整个仓库，导致本地和 CI 体验不可接受。  
控制：默认走差异驱动；只有在适配器升级、配置变更或图谱一致性失效时才做全量重建。

### 风险 R5：镜像内容不可审计

表现：看得到镜像，但无法解释其来源和置信度。  
控制：所有镜像必须写入来源、指纹、生成器版本、置信度和时间戳。

---

## 15. v1 的实施原则

1. **先文件级，再模块级，再知识层**  
   不要在没有文件级镜像和漂移门禁之前先做大而全知识平台。

2. **先单语言主链路，再多语言扩展**  
   v1 应先拿一个主语言和一个真实仓库打通闭环。

3. **先结构化事实，再自然语言表达**  
   结构化语义层是长期可维护性的关键。

4. **先差异驱动，再平台化**  
   没有增量更新能力的平台化通常只会放大成本。

5. **先门禁，再规模化依赖**  
   如果没有硬门禁，团队很快会重新回到文档漂移状态。

---

## 16. v1 的最低验收结论

如果一个实现要自称“符合 ANRSM v1”，它至少必须满足：

1. 对纳入范围的源码文件建立确定性镜像路径映射；
2. 生成包含标准头字段的文件级镜像；
3. 能区分非语义变更与 Material Change；
4. 能针对变更集做增量更新；
5. 能输出漂移报告；
6. 能在本地或 CI 阶段阻止未同步的 Material Change；
7. 能让 Agent 在任务中优先消费镜像；
8. 能解释每个镜像工件的来源和新鲜度。

---

## 17. 一句话结论

ANRSM 不是“给代码配一套文档”，而是 **把仓库增加一层可验证、可同步、面向 Agent 的语义表示层，并用门禁把这层表示固定为仓库治理基础设施的一部分**。


---

# 02. 规范语言与术语表

状态：Normative  
版本：v1  
优先级：P0

---

## 1. 目的

本文件定义 ANRSM 的规范语言、核心术语、命名规则、保留词和状态枚举。  
其目标是消除团队在设计、开发、评审和文档编写中的歧义，防止同一个词被多种含义复用。

本文件中的术语定义优先级高于其他文档中的自然语言表述。

---

## 2. 规范性关键词

下列关键词采用严格规范含义。

### MUST

绝对要求。违反后实现不可宣称符合本规范。

### MUST NOT

绝对禁止。违反后实现不可宣称符合本规范。

### SHOULD

强建议。只有在存在充分、书面化、可审计理由时才可偏离。

### SHOULD NOT

强烈不建议。偏离通常会导致系统复杂度上升、质量下降或后续治理困难。

### MAY

允许但非必需。是否实现取决于阶段目标、资源和上下文。

### REQUIRED

与 MUST 等价。

### OPTIONAL

与 MAY 等价。

---

## 3. 术语总表

### 3.1 真相源相关

#### Source of Truth（真相源）

指最终决定仓库行为的真实工件集合。  
在 ANRSM 中，真相源通常包括源码、关键配置、契约定义、迁移、关键测试和关键脚本。

规范要求：

- 真相源 MUST 是可执行行为或行为边界的直接来源。
- 派生工件 MUST NOT 被标记为真相源。

#### Derived Artifact（派生工件）

从真相源或其结构化提取结果生成的工件。  
包括文件镜像、模块镜像、索引、漂移报告和知识层文档。

#### Managed Configuration（受管配置）

会影响运行时行为、构建边界、接口契约或系统部署的配置文件。  
此类配置 SHOULD 被纳入镜像范围。

### 3.2 镜像相关

#### Mirror（镜像）

在 ANRSM 中，Mirror 专指 **机器优先、受约束、可追溯、可同步的语义压缩工件**。  
普通 README、设计说明或自由文本文档 MUST NOT 被称为 Mirror，除非它们满足本规范要求。

#### File Mirror（文件级镜像）

与单个源码文件一一映射的语义镜像工件。  
它是 v1 的核心工件类型。

#### Module Mirror（模块级镜像）

从一组文件镜像和依赖图聚合而成的镜像工件。  
它表达模块责任、接口边界、依赖关系和影响面。

#### Knowledge Document（知识文档）

面向人类的派生工件。  
它可以更解释性，但 MUST 保持来源可追溯，且 MUST NOT 与源码独立演化。

### 3.3 变化与漂移相关

#### Drift（漂移）

派生工件与当前真相源语义不一致的状态。  
漂移不是“文档不存在”，而是“文档已经不再可信”。

#### Drift Detection（漂移检测）

识别派生工件是否与真相源失配的过程。

#### Material Change（实质语义变更）

任何会改变以下任一维度的源码变更：

- 对外接口；
- 数据契约；
- 事件名或路由；
- 状态模型；
- 关键控制流；
- 副作用；
- 安全属性；
- 性能边界；
- 运维行为；
- 上下游依赖关系。

纯格式化、注释、空白和不影响语义的重命名 SHOULD NOT 被视为 Material Change，除非策略显式声明。

#### Non-material Change（非实质变更）

不会改变镜像所需语义表达的变更。  
例如：空白调整、注释优化、未外露的纯重排。

### 3.4 指纹与新鲜度相关

#### Source Fingerprint（源码指纹）

基于原始源码内容计算的指纹，用于识别文件级输入变化。

#### Semantic Fingerprint（语义指纹）

基于结构化语义内容计算的指纹，用于识别对镜像有效性的语义变化。  
语义指纹 SHOULD 排除注释、空白和与决策无关的格式差异。

#### Freshness State（新鲜度状态）

镜像当前可信程度的离散状态。  
v1 的保留值如下：

- `fresh`：与当前语义状态一致；
- `stale`：已知存在变更，但尚未验证或更新；
- `invalid`：镜像结构错误、字段缺失、解析失败或与当前真相源明显不一致；
- `unknown`：系统无法证明其新鲜度。

### 3.5 路由与消费相关

#### Routing Index（路由索引）

用于把任务描述映射到候选模块、文件、符号和依赖邻域的机器工件。

#### Expansion Read（展开读取）

在镜像不足以支持当前决策时，Agent 针对具体源码文件做的按需阅读。  
Expansion Read MUST 是“按需”的，而非默认全量源码读取。

#### Minimal Sufficient Context（最小充分上下文）

完成当前工程决策所需的最小上下文集合。  
ANRSM 的目标是让 Agent 默认消费最小充分上下文，而非最大可得上下文。

#### Task Slice（任务切片）

针对某个开发任务，经过路由与筛选后得到的最小相关工件集合。

### 3.6 质量与可用性相关

#### Fidelity（保真度）

镜像保留决策关键语义的程度。

#### Coverage（覆盖率）

纳入范围的真相源工件中，成功生成有效镜像的比例。

#### Routeability（可路由性）

系统将任务映射到正确相关工件集合的能力。

#### Auditability（可审计性）

系统解释“镜像来自哪里、何时生成、为何可信或失效”的能力。

#### Conformance（符合性）

实现满足本规范要求的程度。

---

## 4. 保留词与禁止混用

以下词在文档和代码中 SHOULD 使用固定含义，不得随意替换或混用：

- `mirror`：只指机器优先语义镜像，不指一般文档。
- `knowledge`：只指人类优先知识层，不指文件镜像。
- `source`：只指真相源工件，不指镜像。
- `freshness`：只指镜像新鲜度状态，不指“看起来像最新的”主观判断。
- `drift`：只指同步失配状态，不指普通改动。
- `material`：只指实质语义层面的影响，不指文件行数多少。
- `route`：只指任务到工件集合的映射，不指 HTTP Route，除非上下文显式限定。

## 5. 命名规范

### 5.1 工件类型保留值

`artifact_type` 的 v1 保留值：

- `file_mirror`
- `module_mirror`
- `knowledge_doc`
- `routing_index`
- `symbol_index`
- `dependency_index`
- `drift_report`

### 5.2 标识符规则

#### artifact_id

格式：

```text
<artifact_type>:<repo_relative_path_or_logical_id>
```

示例：

```text
file_mirror:src/auth/login.ts
module_mirror:module/auth
knowledge_doc:architecture/system-overview
```

规则：

- MUST 使用正斜杠 `/` 作为路径分隔符；
- MUST 以仓库相对路径或逻辑 ID 表示；
- MUST NOT 以 `./` 开头；
- MUST NOT 使用依赖操作系统的路径分隔符。

#### module_id

建议格式：

```text
<domain>[/<subdomain>]
```

示例：

```text
auth
billing/invoice
mobile/profile
```

要求：

- SHOULD 与目录边界、业务边界或交付边界保持一致；
- SHOULD 避免使用临时命名。

### 5.3 文件命名规则

#### 文件镜像

推荐映射规则：

```text
mirror/file/<source_relative_path>.md
```

示例：

```text
src/auth/login.ts -> mirror/file/src/auth/login.ts.md
apps/web/pages/home.tsx -> mirror/file/apps/web/pages/home.tsx.md
```

这样做的原因：

- 无需做扩展名折叠；
- 减少同名碰撞；
- 与源码路径形成确定性双射。

#### 模块镜像

推荐规则：

```text
mirror/module/<module_id>.md
```

#### 知识文档

推荐规则：

```text
knowledge/<category>/<logical_name>.md
```

---

## 6. 文档头字段命名规则

机器可读文档头 MUST 满足：

1. 字段名 MUST 使用 ASCII `snake_case`。
2. 枚举值 SHOULD 使用小写 ASCII。
3. 时间戳 MUST 使用 RFC 3339。
4. Hash 字段 MUST 显式带算法前缀，例如 `sha256:<hex>`。
5. 路径字段 MUST 使用仓库相对 POSIX 路径。
6. 语言本地化内容 MUST 位于正文，不应污染核心元字段命名。

---

## 7. 保留状态枚举

### 7.1 freshness_state

保留值：

- `fresh`
- `stale`
- `invalid`
- `unknown`

### 7.2 drift_level

保留值：

- `none`
- `low`
- `material`
- `blocking`

建议含义：

- `none`：无漂移；
- `low`：存在差异，但不阻塞；
- `material`：存在实质漂移，应更新镜像；
- `blocking`：根据策略必须阻塞提交或合并。

### 7.3 confidence_band

保留值：

- `high`
- `medium`
- `low`

数值置信度 MAY 与 band 并存，但 band 更适合策略使用。

---

## 8. 正文段落的规范名称

v1 文件镜像正文 SHOULD 使用以下规范章节名，且顺序 SHOULD 保持稳定：

1. `职责`
2. `对外契约`
3. `输入与前置条件`
4. `状态与控制流`
5. `副作用与外部依赖`
6. `关键不变量与风险`
7. `变更影响`
8. `何时必须展开源码`

允许在此基础上增减附加段落，但以上八段 SHOULD 作为默认骨架。

---

## 9. 禁止用语与风格约束

镜像正文 SHOULD NOT 使用以下表达方式：

- “大概”“可能”“似乎”之类无置信度标记的模糊词；
- 未指代清楚的代词，例如“它”“这里”“这个逻辑”；
- 行为主体不明确的句子；
- 大段复述源码细节；
- 修辞化、宣传化、口号化文字；
- 没有来源锚点的臆断结论。

若必须表达不确定性，MUST 通过显式字段或句式说明，例如：

- `confidence_band: low`
- “以下结论基于静态提取，未执行运行时验证”
- “该副作用通过命名和调用位置推断，需展开源码确认”

---

## 10. “镜像”与“摘要”的严格区别

在 ANRSM 中，**摘要** 只是表达形式，**镜像** 是受约束工件类型。

因此：

- 所有镜像都包含摘要；
- 但并非所有摘要都是镜像。

只有满足以下条件的摘要，才可视为镜像的一部分：

1. 有标准头；
2. 有来源和指纹；
3. 有新鲜度状态；
4. 有稳定段落结构；
5. 由受管流程生成或验证；
6. 可被漂移检测与门禁系统识别。

---

## 11. “理解仓库”与“定位文件”的区别

这两个目标不可混为一谈。

- **理解仓库**：建立系统级心智模型，需要模块镜像和知识层。
- **定位文件**：将任务映射到相关文件集合，需要路由索引和文件镜像。

v1 MUST 先保证“定位文件”能力，再逐步提升“理解仓库”能力。  
如果系统能生成大量 prose，却不能把任务稳定路由到文件集合，则不满足 v1 核心目标。

---

## 12. 术语使用检查清单

在评审和实现中，应检查：

1. 是否把普通 README 错称为 Mirror。
2. 是否把漂移等同于“没有更新文档”。
3. 是否把 Freshness 当作主观判断，而非状态机结果。
4. 是否把 Prompt 直接当成事实提取器。
5. 是否把知识层当成独立真相源。
6. 是否把 Material Change 误解为“改动很大”。

---

## 13. 一句话定义合集

为便于工程沟通，以下是推荐的一句话定义：

- **ANRSM**：以源码为唯一真相源的仓库语义镜像基础设施。
- **File Mirror**：与单个源码文件一一映射、面向 Agent 的低 Token 语义工件。
- **Module Mirror**：聚合多个文件镜像的模块语义工件。
- **Drift**：镜像与当前源码语义不一致的状态。
- **Material Change**：足以影响镜像有效性的源码变化。
- **Expansion Read**：为了完成具体决策而按需展开源码的行为。
- **Knowledge Document**：面向人类的派生知识工件，不是真相源。

# 03. 系统边界、假设与非目标

状态：Normative  
版本：v1  
优先级：P0

---

## 1. 目的

本文件用来回答一个经常被忽略但会直接决定项目成败的问题：  
**ANRSM 到底覆盖什么，不覆盖什么；它依赖哪些前提；哪些目标不应被 v1 误背。**

边界不清会导致两类失败：

1. 目标膨胀，v1 过载；
2. 责任模糊，系统无法形成硬约束。

---

## 2. 范围边界

### 2.1 v1 的纳入范围

ANRSM v1 SHOULD 优先覆盖以下工件：

1. **一方业务源码**  
   前端、后端、移动端、共享库等核心业务代码。

2. **关键契约工件**  
   API 定义、Schema、事件定义、类型边界、协议声明。

3. **关键运行时配置**  
   影响行为、接线、部署或安全边界的配置。

4. **数据库迁移和持久化边界**  
   会改变数据模型或行为边界的迁移脚本与 ORM Schema。

5. **关键测试**  
   对外契约测试、集成测试、端到端测试或作为行为规范存在的测试。

6. **关键构建/发布/运维脚本**  
   当它们会影响发布、安全、环境差异或故障恢复时。

### 2.2 v1 的默认排除范围

以下工件 SHOULD 默认排除：

1. 第三方依赖目录；
2. 锁文件；
3. 编译输出目录；
4. 纯静态二进制资源；
5. 自动生成且不应直接编辑的中间产物；
6. 临时日志、缓存和本地工具状态文件。

### 2.3 需要特别策略判断的灰区工件

以下工件不应简单一刀切，需要由 Scope Manager 配置：

- 代码生成产物；
- 样板代码；
- 大型 JSON 配置；
- 文档内嵌脚本；
- 测试快照；
- 基础设施定义文件；
- 查询脚本和分析脚本。

原则是：  
**只有在该工件会显著影响开发任务路由、系统行为理解或风险识别时，才值得进入镜像范围。**

---

## 3. 真相源边界

### 3.1 真相源的组成

ANRSM 中的真相源包括：

- 可执行源码；
- 影响行为的受管配置；
- 决定对外契约的定义文件；
- 决定数据结构的迁移文件；
- 明确承载行为规范的测试；
- 关键发布与运维脚本。

### 3.2 不是真相源的内容

以下内容 MUST NOT 被当作真相源：

- 文件镜像；
- 模块镜像；
- 知识文档；
- PR 描述；
- Issue 讨论；
- Commit Message；
- 会议纪要；
- Wiki 页面；
- 口头知识。

这些内容可以辅助理解，但一旦与真相源冲突，必须以真相源为准。

---

## 4. 角色边界

### 4.1 ANRSM 负责什么

ANRSM v1 负责：

1. 为纳入范围的工件生成机器优先语义镜像；
2. 建立镜像与源码之间的确定性映射；
3. 检测语义漂移；
4. 通过门禁阻止未同步漂移流入主干；
5. 为 Agent 提供任务路由与最小充分上下文入口；
6. 为后续知识层生成提供统一语义基座。

### 4.2 ANRSM 不负责什么

ANRSM v1 不负责：

1. 替代编译、测试、Lint、静态分析；
2. 替代代码审查；
3. 保证所有自然语言总结都绝对完美；
4. 自动修复所有逻辑错误；
5. 对运行时动态行为做完整精确模拟；
6. 代替现有知识管理平台的全部功能；
7. 代替全文搜索；
8. 在无源码展开的前提下完成所有复杂修改。

---

## 5. 系统假设

### 5.1 仓库假设

v1 默认假设：

1. 仓库在 Git 管理之下；
2. 主要源码文件具备相对稳定的路径与扩展名；
3. 主要语言至少存在一种可用的解析策略；
4. 团队接受将镜像纳入版本管理；
5. 团队接受在提交或合并阶段增加同步门禁。

### 5.2 团队工作流假设

v1 假设团队具备以下最小工作流成熟度：

1. 提交与 PR 有明确的代码所有权或责任边界；
2. 团队能接受“源码改动触发镜像改动”的新规范；
3. Agent 的使用不是完全不可控的个人行为，而是可被规范化的协作方式；
4. 有能力维护最基本的 CI 流程。

### 5.3 技术假设

1. 至少一种主语言可做 AST 或等价结构化提取；
2. 对无法完全解析的语言，允许回退到保守策略，但 MUST 显式降低置信度；
3. 仓库中不存在大量无法识别边界的混沌脚本堆积，否则 v1 覆盖率会显著受限。

---

## 6. 非目标

为了防止目标膨胀，以下内容明确列为 v1 非目标。

### 6.1 非目标 N1：替代源码阅读

ANRSM 的目标不是让任何任务都只读镜像。  
它的目标是让大多数任务先通过镜像完成定位、规划和初步理解，再在必要处展开源码。

### 6.2 非目标 N2：一开始就支持任意语言

v1 SHOULD 先支持一到两种主语言，优先打通闭环。  
“支持所有语言”是平台化阶段目标，不应成为 v1 阻塞项。

### 6.3 非目标 N3：自动构建完美知识库

知识层是重要目标，但不是 v1 的最小阻塞项。  
v1 首先要保证文件镜像、漂移检测和门禁。

### 6.4 非目标 N4：让镜像脱离代码独立演进

如果镜像允许独立演进，它就会迅速成为第二真相源。  
这违背 ANRSM 的基本设计。

### 6.5 非目标 N5：用 LLM 直接发明事实

LLM 可以压缩、重写、解释，但不能在没有结构化约束和来源锚点的前提下“自由生成”核心语义事实。

---

## 7. 质量边界

### 7.1 可接受的不完美

以下情况在 v1 是可接受的，只要它们被显式标识：

- 某些语言仅有保守级提取；
- 某些复杂文件的摘要置信度较低；
- 某些低价值工件暂未纳入；
- 模块边界需要经历一轮重构后才能更清晰。

### 7.2 不可接受的状态

以下状态即使在 v1 也不可接受：

1. 镜像无来源、无指纹、无新鲜度；
2. 源码改了，但系统没有任何漂移信号；
3. Agent 可以长期跳过镜像检查；
4. 文档存在但门禁无法阻断过期镜像；
5. 生成器升级后全仓镜像不可重建；
6. 团队无法判断镜像是否可信。

---

## 8. 代码生成与镜像的边界

### 8.1 代码生成的基本原则

如果仓库存在代码生成链路，ANRSM SHOULD 优先镜像以下对象之一：

1. 生成器源头；
2. 模板；
3. Schema；
4. 生成规则。

而不是盲目镜像所有生成产物。

### 8.2 为什么不应默认镜像生成产物

原因有三：

1. 生成产物往往数量大、变化频繁、价值低；
2. 它们的真实语义通常受上游 Schema 或模板决定；
3. 直接镜像生成产物会制造无意义 Token 负担和噪声漂移。

### 8.3 例外条件

当生成产物本身被直接调试、直接引用或直接承载外部契约时，MAY 将其纳入范围，但应在配置中显式声明。

---

## 9. 测试的边界

### 9.1 为什么测试可能值得纳入

对于很多团队，测试不仅是验证工具，还是行为规格。  
因此：

- 契约测试；
- 集成测试；
- 端到端测试；
- 关键业务规则测试

都可能值得被镜像。

### 9.2 v1 的默认建议

v1 SHOULD 优先镜像以下测试：

1. 对外 API/事件/协议契约测试；
2. 关键业务规则测试；
3. 会显著影响任务路由的测试入口。

单元测试全量镜像通常不是 v1 必需能力。

---

## 10. 人类知识层的边界

### 10.1 人类知识层是什么

人类知识层是服务于理解、交接、排障、维护和培训的派生层。  
它不是仓库真实行为的唯一描述来源。

### 10.2 人类知识层不应做什么

人类知识层 MUST NOT：

- 自行定义新的行为真相；
- 绕过镜像直接描述与代码不一致的事实；
- 变成一个由个人长期手工维护但无人校验的孤立 Wiki。

### 10.3 v1 中的人类知识层定位

在 v1 中，人类知识层 SHOULD 作为模块镜像和结构化事实的二次编译物存在。  
若资源有限，知识层 MAY 推迟到里程碑后段。

---

## 11. Agent 边界

### 11.1 Agent 必须做什么

Agent 在合规模式下 MUST：

1. 先读镜像再决定是否展开源码；
2. 在修改源码后触发镜像更新；
3. 在镜像失效时显式说明失效原因或触发修复；
4. 不得把过期镜像当作高可信输入。

### 11.2 Agent 不应做什么

Agent SHOULD NOT：

1. 在没有镜像或镜像已知失效时直接大范围修改源码；
2. 只凭自然语言镜像生成复杂精确补丁；
3. 在未读受影响邻域的情况下跨模块盲改；
4. 修改源码后不更新镜像。

---

## 12. 仓库规模边界

v1 最适合以下量级：

- 单仓或单主仓；
- 主要语言数量有限；
- 代码规模从中小型到中大型；
- 变更频率高，但不是超大单体平台级别。

对于超大多仓、多语言、强动态元编程系统，v1 SHOULD 先从单域试点，而不是一次性全局铺开。

---

## 13. 时间边界

v1 的目标不是“一次初始化后永远正确”，而是：

1. 建立基线；
2. 能基于变更持续维护；
3. 在策略允许范围内快速感知失效；
4. 让过期状态可见、可阻断、可修复。

因此，“实时”在 v1 中应解释为：

- 提交前、本地验证前、PR 检查前或合并前的可接受同步粒度；
- 不要求毫秒级后台常驻分析。

---

## 14. 组织边界

ANRSM 不只是技术实现，还会改变协作契约。  
因此组织层面至少需要接受以下事实：

1. “代码变更不更新镜像”不再被视为正常行为；
2. 评审不仅检查代码，也检查镜像是否反映真实语义变化；
3. 低质量镜像会直接影响 Agent 和人类的后续效率；
4. 团队需要指定镜像策略和适配器的维护责任人。

如果组织不接受这些治理约束，ANRSM 很难长期有效。

---

## 15. 边界判断的决策准则

当团队不知道某类工件是否该纳入 ANRSM 时，可按以下顺序判断：

1. 它是否影响真实行为或行为边界？
2. 它是否能显著帮助 Agent 路由和规划？
3. 它是否会频繁变动且难以维护？
4. 是否存在更上游、更稳定的真相源可替代？
5. 将其纳入后，增量维护成本是否可接受？

只有在 1 和 2 至少满足其一，且 4 不成立时，纳入才有较高价值。

---

## 16. 一句话结论

ANRSM v1 的边界是：  
**用最少但关键的受管范围，建立源码到语义镜像的强同步闭环；它不是全能知识平台，也不是替代源码的魔法层。**

# 04. 架构与组件模型

状态：Normative  
版本：v1  
优先级：P0

---

## 1. 目的

本文件定义 ANRSM 的核心架构、组件责任、控制面/数据面划分、初始化与增量更新流水线，以及语言适配和生成策略。

本文件的目标不是给出唯一实现，而是给出 **必须成立的架构约束** 与 **推荐的实现形态**，使不同团队能够在不破坏规范的前提下完成具体工程落地。

---

## 2. 总体架构

ANRSM v1 推荐采用 **仓库内文件型系统 + 增量计算流水线** 的架构。  
逻辑上分为两部分：

- **控制面（Control Plane）**：定义范围、策略、模块边界、语言适配、门禁强度和生成规则；
- **数据面（Data Plane）**：扫描源码、提取语义、生成镜像、构建索引、检测漂移、输出报告。

### 2.1 控制面包含

1. 范围配置；
2. 排除规则；
3. 模块映射规则；
4. 语言适配器配置；
5. 漂移策略；
6. 门禁策略；
7. 生成模板与压缩规则；
8. 知识层编译策略。

### 2.2 数据面包含

1. 仓库扫描；
2. 结构化提取；
3. 语义归一化；
4. 指纹计算；
5. 镜像生成；
6. 索引构建；
7. 漂移分析；
8. 验证与写回；
9. 报告输出。

---

## 3. 组件清单与职责

### 3.1 Scope Manager

职责：

- 定义镜像范围；
- 维护纳入/排除路径；
- 定义工件分类；
- 识别模块归属规则。

输入：

- `anrsm.yaml` 或等价配置；
- 仓库目录树；
- 可选的模块映射规则。

输出：

- 受管文件清单；
- 文件类型分类；
- 模块归属表。

约束：

- Scope Manager MUST 是确定性的；
- 相同配置和相同仓库状态 MUST 产生相同范围结果。

### 3.2 Repository Scanner

职责：

- 枚举受管文件；
- 读取 Git 状态或差异集；
- 为下游提供稳定的输入记录。

输入：

- Scope Manager 输出；
- Git diff 或全量扫描触发条件。

输出：

- 文件清单；
- 候选变更集；
- 文件元信息（路径、时间戳、大小、hash 等）。

### 3.3 Language Adapter

职责：

- 按语言解析源码；
- 提取 AST、导入导出、符号、签名、注解、配置结构；
- 返回统一的语言中立表示所需原始信息。

输入：

- 单个受管文件；
- 语言配置；
- 可选的项目级解析上下文（tsconfig、build graph 等）。

输出：

- 语言特定解析结果；
- 解析失败信息；
- 置信度与限制说明。

约束：

- 主语言适配器 SHOULD 优先使用 AST 或等价结构化解析；
- 仅基于正则的适配器 MUST 标记低置信度。

### 3.4 Semantic Extractor

职责：

- 将语言特定结果归一到统一语义模型；
- 提取符号、接口、状态、依赖、副作用、关键路径和候选风险点；
- 提供生成镜像所需的结构化事实。

输入：

- Language Adapter 输出；
- 文件分类信息；
- 可选的跨文件上下文。

输出：

- 标准化语义对象；
- 语义邻接边；
- 语义置信度。

### 3.5 Fingerprint Engine

职责：

- 计算 Source Fingerprint；
- 计算 Semantic Fingerprint；
- 支持差异比较和新鲜度判断。

输入：

- 源码内容；
- 标准化语义对象；
- 指纹策略配置。

输出：

- `source_fingerprint`
- `semantic_fingerprint`

约束：

- Semantic Fingerprint SHOULD 稳定排除非语义噪声；
- 指纹算法升级 MUST 可被版本化记录。

### 3.6 Mirror Generator

职责：

- 根据标准模板和结构化事实生成文件镜像；
- 聚合文件镜像生成模块镜像；
- 写入标准头和正文分节。

输入：

- 标准化语义对象；
- 指纹；
- 模板规则；
- 旧镜像（用于稳定更新和 diff）。

输出：

- 文件镜像；
- 模块镜像；
- 可选的知识层素材片段。

约束：

- Generator MUST 保持章节顺序稳定；
- 头字段 MUST 可机读；
- 正文 SHOULD 遵循压缩规范，避免随意 prose 漂移。

### 3.7 Index Builder

职责：

- 构建路由索引、符号索引、依赖索引；
- 提供任务到工件集合的检索基础。

输入：

- 文件镜像；
- 模块镜像；
- 语义图。

输出：

- `routing_index`
- `symbol_index`
- `dependency_index`

### 3.8 Drift Analyzer

职责：

- 比较新旧语义状态；
- 识别 Non-material Change 与 Material Change；
- 标记受影响镜像；
- 输出漂移级别和建议动作。

输入：

- 变更集；
- 新旧指纹；
- 新旧索引；
- 门禁策略。

输出：

- 漂移报告；
- 受影响工件列表；
- 阻塞建议。

### 3.9 Policy Gate

职责：

- 根据策略决定是否允许提交、推送、合并或发布继续；
- 提供可读错误信息和修复提示。

输入：

- 漂移报告；
- 策略配置；
- 当前执行场景（本地、CI、PR、主干合并）。

输出：

- Pass / Fail 结果；
- 失败码；
- 修复提示。

### 3.10 Knowledge Compiler

职责：

- 从文件镜像、模块镜像和索引派生人类知识层；
- 输出架构说明、功能边界、交接材料和 Runbook 草稿。

输入：

- 机器语义镜像层；
- 知识模板；
- 可选的人工补充段。

输出：

- 知识文档集合。

约束：

- Knowledge Compiler 生成的工件 MUST 保留来源；
- 人类知识层 MUST NOT 失去到镜像的可追溯关系。

---

## 4. 两条核心流水线

### 4.1 初始化流水线

初始化阶段的目标是建立一个可验证的基线，而不是只生成一次文档。

流程：

1. Scope Manager 解析仓库范围；
2. Repository Scanner 做全量清单；
3. Language Adapter 对纳入文件解析；
4. Semantic Extractor 归一语义模型；
5. Fingerprint Engine 计算初始指纹；
6. Mirror Generator 生成文件镜像与模块镜像；
7. Index Builder 生成索引；
8. Drift Analyzer 产出基线一致性报告；
9. Policy Gate 检查初始覆盖率和完整性。

初始化结果必须可重建、可审计、可作为后续增量更新的稳定起点。

### 4.2 增量流水线

增量阶段的目标是只重算受影响邻域。

流程：

1. Repository Scanner 从 Git 获取变更集；
2. Scope Manager 判断哪些工件受影响；
3. Language Adapter 重新解析变更文件及必要邻域；
4. Semantic Extractor 更新语义对象；
5. Fingerprint Engine 计算新指纹；
6. Drift Analyzer 对比新旧状态；
7. Mirror Generator 更新受影响镜像；
8. Index Builder 更新相关索引分片；
9. Policy Gate 决定是否放行。

增量模式 SHOULD 是默认路径。  
只有以下情况 MAY 触发全量重建：

- 语言适配器升级；
- 指纹算法升级；
- 范围配置重大变化；
- 模块规则变化；
- 索引损坏或无法证明一致性。

---

## 5. 控制面设计

### 5.1 配置项分类

控制面配置 SHOULD 至少包括：

1. `source_roots`
2. `exclude_patterns`
3. `module_rules`
4. `language_adapters`
5. `policies`
6. `routing`
7. `knowledge`
8. `performance`

### 5.2 配置设计原则

- 人可读；
- 可版本管理；
- 默认安全；
- 不依赖隐式全局状态；
- 变更可触发重建边界的判断。

### 5.3 模块规则设计原则

模块不是纯目录概念。  
一个健康的模块规则 SHOULD 同时满足：

- 业务边界稳定；
- 依赖关系相对集中；
- 对 Agent 任务路由有价值；
- 与代码所有权或维护责任接近。

---

## 6. 数据面设计

### 6.1 统一语义模型

ANRSM v1 建议建立最小统一语义模型，包含以下字段簇：

1. **身份信息**：路径、语言、模块归属、工件类别；
2. **符号信息**：导出符号、主要内部符号、入口点；
3. **依赖信息**：imports、calls、配置读取、外部接线；
4. **契约信息**：输入、输出、Schema、事件、路由；
5. **状态信息**：状态对象、生命周期、状态变化点；
6. **副作用信息**：I/O、网络、存储、日志、缓存、消息；
7. **风险信息**：异常路径、幂等性、鉴权、并发、回滚；
8. **置信信息**：解析等级、摘要等级、未决事项。

### 6.2 统一语义模型的意义

如果没有统一语义模型，后续所有能力都会失稳：

- 指纹无法稳定；
- 漂移难以分级；
- 路由索引缺少统一基础；
- 多语言适配器不可比较；
- 知识层很难保持一致。

---

## 7. 语言适配策略

### 7.1 适配优先级

推荐的事实提取优先级如下：

1. AST / 编译器 API；
2. LSP 或语义服务；
3. 结构化约定文件；
4. 稳定格式的静态解析；
5. 保守文本启发式。

越往后，置信度越低。

### 7.2 对 Prompt 的要求

Prompt MAY 用于：

- 把结构化事实压缩成可读镜像正文；
- 生成知识层；
- 辅助判断复杂影响面。

Prompt MUST NOT 直接取代：

- 符号提取；
- 指纹计算；
- 基础依赖识别；
- Freshness 判定；
- 漂移分级。

### 7.3 低置信度策略

当语言适配器无法稳定解析时，系统 MUST：

1. 输出 `confidence_band: low`；
2. 降低自动化放行力度；
3. 提示需要展开源码或人工复核；
4. 保持可追溯，而不是静默生成高自信 prose。

---

## 8. 生成策略

### 8.1 先事实，后表达

Mirror Generator SHOULD 按以下顺序工作：

1. 接收统一语义事实；
2. 用模板固定段落骨架；
3. 对每段应用压缩规则；
4. 保留关键约束、边界和副作用；
5. 附加展开提示；
6. 写入文档头。

### 8.2 稳定性优先于文采

镜像是工程工件，不是写作作品。  
因此生成策略 SHOULD 优先：

- 术语稳定；
- 句式稳定；
- 章节稳定；
- 可 diff；
- 少同义替换；
- 少无意义润色。

### 8.3 机器区块与人工区块

推荐将镜像正文分为：

- **机器受管区块**：由生成器完全维护；
- **人工补充区块**：允许人工追加上下文，但必须明确标记边界。

机器区块 MUST 可覆盖重写；人工区块 SHOULD 保持有限且有格式约束。

---

## 9. 索引设计

### 9.1 路由索引

职责：

- 从任务描述中召回候选模块、文件和符号；
- 支持 Agent 先读镜像再定向展开源码。

最小字段建议：

- `artifact_id`
- `module_id`
- `keywords`
- `exported_symbols`
- `contracts`
- `risk_tags`
- `adjacent_artifacts`

### 9.2 符号索引

职责：

- 追踪符号定义与引用；
- 支撑影响面分析；
- 为路由和漂移提供更精确粒度。

### 9.3 依赖索引

职责：

- 描述 import、调用、配置和事件邻接；
- 支持变更邻域扩张。

---

## 10. 漂移分析的架构位置

Drift Analyzer 位于“新事实生成之后、写回之前”的关键位置。  
其作用不是单纯检查“文件是否同时改了文档”，而是回答：

1. 当前源码是否引发语义变化；
2. 哪些镜像因此失效；
3. 失效是低风险还是阻塞级；
4. 是否需要扩展受影响邻域；
5. 是否需要升级到全量或更深层检查。

这也是 ANRSM 与普通 docs sync 工具的本质区别。

---

## 11. 策略门禁设计

Policy Gate SHOULD 根据执行场景支持多层策略：

### 11.1 pre-commit

目标：快速本地反馈。  
特点：

- 只检查变更集与直接邻域；
- 优先低延迟；
- 可配置保守放行阈值。

### 11.2 pre-push / CI

目标：提高一致性与阻断效果。  
特点：

- 可扩展影响面；
- 可跑更完整的索引更新；
- 对 `blocking` 级漂移必须失败。

### 11.3 main branch / release

目标：保证主干可信。  
特点：

- 策略最严格；
- 对 `unknown` 和 `invalid` 状态更保守；
- 必要时触发更深层校验。

---

## 12. 可观测性与诊断

一个可用的 ANRSM 系统必须能回答以下问题：

1. 为什么这个文件被纳入或排除？
2. 为什么这个镜像是 `stale`？
3. 哪个字段导致它被判定为 Material Change？
4. 为什么这个任务路由到了这些文件？
5. 为什么这次触发了全量重建？
6. 哪个语言适配器产出了低置信度？

因此系统 SHOULD 输出：

- 详细漂移报告；
- 受影响图；
- 失败码；
- 适配器诊断；
- 覆盖率与新鲜度统计。

---

## 13. 推荐部署形态

v1 推荐三种部署形态，从低到高依次是：

### 13.1 仓库内 CLI 模式

最简单，适合 PoC 和早期采用。  
优点：

- 易集成；
- 易调试；
- 与 Git/CI 兼容好。

### 13.2 CI 服务模式

适合团队级推广。  
优点：

- 强制一致性更好；
- 可以集中化缓存；
- 可积累指标。

### 13.3 守护进程或平台模式

适合后续平台化。  
但不应成为 v1 的首要投入方向。

---

## 14. 架构决策建议

### 决策 D1：先用文件系统作为镜像存储

理由：

- 可直接版本管理；
- 可 diff；
- 可 code review；
- 无外部存储依赖；
- 便于渐进式引入。

### 决策 D2：先做增量，不做全量实时

理由：

- 大多数仓库改动是局部的；
- 全量扫描会迅速拖垮体验；
- 增量才符合提交前和 CI 的实际节奏。

### 决策 D3：先锁定规范，再做体验层

理由：

- 没有稳定规范，任何 UI 或平台都会建立在松散语义上；
- 一旦规范漂移，后续迁移成本极高。

---

## 15. 一句话结论

ANRSM v1 的正确架构不是“一个会写文档的 Agent”，而是 **一个由范围控制、结构化提取、语义压缩、漂移分析、门禁治理和 Agent 契约共同组成的仓库语义基础设施**。

# 05. 信息模型与仓库目录布局

状态：Normative  
版本：v1  
优先级：P0

---

## 1. 目的

本文件定义 ANRSM 的信息模型、工件类型、目录布局、路径映射规则、元数据组织方式，以及各类镜像工件之间的关系。

如果没有统一的信息模型，ANRSM 会在以下环节失控：

- 路径映射不稳定；
- 多语言工件冲突；
- 索引和镜像难以互相引用；
- 漂移报告无法定位问题；
- Agent 难以建立固定读取顺序。

---

## 2. 设计原则

信息模型 MUST 满足以下原则：

1. **确定性**  
   相同源码路径在相同配置下 MUST 映射到相同镜像路径。

2. **双向可追溯**  
   任意镜像工件 MUST 能追溯到其来源源码；任意源码 SHOULD 能找到其镜像工件。

3. **文本化与可 diff**  
   v1 的核心工件 SHOULD 使用文本格式，便于版本控制、差异审阅和生成器调试。

4. **机器优先，人工可读**  
   头字段和索引首先服务于机器；正文同时服务于机器和人类审阅。

5. **局部更新友好**  
   目录结构与索引设计 SHOULD 支持按文件和模块做局部更新。

---

## 3. 顶层目录建议

ANRSM v1 推荐在仓库根目录引入以下结构：

```text
repo/
  src/
  apps/
  packages/
  tests/
  mirror/
    file/
    module/
    index/
    reports/
  knowledge/
    architecture/
    features/
    operations/
    onboarding/
  .anrsm/
    anrsm.yaml
    cache/
    logs/
```

说明：

- `mirror/` 是机器优先语义镜像层；
- `knowledge/` 是人类优先知识层；
- `.anrsm/` 是系统配置、缓存和运行状态目录；
- 真实源码目录保持原样，不应为了适配 ANRSM 强行改造业务布局。

---

## 4. 工件类型模型

### 4.1 file_mirror

作用：

- 与单个源码文件一一映射；
- 提供任务定位、语义快速阅读、漂移判断和依赖入口。

最小组成：

- 文档头；
- 正文规范段落；
- 可选的人工补充区块。

### 4.2 module_mirror

作用：

- 聚合一组文件镜像；
- 表达模块责任、模块边界、外部契约、依赖关系、稳定性和常见风险。

生成来源：

- 文件镜像；
- 模块规则；
- 依赖图；
- 契约汇总。

### 4.3 routing_index

作用：

- 从任务文本映射到候选模块/文件/符号；
- 缩短 Agent 定位路径。

### 4.4 symbol_index

作用：

- 管理导出符号、关键内部符号、定义位置和使用邻域；
- 辅助影响分析与精确路由。

### 4.5 dependency_index

作用：

- 管理 import、调用、事件、配置读取、数据流邻接等关系；
- 支撑变更邻域扩展。

### 4.6 drift_report

作用：

- 记录当前变更集引发的镜像漂移结果；
- 作为门禁输入与诊断输出。

### 4.7 knowledge_doc

作用：

- 为架构师、接手工程师、值班工程师和 Reviewer 提供可读知识材料；
- 来源必须可追溯。

---

## 5. 路径映射规则

### 5.1 文件镜像路径

推荐规则：

```text
<repo_root>/<mirror_root>/file/<source_relative_path>.md
```

示例：

```text
src/auth/login.ts
-> mirror/file/src/auth/login.ts.md

apps/web/pages/home.tsx
-> mirror/file/apps/web/pages/home.tsx.md
```

要求：

1. 目录层级 MUST 保持与源码相同；
2. 源码扩展名 MUST 保留在文件名中；
3. 镜像扩展名 MUST 追加 `.md`；
4. 不允许用路径哈希取代可读路径作为默认规则，除非发生文件系统限制。

### 5.2 模块镜像路径

推荐规则：

```text
mirror/module/<module_id>.md
```

若 `module_id` 包含多级路径，则直接映射为层级目录：

```text
module_id = billing/invoice
-> mirror/module/billing/invoice.md
```

### 5.3 知识文档路径

推荐规则：

```text
knowledge/<category>/<logical_name>.md
```

类别建议：

- `architecture`
- `features`
- `operations`
- `onboarding`

---

## 6. 文件级镜像的信息模型

### 6.1 文件级镜像由两部分组成

1. **机器可读文档头**  
   用于追踪、校验、策略判断和索引关联。

2. **机器优先、人工可读的正文**  
   用于压缩语义、表达边界和提供展开提示。

### 6.2 文档头最小字段集

v1 建议最小字段如下：

- `anrsm_version`
- `artifact_type`
- `artifact_id`
- `source_path`
- `source_language`
- `module_id`
- `source_fingerprint`
- `semantic_fingerprint`
- `source_commit`
- `freshness_state`
- `confidence_band`
- `generator_name`
- `generator_version`
- `generated_at`
- `exports`
- `dependencies`
- `risk_tags`

说明：

- `exports` 和 `dependencies` MAY 做轻量摘要，不必在头中放入全部细节；
- 详细字段可由索引承载。

### 6.3 正文的标准段落

v1 推荐以下标准段落结构：

1. `职责`
2. `对外契约`
3. `输入与前置条件`
4. `状态与控制流`
5. `副作用与外部依赖`
6. `关键不变量与风险`
7. `变更影响`
8. `何时必须展开源码`

这个结构兼顾：

- Agent 的快速定位；
- Reviewer 的快速核验；
- 人类维护者的快速理解。

---

## 7. 模块级镜像的信息模型

模块级镜像不是多个文件摘要的简单拼接。  
它 SHOULD 回答模块层面的七个问题：

1. 这个模块的唯一职责是什么；
2. 对外暴露了哪些稳定契约；
3. 依赖了哪些外部模块或基础设施；
4. 内部有哪些关键子域或子流程；
5. 该模块的主要状态与一致性边界是什么；
6. 哪些文件是高影响入口；
7. 哪些变更会波及上游或下游。

### 7.1 模块级镜像最小头字段建议

- `artifact_type: module_mirror`
- `artifact_id`
- `module_id`
- `source_scope`
- `aggregate_semantic_fingerprint`
- `freshness_state`
- `generator_name`
- `generator_version`
- `generated_at`
- `critical_files`
- `external_contracts`

### 7.2 模块级正文建议

1. 模块职责
2. 模块边界
3. 稳定契约
4. 依赖关系
5. 状态与一致性
6. 关键文件与入口
7. 高频变更面
8. 常见故障与风险

---

## 8. 索引的信息模型

### 8.1 routing_index

路由索引 SHOULD 至少包含：

- `artifact_id`
- `module_id`
- `source_path`
- `semantic_keywords`
- `exported_symbols`
- `contract_terms`
- `risk_tags`
- `adjacent_artifacts`
- `confidence_band`

### 8.2 symbol_index

符号索引 SHOULD 至少包含：

- `symbol_name`
- `symbol_kind`
- `defined_in`
- `exported`
- `module_id`
- `references`
- `related_contracts`

### 8.3 dependency_index

依赖索引 SHOULD 至少包含：

- `from_artifact_id`
- `to_artifact_id`
- `dependency_kind`
- `strength`
- `path`
- `evidence`

`dependency_kind` 可取值示例：

- `import`
- `call`
- `event_publish`
- `event_consume`
- `config_read`
- `schema_bind`
- `storage_access`

---

## 9. 漂移报告的信息模型

漂移报告是门禁的直接输入。  
它必须同时能服务于机器判断和人工诊断。

最小字段建议：

- `report_version`
- `report_id`
- `repo_commit_before`
- `repo_commit_after`
- `generated_at`
- `drift_level`
- `changed_files`
- `affected_artifacts`
- `material_changes`
- `policy_result`
- `recommendations`

说明：

- `changed_files` 记录源码变更集；
- `affected_artifacts` 记录受影响镜像；
- `material_changes` 记录实质变更分类与证据。

---

## 10. 知识文档的信息模型

知识文档不要求与单个源码文件一一映射，但 MUST 具备来源追踪。  
建议头字段：

- `artifact_type: knowledge_doc`
- `artifact_id`
- `knowledge_category`
- `source_modules`
- `source_artifacts`
- `freshness_state`
- `generator_name`
- `generator_version`
- `generated_at`

知识文档正文结构视类别而定，但 SHOULD 避免与镜像正文冲突和重复。

---

## 11. 目录布局与权限边界

### 11.1 机器层与人工层分离

推荐将 `mirror/` 与 `knowledge/` 分离，理由如下：

1. 明确机器优先与人类优先的职责差异；
2. 减少人工误编辑机器区块；
3. 便于策略和 Code Owner 隔离；
4. 便于后续独立发布知识层。

### 11.2 `.anrsm/` 目录的角色

`.anrsm/` SHOULD 用于：

- 配置；
- 缓存；
- 运行日志；
- 临时构建状态。

`.anrsm/` 中的缓存和日志 SHOULD NOT 被视为规范工件。  
真正要进入版本管理的规范工件应位于 `mirror/` 和 `knowledge/`。

---

## 12. 版本管理建议

### 12.1 应进入版本管理的内容

- `mirror/file/**`
- `mirror/module/**`
- `mirror/index/**`（视团队策略，可只提交稳定索引）
- `knowledge/**`
- `.anrsm/anrsm.yaml`

### 12.2 不应进入版本管理的内容

- 临时缓存；
- 本地诊断日志；
- 解析中间产物；
- 大型临时图数据库快照；
- 运行时锁文件。

---

## 13. 稳定性规则

### 13.1 路径稳定性

一旦确定路径映射规则，v1 周期内 SHOULD 保持稳定。  
频繁修改路径规则会导致：

- 历史 diff 失真；
- Code Review 成本上升；
- 索引失配；
- Agent 引用失效。

### 13.2 字段稳定性

头字段的删除、改名和语义改写 MUST 经过版本升级或迁移计划。  
否则上游门禁和下游 Agent 读取器都会失效。

---

## 14. 统一引用规则

工件之间引用 SHOULD 优先使用 `artifact_id`，而不是自由文本标题。  
这样做的好处：

- 稳定；
- 可机读；
- 可跨语言；
- 可用于索引和图谱。

正文中 MAY 出现可读标题，但机器引用 SHOULD 以 `artifact_id` 为准。

---

## 15. 信息模型的最小闭环

一个符合 v1 的实现，至少要形成以下闭环：

1. `source_path`
2. `artifact_id`
3. `source_fingerprint`
4. `semantic_fingerprint`
5. `freshness_state`
6. `module_id`
7. `drift_report`

只要这些对象能稳定联通，系统就已经具备：

- 可追溯；
- 可增量更新；
- 可阻断漂移；
- 可为 Agent 提供入口

这四项核心能力。

---

## 16. 一句话结论

ANRSM 的信息模型不是“把代码旁边再放几篇 Markdown”，而是 **把源码、镜像、索引、漂移报告和知识层组织成一套有稳定路径、有固定字段、有双向追踪关系的工件系统**。

# 06. 语义镜像协议与文档头规范

状态：Normative  
版本：v1  
优先级：P0

---

## 1. 目的

本文件定义文件级语义镜像的协议，包括：

- 文档头字段；
- 正文结构；
- 语义压缩规则；
- 指纹定义要求；
- 人工补充边界；
- 生成与更新约束。

该协议是 ANRSM 的核心。  
如果没有稳定协议，所谓“镜像”就只是一组风格不一的说明文档，无法被 Agent、门禁和索引系统可靠消费。

---

## 2. 协议定位

文件级镜像协议是 **机器优先、文本承载、可审计的语义压缩协议**。  
它不是：

- prose 写作规范；
- 注释风格指南；
- 任意文档模板；
- README 约定。

一个文件只有在满足本协议时，才能被视为 `file_mirror`。

---

## 3. 文档头的规范格式

v1 推荐使用 YAML Front Matter。格式如下：

```markdown
---
anrsm_version: 1
artifact_type: file_mirror
artifact_id: file_mirror:src/auth/login.ts
source_path: src/auth/login.ts
source_language: typescript
module_id: auth
source_commit: 9f8c2ab
source_fingerprint: sha256:...
semantic_fingerprint: sha256:...
freshness_state: fresh
confidence_band: high
generator_name: anrsm
generator_version: 0.1.0
generated_at: 2026-03-20T10:30:00Z
exports:
  - LoginController
  - validateCaptcha
dependencies:
  - src/core/http/client.ts
  - src/core/auth/session.ts
risk_tags:
  - auth
  - side_effect
  - external_api
---
```

要求：

1. 文档头 MUST 出现在文件起始位置；
2. 字段名 MUST 使用 ASCII `snake_case`；
3. 所有必填字段 MUST 存在；
4. 文档头 MUST 可被无歧义解析；
5. 正文 MUST 紧随其后。

---

## 4. 必填字段说明

### 4.1 anrsm_version

类型：integer  
含义：镜像协议版本。  
要求：

- MUST 填写；
- v1 中固定为 `1`。

### 4.2 artifact_type

类型：string  
要求：

- MUST 为 `file_mirror`；
- 不得省略。

### 4.3 artifact_id

类型：string  
要求：

- MUST 唯一标识当前工件；
- MUST 采用规范格式；
- SHOULD 与 `source_path` 保持可推导关系。

### 4.4 source_path

类型：string  
含义：仓库相对路径。  
要求：

- MUST 指向真实源码文件；
- MUST 使用 POSIX 风格路径；
- MUST 与文件映射规则一致。

### 4.5 source_language

类型：string  
含义：主语言标识。  
示例：

- `typescript`
- `python`
- `go`
- `kotlin`

### 4.6 module_id

类型：string  
含义：该文件所属逻辑模块。  
要求：

- MUST 可由配置或规则推导；
- 对无法确定模块归属的文件，可使用 `global` 或显式保守值，但 SHOULD 减少这种情况。

### 4.7 source_commit

类型：string  
含义：本镜像基于哪个源码提交生成。  
要求：

- MUST 填写短 SHA 或全 SHA；
- SHOULD 能用于调试与回溯。

### 4.8 source_fingerprint

类型：string  
含义：原始源码内容指纹。  
要求：

- MUST 带算法前缀；
- MUST 在源码文本变化时变化。

### 4.9 semantic_fingerprint

类型：string  
含义：基于语义对象计算的指纹。  
要求：

- MUST 带算法前缀；
- SHOULD 对非语义噪声保持稳定；
- MUST 用作 Material Change 判断的重要依据。

### 4.10 freshness_state

类型：enum  
允许值：

- `fresh`
- `stale`
- `invalid`
- `unknown`

要求：

- MUST 存在；
- MUST 由受管流程写入，而不是自由手改。

### 4.11 confidence_band

类型：enum  
允许值：

- `high`
- `medium`
- `low`

要求：

- MUST 存在；
- 当解析器能力有限或摘要存在推断时，不得维持虚假的 `high`。

### 4.12 generator_name / generator_version

作用：记录生成器来源和版本。  
要求：

- MUST 同时存在；
- MUST 用于调试和迁移。

### 4.13 generated_at

类型：RFC 3339 时间戳。  
要求：

- MUST 存在；
- MUST 表示本镜像本次生成时间。

---

## 5. 推荐字段说明

以下字段建议在 v1 中支持：

- `exports`
- `dependencies`
- `risk_tags`
- `contracts`
- `adjacent_artifacts`
- `notes_locked`
- `manual_appendix_present`

其中：

- `exports` 用于快速了解外部可见能力；
- `dependencies` 用于快速路由与影响分析；
- `risk_tags` 用于提醒鉴权、事务、缓存、并发、外部 I/O 等风险域；
- `manual_appendix_present` 用于声明是否存在人工补充段。

---

## 6. 正文协议

### 6.1 正文的职责

正文不是对源码的逐行解释。  
正文的职责是把完成工程判断所需的关键语义压缩到最小充分上下文中。

### 6.2 正文的标准结构

v1 推荐严格采用以下结构：

```markdown
## 职责
## 对外契约
## 输入与前置条件
## 状态与控制流
## 副作用与外部依赖
## 关键不变量与风险
## 变更影响
## 何时必须展开源码
```

要求：

- 段落标题 SHOULD 保持完全一致；
- 标题顺序 SHOULD 稳定；
- 段落缺失 MUST 有明确理由；
- 不应为每个文件自由发明新结构。

### 6.3 各段落应回答的问题

#### 职责

这个文件在系统中的唯一职责是什么。  
不应堆砌历史背景和无关上下文。

#### 对外契约

它向外暴露什么：函数、类、类型、路由、事件、命令、配置键等。  
这里只写外部依赖它时需要知道的事实。

#### 输入与前置条件

调用前必须满足什么条件，依赖什么环境、配置、状态或数据形态。

#### 状态与控制流

它如何推进主要流程；有哪些关键分支、状态转换、失败路径、回滚点。

#### 副作用与外部依赖

它会读写什么：数据库、网络、缓存、日志、队列、本地文件、系统时钟等。

#### 关键不变量与风险

什么约束不能被破坏；什么地方最容易引入错误；是否涉及鉴权、幂等性、事务、并发、一致性等。

#### 变更影响

修改本文件最可能影响哪些邻域、模块、契约或测试。

#### 何时必须展开源码

在哪些场景下，仅凭镜像不足以做正确决策，需要阅读源码、测试或调用方。

---

## 7. 语义压缩规则

### 7.1 压缩目标

目标不是尽可能短，而是 **在最小 Token 内保留决策关键语义**。

### 7.2 必须保留的信息

镜像正文 SHOULD 尽量保留：

- 责任边界；
- 对外接口；
- 状态变化点；
- 关键副作用；
- 关键异常路径；
- 不变量；
- 风险点；
- 高影响依赖；
- 修改影响面。

### 7.3 应删除的信息

正文 SHOULD 删除或显著压缩：

- 逐行实现细节；
- 重复的参数名列表；
- 可从签名直接读出的冗余描述；
- 与当前任务路由无关的修辞性背景；
- 纯样板逻辑描述。

### 7.4 不得做的事

正文 MUST NOT：

1. 粘贴大段源码；
2. 把实现细节全部翻译成自然语言；
3. 用一堆形容词替代事实；
4. 把未知当已知；
5. 隐藏关键副作用。

---

## 8. 语义指纹规则

### 8.1 源码指纹与语义指纹的区别

- `source_fingerprint` 反映输入文本是否变化；
- `semantic_fingerprint` 反映镜像有效性相关的语义是否变化。

### 8.2 语义指纹的推荐构成

语义指纹 SHOULD 主要由以下对象归一后计算：

1. 导出符号与签名；
2. 关键输入输出结构；
3. 路由/事件/命令名；
4. 关键依赖边；
5. 关键状态对象与状态转移；
6. 关键副作用类型；
7. 关键风险标签；
8. 关键控制流锚点。

### 8.3 应排除的噪声

语义指纹 SHOULD 尽量排除：

- 空白；
- 注释；
- import 排序；
- 无影响的局部变量重命名；
- 纯格式化；
- 无影响的语法糖替换。

### 8.4 升级原则

指纹算法升级 MUST：

- 记录版本；
- 提供迁移说明；
- 允许全量重建；
- 不得静默改变语义含义。

---

## 9. 新鲜度状态机

### 9.1 状态定义

- `fresh`：当前镜像与当前语义状态一致；
- `stale`：检测到潜在或确定漂移，尚未完成更新与验证；
- `invalid`：文档头缺失、字段错误、解析失败、校验失败或结构破损；
- `unknown`：系统无法确定是否仍然有效。

### 9.2 状态转移建议

```text
fresh -> stale     当源码或语义发生相关变化
stale -> fresh     当镜像更新并通过验证
any   -> invalid   当文档损坏或生成失败
any   -> unknown   当系统缺少足够信息判断
unknown -> fresh   当补齐信息并通过验证
invalid -> fresh   当修复后重新生成并验证
```

### 9.3 策略建议

- `fresh` 可作为默认高可信输入；
- `stale` 不应作为唯一决策依据；
- `invalid` 在阻塞策略下 SHOULD 直接失败；
- `unknown` 在主干和发布策略下 SHOULD 被保守对待。

---

## 10. 人工补充边界

### 10.1 为什么允许人工补充

有些信息难以从静态结构化提取中获得，例如：

- 历史坑位；
- 线上事故教训；
- 业务约束来源；
- 外部系统不可见约束。

因此，系统 MAY 允许受限人工补充区块。

### 10.2 人工补充的边界

人工补充 MUST：

1. 放在明确标记的区块；
2. 不得篡改机器头字段；
3. 不得改写机器受管段落的事实含义；
4. 应说明责任人和更新时间。

推荐格式：

```markdown
## 人工补充
- 线上约束：...
- 已知坑：...
- 复核建议：...
```

### 10.3 人工补充不应解决的问题

人工补充 SHOULD NOT 被用来：

- 掩盖镜像本身结构缺失；
- 替代受管更新流程；
- 直接修正源码事实而不改源码。

---

## 11. 生成规则

### 11.1 生成顺序

Mirror Generator SHOULD 按如下顺序工作：

1. 装载结构化事实；
2. 装载旧镜像以保持稳定性；
3. 计算新指纹；
4. 填充头字段；
5. 生成标准段落；
6. 合并人工补充区块；
7. 写回文件；
8. 重新校验解析性。

### 11.2 稳定更新原则

生成器 SHOULD 尽量保持：

- 段落顺序不变；
- 术语不变；
- 已知事实表达方式不随意改写；
- 只有在事实变化时才改对应段落。

这样做的价值是：

- diff 更干净；
- Reviewer 更容易定位真实变化；
- Agent 更容易缓存和增量读取。

---

## 12. 镜像质量的最低要求

一个合格的文件镜像至少必须满足：

1. 文档头完整且可解析；
2. 文本与 `source_path` 一一对应；
3. `freshness_state` 可解释；
4. 正文包含关键责任与副作用；
5. 当需要展开源码时，正文明确指出触发条件；
6. 不存在大段无效 prose；
7. 不隐瞒关键风险。

---

## 13. 常见坏味道

以下情况说明镜像协议执行失败：

1. 正文看起来像 README，而不是文件镜像；
2. 标题命名每个文件都不一样；
3. 几乎没有副作用和风险描述；
4. 完全没有“何时必须展开源码”段；
5. `semantic_fingerprint` 每次格式化都会变化；
6. 开发者开始直接手改机器头。

---

## 14. 示例

### 14.1 规范示意

```markdown
---
anrsm_version: 1
artifact_type: file_mirror
artifact_id: file_mirror:src/auth/login.ts
source_path: src/auth/login.ts
source_language: typescript
module_id: auth
source_commit: 9f8c2ab
source_fingerprint: sha256:9e...
semantic_fingerprint: sha256:7a...
freshness_state: fresh
confidence_band: high
generator_name: anrsm
generator_version: 0.1.0
generated_at: 2026-03-20T10:30:00Z
exports:
  - login
dependencies:
  - src/core/http/client.ts
  - src/core/auth/session.ts
risk_tags:
  - auth
  - external_api
---

## 职责
处理账号登录主流程，负责参数校验、验证码检查、令牌请求和会话建立。

## 对外契约
对外暴露 `login(credentials)`，成功时返回会话对象，失败时抛出鉴权或验证错误。

## 输入与前置条件
要求调用方已提供标准化凭证对象；当风控开关开启时，必须携带验证码令牌。

## 状态与控制流
流程为参数校验 -> 验证码检查 -> 请求鉴权服务 -> 建立本地会话 -> 返回会话；鉴权失败时不会落盘会话。

## 副作用与外部依赖
依赖 HTTP Client 调用鉴权服务；写入会话存储；记录失败日志。

## 关键不变量与风险
不得在鉴权失败时创建会话；验证码校验与主登录调用必须共享同一风控上下文。

## 变更影响
会影响登录页、会话恢复逻辑、鉴权错误展示和相关契约测试。

## 何时必须展开源码
当修改错误分支、重试策略、风控上下文或会话持久化实现时，必须展开源码与调用方。
```

## 15. 一句话结论

文件级镜像协议的本质，是 **把“代码旁边的一篇说明”升级成“有标准头、有新鲜度、有指纹、有正文骨架、有生成与验证规则的语义工件”**。

# 07. 漂移检测、同步策略与 CI 门禁

状态：Normative  
版本：v1  
优先级：P0

---

## 1. 目的

本文件定义 ANRSM 的同步闭环：  
如何检测源码变化、如何判断镜像是否失效、如何基于漂移结果决定是否阻塞提交或合并。

ANRSM 是否真正成立，关键不在于“能生成镜像”，而在于：

1. 源码变了能否被感知；
2. 漂移能否被分类；
3. 同步能否自动或半自动发生；
4. 未同步的实质漂移能否被门禁阻断。

---

## 2. 为什么普通“源码改了文档也要改”规则不够

传统的 docs sync 规则通常是：

- 某个目录改了；
- 要求另一个目录也有改动；
- 否则失败。

这类规则的问题是：

1. 它只检查“有没有文件一起改”，不检查“改得对不对”；
2. 它不知道格式化和注释改动是不是值得更新镜像；
3. 它无法判断影响邻域；
4. 它不能输出工程上可用的漂移解释。

因此，ANRSM 的门禁必须建立在 **语义漂移分析** 上，而不是简单的路径联动。

---

## 3. 漂移检测的输入

Drift Analyzer 在 v1 中 SHOULD 使用以下输入：

1. Git 变更集；
2. 变更前后的源码指纹；
3. 变更前后的语义指纹；
4. 新旧索引；
5. 模块规则；
6. 策略配置；
7. 解析置信度。

---

## 4. 漂移检测的输出

漂移检测的输出 MUST 至少包含：

1. 变更文件列表；
2. 受影响镜像工件列表；
3. 每个工件的 `freshness_state` 建议；
4. 漂移级别 `drift_level`；
5. 实质变更分类；
6. 证据摘要；
7. 门禁建议。

---

## 5. 变更分类体系

为了降低误报和漏报，v1 推荐使用分级变更分类。

### 5.1 C0：Non-material Change

定义：不会改变镜像有效性的变化。  
示例：

- 空白、注释、格式化；
- import 顺序调整；
- 无影响的代码移动；
- 无外部可见影响的局部命名清理。

默认处理：

- 可以更新 `source_fingerprint`；
- `semantic_fingerprint` SHOULD 保持稳定；
- 不要求阻塞级镜像更新。

### 5.2 C1：Local Material Change

定义：本文件语义发生实质变化，但外部契约未明显改变。  
示例：

- 关键控制流变化；
- 本地状态管理变化；
- 副作用时机变化；
- 错误处理逻辑变化；
- 缓存策略变化。

默认处理：

- 文件镜像 MUST 更新；
- 模块镜像 MAY 更新；
- 若风险域受影响，SHOULD 升级检查邻域。

### 5.3 C2：Contract Material Change

定义：对外契约或可见边界变化。  
示例：

- 导出函数签名变化；
- API 路由或 Schema 变化；
- 事件名变化；
- 配置键变化；
- 权限要求变化。

默认处理：

- 文件镜像 MUST 更新；
- 模块镜像 SHOULD 更新；
- 依赖邻域 SHOULD 被扩展；
- 若未同步，CI SHOULD 失败。

### 5.4 C3：Systemic Material Change

定义：跨模块、跨边界或高风险系统行为变化。  
示例：

- 持久化模型变化；
- 事务边界变化；
- 鉴权链变化；
- 并发模型变化；
- 外部依赖接线变化；
- 发布/运维关键脚本变化。

默认处理：

- 相关文件镜像 MUST 更新；
- 模块镜像 MUST 更新；
- 知识层 MAY 被标记待更新；
- 主干策略下 SHOULD 视为阻塞级。

---

## 6. 漂移级别定义

### none

无漂移。  
条件：新旧语义状态一致，相关镜像仍然有效。

### low

有差异，但不构成阻塞。  
常见于：

- 局部低风险变化；
- 置信度下降但未达阻塞阈值；
- 辅助索引需要刷新。

### material

存在必须修复的镜像漂移。  
通常意味着文件镜像已不再可信。

### blocking

存在会影响主干可信度或 Agent 决策质量的漂移。  
在对应策略下，提交或合并 MUST 被阻止。

---

## 7. Git Diff 的使用原则

### 7.1 Git Diff 是候选集，不是真相

Git Diff SHOULD 用于缩小重新分析范围，但 MUST NOT 直接替代语义比较。  
原因：

- 同样的路径变更可能只是格式化；
- 某些语义变化会波及邻域，而邻域文件本身未改；
- 重命名、移动和生成文件会混淆简单路径匹配。

### 7.2 变更邻域扩展

当发生以下情况时，系统 SHOULD 扩展分析邻域：

1. 导出符号变化；
2. 契约变化；
3. 关键依赖边变化；
4. 高风险标签变化；
5. 模块入口文件变化；
6. 解析器置信度下降。

邻域扩展策略 MAY 采用：

- 直接 import 邻域；
- 反向引用邻域；
- 模块内高影响文件集合；
- 契约关联文件集合。

---

## 8. 同步策略

### 8.1 自动同步

在适配器和生成器足够成熟时，系统 SHOULD 自动更新受影响镜像。  
自动同步适用于：

- 文件镜像正文重写；
- 指纹与头字段刷新；
- 索引更新；
- 漂移报告生成。

### 8.2 半自动同步

当解析能力有限或知识层需要补充上下文时，系统 MAY 进入半自动模式：

1. 自动标记受影响工件；
2. 自动生成候选镜像；
3. 要求人或 Agent Reviewer 复核低置信度部分。

### 8.3 禁止的同步方式

系统 MUST NOT 仅通过“把 old mirror 的时间戳改新”来伪装同步完成。  
同步的判定必须建立在：

- 新的结构化事实；
- 新的语义指纹；
- 新的验证结果。

---

## 9. freshness_state 的判定规则

### 9.1 fresh

要求同时满足：

1. 当前源码可成功解析或以保守方式解释；
2. 当前语义指纹与镜像声明一致；
3. 当前正文与结构化事实一致；
4. 门禁未发现阻塞漂移。

### 9.2 stale

以下任一情况成立即可判定为 `stale`：

- 源码已改动但镜像未更新；
- 语义指纹变化但镜像头未刷新；
- 受影响邻域扩展后发现模块镜像已过时；
- 系统知道它应被更新但尚未完成更新。

### 9.3 invalid

以下情况应判定为 `invalid`：

- 文档头不可解析；
- 必填字段缺失；
- 字段值非法；
- `source_path` 不存在；
- 生成器失败写入；
- 正文结构损坏到无法被消费；
- 指纹自相矛盾。

### 9.4 unknown

以下情况可判定为 `unknown`：

- 解析器无法确定语义；
- 配置缺失；
- 增量状态损坏，但还未完整重建；
- 某些证据不足以得出结论。

---

## 10. 门禁层级设计

### 10.1 本地 pre-commit

目标：快速反馈。  
建议策略：

- 仅检查变更文件及直接邻域；
- 对 `blocking` 失败；
- 对 `material` 失败或警告，视项目节奏决定；
- 对 `unknown` 给出显式提示。

### 10.2 PR CI

目标：保证合并前可信。  
建议策略：

- 对 `material` 和 `blocking` 一律失败；
- 对 `unknown` 在高风险目录下失败；
- 允许低风险目录保守放行，但必须留下注释说明。

### 10.3 主干合并/发布

目标：保护主干与发布质量。  
建议策略：

- `blocking` MUST 失败；
- `material` MUST 失败；
- `invalid` MUST 失败；
- `unknown` SHOULD 失败，除非配置中存在显式例外。

---

## 11. 失败码建议

为便于自动化处理，建议输出标准失败码。

- `ANRSM-001`：受管文件缺少镜像
- `ANRSM-002`：镜像头不可解析
- `ANRSM-003`：必填字段缺失
- `ANRSM-004`：语义指纹不一致
- `ANRSM-005`：检测到 Material Change 但镜像未更新
- `ANRSM-006`：模块镜像失效
- `ANRSM-007`：索引失效或缺失
- `ANRSM-008`：解析器低置信度且策略要求阻塞
- `ANRSM-009`：配置错误导致无法判定新鲜度
- `ANRSM-010`：需要全量重建但未执行

失败码 SHOULD 伴随：

- 问题对象；
- 证据摘要；
- 修复建议。

---

## 12. 典型门禁规则矩阵

| 变更类型 | 文件镜像 | 模块镜像 | 索引 | 建议门禁 |
| --- | --- | --- | --- | --- |
| C0 Non-material | 可不更新正文 | 通常不更新 | 可局部刷新 | 默认放行 |
| C1 Local Material | 必须更新 | 视影响更新 | 应刷新 | 可阻塞 |
| C2 Contract Material | 必须更新 | 应更新 | 必须刷新 | 应阻塞 |
| C3 Systemic Material | 必须更新 | 必须更新 | 必须刷新 | 必须阻塞 |

注：表中“可不更新正文”不等于“完全不处理”。  
即使 C0，也 MAY 更新 `source_commit`、缓存或统计信息，但不应制造伪漂移。

---

## 13. 低置信度与保守策略

### 13.1 低置信度的本质

低置信度不是失败，但它意味着系统无法为某些判断承担完全信任。

### 13.2 推荐策略

当满足以下任一情况时，系统 SHOULD 采用保守策略：

- 高风险目录；
- 契约相关文件；
- 鉴权、事务、并发、迁移、部署脚本；
- 低置信度且发生 Material Change；
- 适配器升级后首次运行。

保守策略包括：

- 扩大邻域分析；
- 要求展开源码；
- 提高门禁级别；
- 要求人工复核。

---

## 14. 全量重建触发条件

下列情况 SHOULD 触发全量重建，而不是仅靠增量：

1. 语言适配器升级；
2. 语义指纹算法升级；
3. 模块规则变更；
4. 索引结构变更；
5. 基线损坏；
6. 大规模路径重命名；
7. Git 历史缺失导致差异不可用。

全量重建是恢复一致性的手段，不应成为默认路径。

---

## 15. 诊断信息要求

任何门禁失败都 SHOULD 输出足够诊断信息，至少包括：

1. 哪个源码文件变化触发了问题；
2. 哪个镜像工件失效；
3. 变化被归类为哪一类；
4. 证据是什么；
5. 需要执行什么修复操作。

坏诊断信息会导致团队把门禁视为噪声，从而绕过系统。

---

## 16. 对 Review 的建议

Code Review SHOULD 检查以下问题：

1. 这次代码改动是否真的改变了镜像所描述的语义；
2. 镜像更新是否只改了必要段落；
3. 是否误把 C0 识别成 C1/C2/C3；
4. 是否遗漏了受影响邻域；
5. 低置信度提示是否被合理对待。

---

## 17. 与 Agent 的关系

门禁系统不是和 Agent 对立，而是 Agent 工作流的一部分。  
一个合规的 Agent 应该把以下步骤视为默认义务：

1. 修改源码前读取镜像；
2. 修改后触发镜像更新；
3. 读取漂移报告；
4. 在失败时补足遗漏，而不是绕过门禁。

---

## 18. 一句话结论

ANRSM 的同步与门禁机制不是“检查有没有一起改 Markdown”，而是 **用源码语义变化去驱动镜像失效、更新、验证和阻断，从而把镜像真正纳入仓库治理闭环**。


---

# 08. Agent 运行时契约与工作流

状态：Normative  
版本：v1  
优先级：P0

---

## 1. 目的

本文件定义 Agent 在 ANRSM 仓库中的默认读取顺序、任务路由方式、源码展开条件、写回义务和失败回退策略。

如果没有运行时契约，ANRSM 最终只会沦为“仓库里多了一层文档”。  
真正的价值在于：**Agent 的默认行为必须被镜像层重塑。**

---

## 2. 基本原则

### 原则 P1：先镜像，后源码

除非任务天然要求具体实现细节，Agent MUST 先读索引和镜像，再决定是否展开源码。

### 原则 P2：按需展开，不做默认全仓扫描

Agent SHOULD 只展开当前任务切片需要的源码文件。  
全仓扫描 MUST 被视为异常回退，而不是默认策略。

### 原则 P3：写源码即写镜像

当 Agent 修改受管源码后，MUST 触发对应镜像更新与验证。

### 原则 P4：低可信输入不得伪装成高可信

若镜像处于 `stale`、`invalid` 或 `unknown`，Agent MUST 降低信任等级，并在必要时展开源码或请求复核。

---

## 3. 标准读取顺序

v1 推荐 Agent 使用以下读取顺序：

1. 任务描述；
2. 路由索引；
3. 相关模块镜像；
4. 相关文件镜像；
5. 必要的相邻文件镜像；
6. 按需展开源码；
7. 测试、配置和契约文件；
8. 变更后重新读取漂移报告。

这个顺序体现了“最小充分上下文”原则。

---

## 4. 标准工作流

### 4.1 Phase 0：任务理解

输入：自然语言任务、Issue、PR 评论或缺陷描述。  
输出：任务切片假设，包括：

- 候选模块；
- 关键能力词；
- 风险域；
- 是否可能涉及契约变更。

### 4.2 Phase 1：任务路由

Agent MUST：

1. 查询 `routing_index`；
2. 召回候选模块和文件；
3. 读取前 N 个高相关模块镜像；
4. 生成初步受影响文件集合。

### 4.3 Phase 2：镜像理解

Agent MUST：

1. 读取受影响文件的 `file_mirror`；
2. 检查 `freshness_state`；
3. 理解责任、契约、状态、副作用和风险；
4. 判断是否已有足够信息制定修改计划。

### 4.4 Phase 3：源码展开

仅当出现以下任一条件时，Agent SHOULD 展开源码：

1. 需要精确修改控制流；
2. 需要判断具体实现细节；
3. 镜像显式提示必须展开；
4. 镜像新鲜度不足；
5. 涉及高风险边界；
6. 需要理解调用栈或测试断言。

### 4.5 Phase 4：实施修改

Agent 在修改源码时 SHOULD：

1. 限定修改边界；
2. 同步修改受影响测试；
3. 记录可能的影响面；
4. 尽量避免无关文件噪声改动。

### 4.6 Phase 5：写回与验证

Agent MUST：

1. 触发镜像更新；
2. 运行漂移检测；
3. 读取漂移报告；
4. 修复未同步工件；
5. 在门禁通过后才结束任务。

---

## 5. 任务路由协议

### 5.1 输入

任务路由至少接收：

- 自然语言任务文本；
- 可选的文件、模块或错误上下文；
- 当前仓库分支状态。

### 5.2 输出

路由结果 SHOULD 至少包含：

- 候选模块列表；
- 候选文件列表；
- 风险标签；
- 需要优先展开的文件；
- 可能受影响的测试；
- 置信度。

### 5.3 路由失败的定义

以下情况可视为路由失败：

1. 召回结果与任务明显无关；
2. 高影响文件未进入候选集；
3. 召回结果过宽，无法缩减任务切片；
4. 模块定位长期依赖人工猜测。

路由失败会直接导致 Agent 重新回到“大量读源码”的低效模式。

---

## 6. 上下文预算策略

### 6.1 预算优先级

对于多数工程任务，推荐的上下文预算顺序是：

1. 路由索引；
2. 模块镜像；
3. 文件镜像；
4. 关键源码片段；
5. 关键测试；
6. 其他邻域。

### 6.2 为什么不能默认读源码

默认全量读源码有三个问题：

1. Token 成本高；
2. 噪声多；
3. 重要边界被稀释。

ANRSM 的价值就在于把“先理解全仓”变成“先读结构化语义入口”。

### 6.3 何时允许扩大上下文

以下情况 MAY 扩大上下文：

- 跨模块变更；
- 大规模重构；
- 性能或并发问题；
- 漂移报告显示影响邻域广；
- 路由置信度低。

---

## 7. 对 Freshness State 的运行时解释

### 7.1 fresh

Agent 可以把 `fresh` 镜像作为默认高可信输入，但对高风险修改仍应按需展开源码。

### 7.2 stale

Agent MAY 使用 `stale` 镜像做粗定位，但 MUST NOT 仅依赖其做最终修改决策。  
通常应：

1. 展开源码；
2. 优先触发镜像更新；
3. 降低自动修改范围。

### 7.3 invalid

`invalid` 镜像不得作为行为判断依据。  
Agent SHOULD：

1. 重新生成镜像；
2. 若失败则回退到源码；
3. 将该工件标为修复优先项。

### 7.4 unknown

Agent 对 `unknown` 的处理 SHOULD 比 `stale` 更保守。  
原因是它不只是过期，而是“系统无法证明其可信度”。

---

## 8. 写回义务

### 8.1 写回触发条件

只要 Agent 修改了受管源码，以下写回义务 MUST 成立：

1. 更新对应文件镜像；
2. 必要时更新模块镜像；
3. 刷新相关索引；
4. 生成漂移报告；
5. 通过门禁。

### 8.2 写回的最小完成标准

以下条件必须同时满足，才算写回完成：

1. 变更文件不再触发 `material` 或 `blocking` 级漂移；
2. 相关镜像为 `fresh`；
3. 索引可用；
4. 失败码清零或在允许策略内。

### 8.3 不允许的结束方式

Agent MUST NOT 以以下方式结束任务：

- 代码改了，但镜像仍然 `stale`；
- 发现漂移后忽略不管；
- 仅更新时间戳不更新语义内容；
- 在 `blocking` 未消除时宣称任务完成。

---

## 9. 何时必须展开源码

即使镜像是 `fresh`，以下情况 Agent 仍 MUST 展开源码：

1. 需要精准修改算法、分支条件或异常路径；
2. 需要调整并发、事务、缓存或性能敏感逻辑；
3. 需要改动序列化、协议、迁移或安全逻辑；
4. 需要验证某段镜像描述是否遗漏具体实现约束；
5. 需要判断是否存在难以从镜像读出的副作用顺序。

### 9.1 何时可以只读镜像完成任务

以下任务 MAY 主要依赖镜像：

- 初步定位问题；
- 判断模块归属；
- 识别高风险入口；
- 估计影响范围；
- 制定阅读计划；
- 做交接和 Onboarding；
- 做初步 code review 预判。

---

## 10. 多 Agent 协作建议

v1 不要求复杂多 Agent 编排，但若采用多 Agent 模式，建议职责分离如下：

### Planner Agent

- 读取任务与路由索引；
- 输出候选模块、文件和计划。

### Executor Agent

- 展开必要源码；
- 修改代码和测试；
- 触发镜像更新。

### Reviewer Agent

- 检查漂移报告；
- 核对镜像是否反映真实变化；
- 识别遗漏邻域与风险。

关键要求：

- 三者 MUST 使用相同术语体系和相同镜像规范；
- 不得各自维护一套私有理解层。

---

## 11. 失败回退策略

当 ANRSM 运行时遇到问题，Agent SHOULD 按以下顺序回退：

1. 重新读取路由索引；
2. 重新生成或刷新相关镜像；
3. 展开直接相关源码；
4. 展开相邻调用方和测试；
5. 在必要时扩大范围到模块级源码邻域。

注意：  
回退是为了恢复任务可执行性，不是为了永久绕过 ANRSM。  
一旦任务完成，镜像状态仍必须被修复。

---

## 12. 提交说明建议

合规的 Agent 提交说明 SHOULD 包含：

1. 任务目标；
2. 修改的真相源工件；
3. 更新的镜像工件；
4. 漂移结果摘要；
5. 是否存在低置信度点；
6. 是否需要后续知识层更新。

这有助于 Reviewer 快速判断：

- 修改是否收敛；
- 镜像是否同步；
- 是否存在风险尾项。

---

## 13. Agent 的禁止行为

Agent MUST NOT：

1. 默认跳过镜像读取；
2. 在 `stale`/`invalid` 镜像基础上做大规模修改；
3. 修改源码后不更新镜像；
4. 手工篡改机器头字段来伪造 `fresh` 状态；
5. 在无验证的前提下用旧镜像指导跨模块重构；
6. 把知识层文档当作唯一输入而跳过镜像与源码。

---

## 14. 评估 Agent 是否“真正使用了 ANRSM”的标准

一个 Agent 只有在满足以下条件时，才算真正集成了 ANRSM：

1. 它的默认读取顺序以索引和镜像为起点；
2. 它会根据 `freshness_state` 调整行为；
3. 它会在修改后主动写回镜像；
4. 它会读取并处理漂移报告；
5. 它不会把镜像当作源码替代物。

如果一个 Agent 仍然习惯先扫源码，只把镜像当附带材料，那么它并未真正使用 ANRSM。

---

## 15. 一句话结论

ANRSM 的运行时契约，本质上是在强制 Agent 遵守一种新的仓库消费顺序：  
**先路由、再镜像、后源码、最后写回与验证。**

# 09. 验证、验收、测试与指标

状态：Normative  
版本：v1  
优先级：P0

---

## 1. 目的

本文件定义 ANRSM v1 的验证体系、测试层级、关键指标和验收标准。

ANRSM 不是一个“看上去挺合理”的系统。  
它必须能被验证、被基准测试、被审计，并且能在真实仓库工作流中稳定表现。

---

## 2. 验证维度

v1 SHOULD 至少验证以下五个维度：

1. **覆盖率（Coverage）**  
   纳入范围的源码中，有多少成功生成有效镜像。

2. **保真度（Fidelity）**  
   镜像是否保留了完成工程判断所需的关键语义。

3. **新鲜度（Freshness）**  
   镜像是否能在代码变化后及时进入正确状态并恢复为 `fresh`。

4. **可路由性（Routeability）**  
   系统是否能把任务正确映射到候选模块和文件集合。

5. **可阻断性（Gate Effectiveness）**  
   当发生未同步的 Material Change 时，门禁是否真的能阻止流入主干。

---

## 3. 测试层级

### 3.1 单元测试

应覆盖：

- 路径映射函数；
- 文档头解析与校验；
- 语义指纹计算；
- 状态机转换；
- 失败码输出；
- 配置解析；
- 单文件摘要生成规则。

### 3.2 适配器测试

应覆盖：

- AST 提取正确性；
- 导出符号识别；
- 依赖边识别；
- 契约识别；
- 低置信度回退行为。

### 3.3 集成测试

应覆盖：

- 初始化全链路；
- 增量更新链路；
- 漂移分析；
- 镜像写回；
- 索引更新；
- 门禁结果。

### 3.4 端到端测试

应覆盖真实仓库上的完整工作流：

1. 生成基线；
2. 修改源码；
3. 检测漂移；
4. 自动或半自动更新镜像；
5. 通过门禁；
6. Agent 使用镜像完成任务。

### 3.5 回归测试

每次变更以下对象时 MUST 补回归测试：

- 语义指纹算法；
- 语言适配器；
- 路由算法；
- 模块规则；
- 生成模板；
- 门禁策略。

---

## 4. 推荐指标

### 4.1 覆盖率指标

- `file_mirror_coverage`
- `module_mirror_coverage`
- `valid_mirror_ratio`

建议初始目标：

- P0 主语言与主目录覆盖率 >= 95%
- `invalid` 工件比例 <= 1%

### 4.2 新鲜度指标

- `fresh_ratio`
- `stale_ratio`
- `invalid_ratio`
- `mean_recovery_time_from_stale`

建议目标：

- 在常规增量链路下，变更相关工件应在一次更新流程后恢复为 `fresh`；
- 主干分支 `stale_ratio` SHOULD 接近 0。

### 4.3 漂移检测质量指标

- `material_change_recall`
- `material_change_precision`
- `false_block_rate`
- `missed_block_rate`

建议目标：

- 对种子用例的 Material Change 召回率 SHOULD 尽量接近 100%；
- 对 C0 误阻塞率 SHOULD 足够低，避免团队失去信任。

### 4.4 路由指标

- `top_k_module_recall`
- `top_k_file_recall`
- `route_precision_at_k`
- `mean_route_width`

建议目标：

- 常见任务的关键文件应能进入 Top-K 结果；
- `mean_route_width` 不应大到迫使 Agent 重新全仓扫描。

### 4.5 压缩指标

- `source_tokens / mirror_tokens`
- `mirror_token_budget_overrun_rate`

建议目标：

- 镜像相对于源码应有显著压缩收益；
- 压缩不能以丢失关键副作用和边界为代价。

---

## 5. 验证数据集设计

### 5.1 代表性样本

应选择覆盖以下情况的真实或准真实样本：

- 前端状态逻辑；
- 后端服务逻辑；
- API 契约；
- 数据库迁移；
- 配置接线；
- 测试入口；
- 高风险基础设施脚本。

### 5.2 种子变更集

验证漂移检测和门禁时，应预置种子变更集，至少包括：

1. 纯格式化改动；
2. 局部控制流变更；
3. 导出签名变更；
4. 配置键变更；
5. 事件名变更；
6. 数据模型变更；
7. 权限链路变更；
8. 路由文件移动。

这些种子样本能验证系统是否正确区分 C0/C1/C2/C3。

### 5.3 路由基准集

应建立一批真实任务描述，例如：

- 修复某错误提示；
- 调整鉴权逻辑；
- 修改某事件发送参数；
- 替换某缓存策略；
- 新增某 API 字段。

然后验证路由索引是否能召回相关模块和文件。

---

## 6. 验收标准

### 6.1 P0 验收标准

一个实现若要通过 v1 P0 验收，至少 MUST 满足：

1. 能对主语言主目录生成文件镜像；
2. 能为每个镜像生成标准头；
3. 能计算并使用语义指纹；
4. 能区分至少 C0/C1/C2 三类变化；
5. 能在变更后产出漂移报告；
6. 能在 PR 或等价场景下阻断未同步的 Material Change；
7. 能让 Agent 以索引和镜像为默认入口；
8. 能把 `freshness_state` 作为运行时信号暴露。

### 6.2 P1 验收标准

建议附加：

1. 模块镜像稳定生成；
2. 低置信度工件有明确复核队列；
3. 路由基准达到可接受精度；
4. 关键知识文档可从镜像自动编译。

---

## 7. 质量审计问题清单

上线前应回答以下审计问题：

1. 哪些文件仍无镜像，原因是什么？
2. 哪些高风险目录解析置信度仍低？
3. 门禁是否曾经被大量误阻塞？
4. 是否存在系统性漏检的 Material Change 类型？
5. Agent 是否真的读取了镜像，而不是继续直扫源码？
6. 知识层是否仍有脱离镜像独立维护的风险？

---

## 8. 人工评审要求

尽管 ANRSM 强调自动化，v1 仍应保留人工抽样评审：

### 8.1 抽样内容

- 高风险文件镜像；
- 低置信度镜像；
- 高频变更模块；
- 漂移误判样本；
- 路由失败样本。

### 8.2 抽样问题

1. 镜像是否遗漏了关键副作用？
2. 是否误把“功能描述”当成了“契约描述”？
3. 是否给出了明确的展开源码条件？
4. 是否存在空洞、模板化、没有判定价值的段落？
5. 是否存在术语不一致问题？

---

## 9. 发布门槛建议

在主仓正式启用门禁前，建议至少通过以下门槛：

1. 主目录镜像覆盖率达到目标；
2. 关键高风险目录误阻塞率可接受；
3. Agent 集成链路已跑通；
4. 漂移报告可读且可定位；
5. 维护者团队已理解失败码与修复方法；
6. 至少完成一轮真实项目试点。

---

## 10. 性能验收建议

v1 不必极端优化，但应保证在目标仓库规模下可用。

建议跟踪：

- 初始化耗时；
- 增量更新耗时；
- 单文件解析耗时；
- 索引更新时间；
- 门禁附加时间。

验收原则：

- 本地 pre-commit 不应慢到被团队系统性绕过；
- CI 不应慢到开发者倾向关闭门禁；
- 全量重建可慢，但必须可解释、可恢复。

---

## 11. 失败样本库

团队 SHOULD 建立失败样本库，记录：

- 漏检案例；
- 误检案例；
- 低置信度误导案例；
- 路由失败案例；
- 镜像质量差案例；
- 适配器退化案例。

每个样本 SHOULD 包含：

- 仓库快照或最小复现；
- 预期；
- 实际结果；
- 根因；
- 修复状态。

这是系统长期进化的关键资产。

---

## 12. 指标解释原则

指标不能脱离工程目标单独优化。  
例如：

- 过分追求压缩率会损害保真度；
- 过分追求召回率会扩大路由宽度；
- 过分追求零误报会增加漏报。

因此指标 MUST 组合解释，而不是单点最优。

- [ ] 模块镜像基础可用

## 14. 一句话结论

ANRSM 的验收标准不是“生成了一堆文档”，而是：  
**在真实变更和真实任务中，它能稳定生成、正确失效、正确更新、正确阻断，并实质性改变 Agent 的默认读取方式。**

# 10. 里程碑、路线图与交付计划

状态：Planning  
版本：v1  
优先级：P0

---

## 1. 目的

本文件把 ANRSM 从概念和规范推进到可实施的阶段性工程计划。  
目标不是给出唯一排期，而是定义一个对中小型到中大型仓库都适用的 **最小风险递进路径**。

核心原则：

1. 先做闭环，再做平台；
2. 先做主语言，再做多语言；
3. 先做文件镜像，再做知识层；
4. 先确保门禁有效，再扩大依赖范围。

---

## 2. 里程碑总览

推荐采用六阶段路线：

- `M0` 概念验证与规范冻结
- `M1` 文件级镜像基线
- `M2` 增量同步与漂移门禁
- `M3` 路由索引与 Agent 集成
- `M4` 模块镜像与知识层编译
- `M5` 规模化、可观测性与多语言扩展

---

## 3. M0：概念验证与规范冻结

### 目标

把“我们要做什么”从想法变成稳定规范与试点边界。

### 必做事项

1. 冻结术语体系；
2. 冻结 v1 边界；
3. 选定试点仓库；
4. 选定第一主语言；
5. 选定镜像目录结构；
6. 选定文档头 schema；
7. 选定 Material Change 的初始分类。

### 交付物

- 设计母稿；
- 术语与规范语言；
- 路径映射规则；
- 初版配置文件；
- 试点仓库清单；
- 风险登记表。

### 退出条件

- 主要争议项有明确决策；
- 至少一个试点仓库和一条主链路可实现；
- 团队接受“源码改动必须触发镜像同步”。

### 风险

- 目标膨胀；
- 试点仓库选得过大或过乱；
- 未区分 v1 和长期愿景。

---

## 4. M1：文件级镜像基线

### 目标

建立从源码到文件镜像的第一条可运行闭环。

### 必做事项

1. 实现 Scope Manager；
2. 实现主语言适配器；
3. 实现统一语义模型最小子集；
4. 实现文件级镜像生成；
5. 实现头字段写入；
6. 生成基线指纹；
7. 生成基础索引雏形。

### 交付物

- `mirror/file/**`
- `mirror/index/symbol_index.json` 初版
- `mirror/index/dependency_index.json` 初版
- 基线生成 CLI
- 首轮覆盖率报告

### 退出条件

- 主目录文件镜像覆盖率达到目标；
- 高风险文件可生成有效镜像；
- 头字段完整；
- 生成过程可重复。

### 风险

- 主语言解析能力不足；
- 生成内容空洞；
- 路径映射不稳定。

---

## 5. M2：增量同步与漂移门禁

### 目标

让系统从“会生成”升级为“会维护、会阻断漂移”。

### 必做事项

1. 接入 Git diff；
2. 实现语义指纹；
3. 实现 C0/C1/C2/C3 分类；
4. 实现漂移报告；
5. 接入 pre-commit 或 CI 门禁；
6. 实现失败码和诊断输出。

### 交付物

- 增量更新 CLI
- `mirror/reports/drift_report.json`
- CI 工作流或 pre-commit hook
- 失败码说明文档
- 种子变更测试集

### 退出条件

- 常见 Material Change 可被识别；
- 未同步的 C2/C3 能阻塞；
- C0 误阻塞率可接受；
- 增量链路性能可用。

### 风险

- 误报过高导致团队绕过门禁；
- 漏报导致镜像失去信任；
- 增量邻域扩展不稳定。

---

## 6. M3：路由索引与 Agent 集成

### 目标

让 Agent 真正以镜像为默认入口，而不是让镜像只停留在文件层。

### 必做事项

1. 完善 `routing_index`；
2. 定义 Agent 读取顺序；
3. 定义 Expansion Read 触发条件；
4. 实现任务到模块/文件的基础路由；
5. 将镜像更新纳入 Agent 工作流；
6. 建立路由基准集。

### 交付物

- `mirror/index/routing_index.json`
- Agent 集成指南
- 路由基准测试结果
- Agent 运行时日志规范

### 退出条件

- 多数试点任务可先经镜像完成定位；
- Agent 默认不再全仓读源码；
- 修改后会自动更新镜像并通过门禁；
- Reviewer 能看到镜像同步结果。

### 风险

- 路由过宽；
- Agent 仍旧惯性跳过镜像；
- 镜像质量不足以支持定位。

---

## 7. M4：模块镜像与知识层编译

### 目标

把文件级能力提升为模块级理解和人类可用知识层。

### 必做事项

1. 定义模块规则；
2. 生成模块镜像；
3. 设计知识层模板；
4. 从镜像编译架构总览和功能边界文档；
5. 建立人工补充机制；
6. 为交接与排障生成最小知识集。

### 交付物

- `mirror/module/**`
- `knowledge/architecture/**`
- `knowledge/features/**`
- `knowledge/operations/**`
- 模块级验收报告

### 退出条件

- 模块级责任与边界表达清晰；
- 人类接手可以优先读知识层而非到处翻源码；
- 知识层来源可追溯；
- 未形成第二真相源。

### 风险

- 知识层开始脱离镜像独立维护；
- 模块边界定义混乱；
- 模块镜像只是文件摘要拼接。

---

## 8. M5：规模化、可观测性与多语言扩展

### 目标

把试点能力推进为更可运维、更可治理、更可扩展的平台级能力。

### 必做事项

1. 增加多语言适配器；
2. 引入缓存与性能优化；
3. 增强可观测性；
4. 引入失败样本库；
5. 完善策略分级；
6. 建立适配器升级和全量重建流程。

### 交付物

- 多语言支持矩阵
- 指标看板或报告
- 失败样本库
- 全量重建运维文档
- 版本迁移策略

### 退出条件

- 至少两种主要语言稳定支持；
- 全量与增量链路可运维；
- 误报/漏报持续受控；
- 团队对系统有持续信任。

### 风险

- 过早平台化；
- 复杂度上升快于价值释放；
- 适配器维护成本失控。

---

## 9. 各里程碑的依赖关系

```text
M0 -> M1 -> M2 -> M3 -> M4 -> M5
```

强依赖说明：

- 没有 M1，就不存在可验证镜像；
- 没有 M2，镜像无法进入治理闭环；
- 没有 M3，Agent 不会真正改变工作方式；
- 没有 M4，人类知识层无法共享同一语义基座；
- M5 只能在前四者稳定后才值得投入。

## 11. 推荐实施顺序

如果资源有限，建议按以下具体顺序实施：

1. 选试点仓库与主语言；
2. 落地术语、路径映射和头 schema；
3. 做文件镜像生成；
4. 做语义指纹与漂移报告；
5. 接 CI 门禁；
6. 接 Agent 读取顺序；
7. 做模块镜像；
8. 做知识层；
9. 扩多语言；
10. 做平台化与指标系统。

这样做的原因是：  
它优先保证“价值闭环”，而不是“架构看上去很完整”。

---

## 12. 最小 PoC 建议

如果团队要用最短路径验证概念，PoC SHOULD 只包含：

1. 一个真实仓库；
2. 一个主语言；
3. 文件镜像生成；
4. 语义指纹；
5. 漂移报告；
6. 一个 CI 阻塞规则；
7. 一个 Agent 定位任务链路。

PoC 不应一开始就做：

- UI；
- 多语言；
- 知识平台；
- 分布式服务；
- 复杂权限系统。

---

## 13. 交付节奏建议

每个里程碑都应产生以下三类输出：

1. **规范层输出**  
   文档、ADR、schema、策略说明。

2. **系统层输出**  
   CLI、适配器、生成器、索引器、门禁链路。

3. **验证层输出**  
   指标、测试结果、失败样本、抽样评审记录。

这样可以避免“代码写了很多，但没有形成可治理能力”。

---

## 14. 里程碑退出评审问题

每个里程碑结束时，应回答以下问题：

1. 该阶段新增能力是否形成闭环？
2. 是否新增了新的不透明复杂度？
3. 是否开始偏离“源码唯一真相源”原则？
4. Agent 的默认行为是否真的被改变？
5. 下一阶段是否建立在稳定基础上？

只要其中有两项回答不清楚，就不应轻易进入下一阶段。

---

## 15. 一句话结论

ANRSM 的正确落地顺序，不是一次性造一个大而全平台，而是：  
**先把文件镜像、漂移门禁和 Agent 读取顺序这三个核心闭环打通，再逐步扩展到模块理解和人类知识层。**

# 11. 反模式、失败模式与决策记录指南

状态：Normative / Advisory  
版本：v1  
优先级：P0

---

## 1. 目的

本文件定义 ANRSM 在设计和落地中最常见的反模式、失败模式以及推荐的架构决策记录方式。

反模式文档的价值在于：  
**提前定义哪些路绝对不该走，避免团队在实现过程中用局部捷径破坏系统长期可维护性。**

---

## 2. 一级反模式

### 反模式 A1：把 ANRSM 当作文档生成器

表现：

- 团队只关心能否“批量出 Markdown”；
- 不做指纹、不做漂移、不做门禁；
- Agent 继续默认直扫源码。

问题本质：

- 没有治理闭环；
- 没有运行时契约；
- 没有新鲜度。

纠偏要求：

- 立刻把重点转回“镜像是否可验证、可同步、可阻断漂移”。

---

### 反模式 A2：Prompt 直接生成镜像事实

表现：

- 没有结构化提取；
- 让 LLM 直接看源码输出镜像；
- 结果写得像文档，但事实基础不稳定。

风险：

- 幻觉；
- 不可复现；
- 无法做可靠差异分析；
- 难以做自动门禁。

纠偏要求：

- 先建立结构化事实层；
- Prompt 只负责压缩、解释和重组。

---

### 反模式 A3：让镜像成为第二真相源

表现：

- 工程师习惯先改镜像，再决定要不要改代码；
- 人类知识层长期独立维护；
- 线上事实与镜像长期背离。

风险：

- 决策污染；
- Review 失真；
- Agent 被错误上下文误导。

纠偏要求：

- 强化“源码唯一真相源”；
- 只允许镜像做派生，不允许做主导。

---

### 反模式 A4：只做路径联动，不做语义漂移

表现：

- `src/**` 改了就要求 `mirror/**` 也改；
- 但并不检查镜像是否真的更新了关键语义；
- 格式化改动也频繁触发失败。

风险：

- 团队会把门禁视为噪声；
- 真正的 Material Change 可能仍然漏检。

纠偏要求：

- 引入语义指纹与分类；
- 用漂移报告驱动门禁，而不是只看文件是否同改。

---

### 反模式 A5：默认全量重建

表现：

- 每次提交都全量重扫全仓；
- 增量能力长期不上线；
- 本地与 CI 都变得缓慢难用。

风险：

- 团队绕过系统；
- 成本高；
- 规模扩展失败。

纠偏要求：

- 以差异驱动为默认路径；
- 全量重建只作为恢复或升级手段。

---

### 反模式 A6：镜像正文写成“漂亮空话”

表现：

- 段落很多，但没有责任边界、状态、副作用和风险；
- 全是“这个文件负责处理某某逻辑”之类低信息密度句子。

风险：

- Agent 和人类都无法真正决策；
- Token 节约了，但信息价值也消失了。

纠偏要求：

- 正文必须写关键契约、控制流、副作用、不变量和展开条件。

---

### 反模式 A7：忽略“何时必须展开源码”

表现：

- 团队假设镜像足够强大，可以替代具体实现阅读；
- Agent 在复杂变更上只看镜像不看源码。

风险：

- 精确修改失败；
- 影响面漏判；
- 高风险变更引入回归。

纠偏要求：

- 每个镜像明确给出展开条件；
- 高风险变更默认要求源码展开。

---

### 反模式 A8：没有低置信度机制

表现：

- 解析不稳定时仍输出高自信镜像；
- 没有 `confidence_band`；
- 门禁无法区分高信与低信工件。

风险：

- 系统表面正常，实际在关键时刻误导最严重。

纠偏要求：

- 显式输出置信度；
- 在策略层对低置信度做保守处理。

---

## 3. 二级反模式

### 反模式 B1：术语不固定

表现：

- 团队一会儿说 mirror，一会儿说 doc，一会儿说 note；
- knowledge 和 mirror 混用；
- drift 和 change 混用。

结果：

- 规范不可执行；
- 工具接口混乱；
- Agent prompt 难以稳定。

纠偏：

- 术语表必须先冻结；
- 代码、文档、CLI、报告统一命名。

### 反模式 B2：模块边界靠感觉定义

表现：

- 模块命名不稳定；
- 同一文件今天属于 A，明天属于 B；
- 模块与业务或维护边界脱节。

结果：

- 模块镜像没有长期价值；
- 路由索引精度下降；
- 知识层变得混乱。

纠偏：

- 模块边界要与业务责任、依赖关系和维护责任一致。

### 反模式 B3：把所有文件都镜像到同一强度

表现：

- 低价值样板代码和高风险核心文件使用相同镜像粒度；
- 系统在低价值区域浪费太多成本。

结果：

- 整体 ROI 下降；
- 维护成本过高。

纠偏：

- 建立工件分层和优先级；
- 高价值、高风险、高频变更区域优先。

### 反模式 B4：人类知识层提前平台化

表现：

- 在文件镜像和门禁没稳定前，先做漂亮 UI 和知识门户。

结果：

- 表层体验很好，但底层事实不稳定；
- 平台放大错误。

纠偏：

- 先把底层镜像和漂移治理打稳，再做知识层体验。

---

## 4. 常见失败模式

### F1：覆盖率高，但保真度低

症状：

- 大多数文件都有镜像；
- 但镜像看不出关键副作用和边界；
- Agent 仍然频繁误改。

根因：

- 过度追求覆盖；
- 摘要模板空洞；
- 结构化事实层薄弱。

修复：

- 提升高价值文件的语义抽取深度；
- 引入人工抽样审计。

### F2：门禁严格，但误报过高

症状：

- 团队频繁被阻塞；
- 大量失败来自 C0；
- 开发者开始绕过或关闭门禁。

根因：

- 语义指纹太敏感；
- 缺少 C0/C1 分类边界；
- 缺少保守豁免策略。

修复：

- 重新设计指纹排噪；
- 调整策略分级；
- 加入失败样本回归。

### F3：Agent 已接入，但行为未改变

症状：

- 系统里有镜像；
- 但 Agent 仍先扫源码，再偶尔看镜像。

根因：

- 路由索引无效；
- 镜像质量不够高；
- 运行时契约没有被硬编码到 Agent 工作流。

修复：

- 把读取顺序写进 Agent runtime；
- 把镜像读取变成默认步骤；
- 评估实际上下文消费路径。

### F4：知识层逐渐失真

症状：

- 架构文档越来越多；
- 但与真实代码越走越远。

根因：

- 知识层脱离镜像演化；
- 没有来源追踪；
- 没有更新策略。

修复：

- 强制知识层保留 `source_artifacts`；
- 定期从镜像重编译；
- 对关键知识文档引入 freshness。

### F5：系统只能在一个仓库一次性工作

症状：

- 第一次初始化可用；
- 后续升级、迁移、适配器变化就失控。

根因：

- 缺少版本化；
- 缺少重建策略；
- 缺少 schema 管理。

修复：

- 版本化协议、指纹和适配器；
- 建立升级和重建流程。

---

## 5. 设计评审时必须追问的问题

每次设计评审至少应追问：

1. 这项设计是否强化了“源码唯一真相源”？
2. 它是否提升了而不是削弱了漂移可观测性？
3. 它会不会让 Agent 又回到默认全仓读源码？
4. 它引入的新复杂度是否可审计、可重建？
5. 它是否把短期方便变成长期双真相源？

---

## 6. ADR（架构决策记录）建议

ANRSM 项目 SHOULD 使用 ADR 记录关键决策。  
原因：

- 术语和策略一旦漂移，系统会迅速失去一致性；
- 里程碑推进中会反复碰到“为什么当初这样设计”。

### 6.1 适合写 ADR 的主题

- 路径映射规则；
- 主语言优先级；
- 模块边界定义；
- 语义指纹算法；
- 低置信度策略；
- 全量重建触发条件；
- 是否纳入测试/配置/生成产物；
- 知识层生成策略。

### 6.2 ADR 模板建议

```markdown
# ADR-00X: <决策标题>

## 背景
问题是什么，为什么必须现在决定。

## 决策
最终采用什么方案。

## 备选方案
还有哪些方案被考虑过。

## 取舍
为什么放弃其他方案。

## 影响
对架构、流程、性能、团队协作有什么影响。

## 风险
该决策最可能导致的失败模式是什么。

## 复核条件
在什么情况下应重新打开此决策。
```

---

## 7. 审查反模式的操作方式

建议在每个里程碑结束时，用反模式文档做一次逆向审查：

1. 本阶段是否落入了任何一级反模式？
2. 是否出现二级反模式萌芽？
3. 是否有新的失败样本未进入回归集？
4. 是否需要新增 ADR 解释边界变化？

这样做的价值是：  
它能防止团队在工程推进中“逐渐偏离而不自知”。

---

## 8. 最危险的三个信号

如果项目出现以下三个信号，说明 ANRSM 已经开始偏离正确方向：

1. 团队开始直接手改镜像并以此代替源码事实；
2. Agent 依然默认大面积读源码，镜像只做装饰；
3. 门禁经常被关闭或绕过。

一旦出现这三个信号中的任意两个，项目必须停下来做一次规范回正。

---

## 9. 一句话结论

ANRSM 最常见的失败，不是技术做不到，而是团队把它悄悄做成了：  
**一套没有语义闭环、没有门禁、没有运行时契约的“高级文档生成器”。**
