# Phase 999.1: MCP Server — Summary

**One-liner:** MCP Server exposing WTCD's semantic analysis capabilities via rmcp 1.2.0 stdio transport, integrated into the CLI with `wtcd mcp` and `wtcd mcp-install` commands.

## What Was Built

### Plan 01: wtcd-mcp Crate

Created a new `wtcd-mcp` crate in the Cargo workspace implementing an MCP Server using the rmcp 1.2.0 SDK.

**WtcdMcpServer** implements `ServerHandler` with `ToolRouter<Self>` pattern:

- **wtcd_parse** (完整实现): Parses a single TS/JS file via `wtcd-adapters` API. Accepts `file_path` (required) and `language` (optional). Returns `FileResult` JSON with exports, imports, signatures, side effects.

- **wtcd_run** (完整实现): Scans entire project via `wtcd-scope` + `wtcd-adapters` pipeline. Accepts optional `project_root`. Returns `RunOutput` JSON with files list and summary statistics.

- **wtcd_check** (stub): Returns "not implemented" — Phase 3 capability.

- **wtcd_mirror** (stub): Returns "not implemented" — Phase 2 capability.

All tool implementations use `tokio::task::spawn_blocking()` to bridge the synchronous wtcd-core APIs into the async MCP server runtime.

### Plan 02: CLI Integration

Extended `wtcd-cli` with two new subcommands:

- **`wtcd mcp`**: Starts the MCP Server on stdio transport. Logs to stderr, JSON-RPC on stdout. Uses `#[tokio::main]` async main.

- **`wtcd mcp-install --agent <name>`**: Generates MCP configuration snippets. Supports `claude` (outputs `~/.claude.json` format) and `cursor` (outputs `.cursor/mcp.json` format).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] rmcp 1.2.0 requires Rust 1.88+, project pinned to 1.85**
- **Found during:** Plan 01, Task 1
- **Issue:** `darling@0.23.0` (transitive dep of rmcp 1.2.0) requires `rustc 1.88.0`
- **Fix:** Updated `rust-toolchain.toml` from `channel = "1.85"` to `channel = "1.92"`
- **Files modified:** `rust-toolchain.toml`
- **Reason:** User explicitly specified rmcp 1.2.0; 0.16 would work with 1.85

**2. [Rule 3 - Blocking] rmcp 1.x API differs from 0.16 examples in plan**
- **Found during:** Plan 01, Task 2
- **Issue:** rmcp 1.x uses non-exhaustive structs (ServerInfo, Implementation), builder pattern, and different import paths
- **Fix:** Used `ServerInfo::new(capabilities).with_*()` builder, `Implementation::new(name, version)`, `rmcp::handler::server::wrapper::Parameters`, `rmcp::handler::server::router::tool::ToolRouter`
- **Files modified:** `crates/wtcd-mcp/src/server.rs`, all tool files

**3. [Rule 3 - Blocking] `CallToolResult::error_text` doesn't exist in rmcp 1.x**
- **Found during:** Plan 01, Task 2
- **Issue:** rmcp 1.x uses `CallToolResult::error(vec![Content::text(msg)])` instead of `error_text(msg)`
- **Fix:** Changed all stub tool implementations and error paths
- **Files modified:** `crates/wtcd-mcp/src/tools/check.rs`, `crates/wtcd-mcp/src/tools/mirror.rs`, `crates/wtcd-mcp/src/tools/parse.rs`, `crates/wtcd-mcp/src/tools/run.rs`

**4. [Rule 3 - Blocking] ServiceExt import path in rmcp 1.x**
- **Found during:** Plan 02, Task 1
- **Issue:** `rmcp::ServiceExt` not found; must use `rmcp::service::ServiceExt`
- **Fix:** Updated import in `crates/wtcd-cli/src/commands/mcp.rs`
- **Files modified:** `crates/wtcd-cli/src/commands/mcp.rs`

### Architectural Decisions

**1. Tool definitions centralized in server.rs**
- **Reason:** rmcp `#[tool_router]` macro requires all `#[tool]` methods in the same `#[tool_router]` impl block
- **Impact:** Tool logic is delegated to handler functions in `tools/*.rs`, but the method signatures are defined in `server.rs`
- **Alternative considered:** Custom tool routing — rejected as it defeats the purpose of rmcp macros
