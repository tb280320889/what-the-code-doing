# Phase 999.1: MCP Server 探索 - Context

**Gathered:** 2026-03-21
**Status:** Ready for planning

<domain>
## Phase Boundary

重新评估 Agent 集成方式，从纯 CLI 驱动扩展为 MCP Server + CLI 双轨制。MCP Server 通过 stdio 传输暴露 WTCD 核心能力为 MCP Tools，让任何支持 MCP 协议的 Agent（Claude Code、Cursor、Cline 等）自动发现并使用 WTCD，无需手动调用 CLI。

核心引擎（wtcd-core、wtcd-adapters、wtcd-scope）不变，MCP 是新的交付层。

</domain>

<decisions>
## Implementation Decisions

### MCP 与 CLI 的关系
- **D-01:** MCP 和 CLI 完全平等共存，共享 wtcd-core/wtcd-adapters/wtcd-scope 等核心 crate
- **D-02:** MCP Server 直接调用 Rust API，不走 CLI 子进程（无序列化/进程启动开销）
- **D-03:** 仅支持 stdio 传输，不提供 HTTP/SSE 后端服务（本地模式，无常驻进程）
- **D-04:** 兼容所有支持 MCP stdio 传输的 Agent（Claude Code、Cursor、Cline、Continue 等）
- **D-05:** 需要新增 MCP 规范文档（docs/12），定义工具契约

### MCP 工具表面设计
- **D-06:** 4 个核心 MCP Tools：`wtcd_parse`、`wtcd_route`、`wtcd_check`、`wtcd_mirror`
- **D-07:** stdio MVP 阶段，所有数据通过 Tool 返回值传递，不使用 MCP Resources（本地场景下 Resource URI 无额外价值）
- **D-08:** 配置来源复用 `anrsm.yaml`，MCP Server 通过工作目录自动发现项目配置
- **D-09:** Tool 参数和返回值 schema 从 Rust 类型派生（rmcp `#[tool]` 宏 + schemars），保持 DRY
- **D-10:** MVP 实现 `wtcd_parse` + `wtcd_run`（Phase 1 对等能力），其余工具注册但实现 stub

### Agent 发现与连接方式
- **D-11:** 提供专用安装命令 `wtcd mcp-install --agent <name>`，自动生成 Agent 配置片段
- **D-12:** 单一二进制含 MCP 模式（`wtcd mcp` 子命令），`cargo install wtcd` 即可，无额外分发产物
- **D-13:** MCP Server 生命周期 = 预加载 + 按需：启动时读取 cwd 的 anrsm.yaml 并预加载索引，首次 Tool 调用无需等待
- **D-14:** 单实例多项目：首次访问新项目路径时自动加载其配置和索引

### 路线图影响
- **D-15:** 999.1 从 backlog 提升为正式 phase（插入 Phase 1 和 Phase 2 之间）
- **D-16:** Phase 2-4 计划不变，MCP 是独立并行 track
- **D-17:** Phase 2-4 的能力在 MCP 完成后自动通过 MCP 工具可用（parse → mirror → check → route 逐步实现）
- **D-18:** 新增规范文档 `docs/12_MCP_Server_Contract.md`，走 ADR 流程

### Agent's Discretion
- rmcp crate 的具体版本和 feature flags
- wtcd_parse 和 wtcd_run 的精确 Tool 参数定义
- mcp-install 支持的 Agent 列表和配置文件格式
- 内存中索引缓存的数据结构和失效策略
- 多项目的配置发现顺序（cwd → 参数 → 环境变量）

</decisions>

<code_context>
## Existing Code Insights

### 可复用资产
- **wtcd-core crate** — 核心类型（ArtifactId, ConfidenceBand, ExportInfo, ImportEdge, SideEffect 等）直接被 MCP Server 复用
- **wtcd-adapters crate** — TS/JS tree-sitter 解析适配器，MCP wtcd_parse 工具直接调用
- **wtcd-scope crate** — 配置解析 + 文件扫描，MCP wtcd_run 工具直接调用
- **anrsm.yaml 配置格式** — MCP Server 复用同一配置，无需额外配置文件
- **JSON Schema (mirror-header, drift-report)** — 可作为 MCP Tool 返回值 schema 的参考

### 已锁定的技术模式
- Cargo workspace 多 crate 结构 — MCP Server 作为新 crate `wtcd-mcp` 加入 workspace
- Trait-based 适配器 — MCP Server 通过 trait 调用解析能力，不耦合具体适配器实现
- Pipeline 架构（Config → Scope → Scan → Parse）— MCP wtcd_run 复用同一 pipeline

### 集成点
- `Cargo.toml` workspace — 添加 `wtcd-mcp` crate，依赖 rmcp + 现有 wtcd-* crates
- `wtcd-core/src/types.rs` — MCP Tool 返回值直接使用这些类型（serde 派生已就绪）
- `wtcd-cli/src/commands/run.rs` — MCP wtcd_run 的核心逻辑参考，但直接调用 Rust API 而非 CLI

### 注意事项
- rmcp 依赖 tokio async runtime — wtcd-mcp 需要 tokio 依赖
- rmcp 的 `#[tool]` 宏需要 schemars 用于 JSON Schema 生成
- 现有 crate 没有 async 代码 — MCP Server 层面需要 async 包装同步调用
- MSRV 对齐：rmcp 当前版本的 MSRV 需要与项目 MSRV 1.85 兼容

</code_context>

<specifics>
## Specific Ideas

- 用户原话："还是要做成MCP, MCP 比 cli 更有优势, 会自动加载到 agent的上下文中,看看如何开发"
- MCP 的核心优势是自动注册到 Agent 上下文 — Agent 不需要手动调用 CLI，工具自动可用
- stdio 模式 = Agent 启动时 fork wtcd mcp 进程，通过 stdin/stdout JSON-RPC 通信
- mcp-install 命令参考：类似 `gh auth setup-git` 自动生成 Git credential helper 配置

</specifics>

<deferred>
## Deferred Ideas

- HTTP/SSE 传输支持 — 留到 stdio 版本验证后的后续阶段
- MCP Resources 暴露（镜像文件、路由索引作为可订阅资源）— stdio 阶段不需要
- 多实例/多租户支持 — 单用户本地场景不需要
- MCP Prompts 暴露（预定义的 Agent 工作流模板）— 有价值但不在 MVP 范围

</deferred>
