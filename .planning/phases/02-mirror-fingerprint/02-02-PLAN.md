<objective>
Implement the dual fingerprint engine: source_fingerprint (changes on any text edit) and semantic_fingerprint (stable across whitespace/comment/formatting changes).
</objective>

<implementation>
<step number="1" name="source-fingerprint">
<action>
1. Create `crates/wtcd-mirror/src/fingerprint.rs` — implement `compute_source_fingerprint`:
   - Input: raw source string (as bytes)
   - Algorithm: SHA-256 of raw source bytes (no normalization for source — any byte change must produce different hash)
   - Output: `"sha256:<hex>"` prefix format matching `mirror-header.schema.json` pattern
   - Use `sha2` crate: `Sha256::digest(source.as_bytes())`
   - Format as lowercase hex with `sha256:` prefix

2. Implement `compute_source_fingerprint_from_path(path: &Path) -> Result<String>`:
   - Reads file from disk, calls `compute_source_fingerprint`
   - Returns error if file not readable

3. Write unit tests:
   - Empty string → deterministically produces known hash
   - Single char change → different hash
   - Same input → same hash (deterministic)
   - Prefix format: starts with `"sha256:"` followed by 64 hex chars
</action>

<step number="2" name="semantic-fingerprint">
<action>
1. In `crates/wtcd-mirror/src/fingerprint.rs`, implement `compute_semantic_fingerprint`:
   - Input: `&FileResult` (from wtcd-core)
   - Normalize process (D-06, D-07):
     a. Build a canonical semantic object from FileResult fields:
        - `exports`: sorted by name, each entry = `{name, kind}` only (no line numbers — line changes are non-semantic)
        - `imports`: sorted by source path, each entry = `{source, symbols: sorted, kind}`
        - `signatures`: sorted by name, each entry = `{name, parameters, return_type}`
        - `side_effects`: sorted by `{kind, target}` (no line numbers)
        - `confidence`: included as-is
     b. Serialize with `serde_json::to_string` (deterministic key ordering ensured by sorted insertion)
     c. SHA-256 of canonical JSON string
     d. Output: `"sha256:<hex>"` prefix

2. Define internal `CanonicalSemantic` struct for the normalized object:
   ```rust
   #[derive(Serialize)]
   struct CanonicalSemantic {
       exports: Vec<CanonicalExport>,
       imports: Vec<CanonicalImport>,
       signatures: Vec<CanonicalSignature>,
       side_effects: Vec<CanonicalSideEffect>,
       confidence: String,
   }
   ```
   - All inner types derive `Serialize` with fields sorted alphabetically by name
   - Vec fields are sorted by primary key before serialization

3. Key normalization rules (D-06):
   - Line numbers excluded (whitespace/formatting changes line numbers)
   - Exports sorted alphabetically by name
   - Imports sorted by source path; symbols within each import sorted alphabetically
   - Side effects sorted by kind then target
   - Signatures sorted by name

4. Write unit tests:
   - Same FileResult → same semantic fingerprint (deterministic)
   - Different exports → different fingerprint
   - Line number changes only → SAME fingerprint (semantic stability)
   - Reordered imports → same fingerprint (sorted internally)
   - Different confidence → different fingerprint
   - Prefix format: starts with `"sha256:"` followed by 64 hex chars
</action>
