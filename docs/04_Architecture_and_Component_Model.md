# 04. 架构与组件模型

状态：Normative  
版本：v1  
优先级：P0

---

## 1. 目的

本文件定义 ANRSM 的核心架构、组件责任、控制面/数据面划分、初始化与增量更新流水线，以及语言适配和生成策略。

本文件的目标不是给出唯一实现，而是给出 **必须成立的架构约束** 与 **推荐的实现形态**，使不同团队能够在不破坏规范的前提下完成具体工程落地。

---

## 2. 总体架构

ANRSM v1 推荐采用 **仓库内文件型系统 + 增量计算流水线** 的架构。  
逻辑上分为两部分：

- **控制面（Control Plane）**：定义范围、策略、模块边界、语言适配、门禁强度和生成规则；
- **数据面（Data Plane）**：扫描源码、提取语义、生成镜像、构建索引、检测漂移、输出报告。

### 2.1 控制面包含

1. 范围配置；
2. 排除规则；
3. 模块映射规则；
4. 语言适配器配置；
5. 漂移策略；
6. 门禁策略；
7. 生成模板与压缩规则；
8. 知识层编译策略。

### 2.2 数据面包含

1. 仓库扫描；
2. 结构化提取；
3. 语义归一化；
4. 指纹计算；
5. 镜像生成；
6. 索引构建；
7. 漂移分析；
8. 验证与写回；
9. 报告输出。

## 3. 组件清单与职责

### 3.1 Scope Manager

职责：

- 定义镜像范围；
- 维护纳入/排除路径；
- 定义工件分类；
- 识别模块归属规则。

输入：

- `anrsm.yaml` 或等价配置；
- 仓库目录树；
- 可选的模块映射规则。

输出：

- 受管文件清单；
- 文件类型分类；
- 模块归属表。

约束：

- Scope Manager MUST 是确定性的；
- 相同配置和相同仓库状态 MUST 产生相同范围结果。

### 3.2 Repository Scanner

职责：

- 枚举受管文件；
- 读取 Git 状态或差异集；
- 为下游提供稳定的输入记录。

输入：

- Scope Manager 输出；
- Git diff 或全量扫描触发条件。

输出：

- 文件清单；
- 候选变更集；
- 文件元信息（路径、时间戳、大小、hash 等）。

### 3.3 Language Adapter

职责：

- 按语言解析源码；
- 提取 AST、导入导出、符号、签名、注解、配置结构；
- 返回统一的语言中立表示所需原始信息。

输入：

- 单个受管文件；
- 语言配置；
- 可选的项目级解析上下文（tsconfig、build graph 等）。

输出：

- 语言特定解析结果；
- 解析失败信息；
- 置信度与限制说明。

约束：

- 主语言适配器 SHOULD 优先使用 AST 或等价结构化解析；
- 仅基于正则的适配器 MUST 标记低置信度。

### 3.4 Semantic Extractor

职责：

- 将语言特定结果归一到统一语义模型；
- 提取符号、接口、状态、依赖、副作用、关键路径和候选风险点；
- 提供生成镜像所需的结构化事实。

输入：

- Language Adapter 输出；
- 文件分类信息；
- 可选的跨文件上下文。

输出：

- 标准化语义对象；
- 语义邻接边；
- 语义置信度。

### 3.5 Fingerprint Engine

职责：

- 计算 Source Fingerprint；
- 计算 Semantic Fingerprint；
- 支持差异比较和新鲜度判断。

输入：

- 源码内容；
- 标准化语义对象；
- 指纹策略配置。

输出：

- `source_fingerprint`
- `semantic_fingerprint`

约束：

- Semantic Fingerprint SHOULD 稳定排除非语义噪声；
- 指纹算法升级 MUST 可被版本化记录。

### 3.6 Mirror Generator

职责：

- 根据标准模板和结构化事实生成文件镜像；
- 聚合文件镜像生成模块镜像；
- 写入标准头和正文分节。

输入：

- 标准化语义对象；
- 指纹；
- 模板规则；
- 旧镜像（用于稳定更新和 diff）。

输出：

- 文件镜像；
- 模块镜像；
- 可选的知识层素材片段。

约束：

- Generator MUST 保持章节顺序稳定；
- 头字段 MUST 可机读；
- 正文 SHOULD 遵循压缩规范，避免随意 prose 漂移。

### 3.7 Index Builder

职责：

- 构建路由索引、符号索引、依赖索引；
- 提供任务到工件集合的检索基础。

输入：

- 文件镜像；
- 模块镜像；
- 语义图。

输出：

- `routing_index`
- `symbol_index`
- `dependency_index`

### 3.8 Drift Analyzer

职责：

- 比较新旧语义状态；
- 识别 Non-material Change 与 Material Change；
- 标记受影响镜像；
- 输出漂移级别和建议动作。

输入：

- 变更集；
- 新旧指纹；
- 新旧索引；
- 门禁策略。

输出：

- 漂移报告；
- 受影响工件列表；
- 阻塞建议。

### 3.9 Policy Gate

职责：

- 根据策略决定是否允许提交、推送、合并或发布继续；
- 提供可读错误信息和修复提示。

输入：

- 漂移报告；
- 策略配置；
- 当前执行场景（本地、CI、PR、主干合并）。

输出：

- Pass / Fail 结果；
- 失败码；
- 修复提示。

### 3.10 Knowledge Compiler

职责：

- 从文件镜像、模块镜像和索引派生人类知识层；
- 输出架构说明、功能边界、交接材料和 Runbook 草稿。

输入：

- 机器语义镜像层；
- 知识模板；
- 可选的人工补充段。

输出：

- 知识文档集合。

约束：

- Knowledge Compiler 生成的工件 MUST 保留来源；
- 人类知识层 MUST NOT 失去到镜像的可追溯关系。

## 4. 两条核心流水线

### 4.1 初始化流水线

初始化阶段的目标是建立一个可验证的基线，而不是只生成一次文档。

流程：

1. Scope Manager 解析仓库范围；
2. Repository Scanner 做全量清单；
3. Language Adapter 对纳入文件解析；
4. Semantic Extractor 归一语义模型；
5. Fingerprint Engine 计算初始指纹；
6. Mirror Generator 生成文件镜像与模块镜像；
7. Index Builder 生成索引；
8. Drift Analyzer 产出基线一致性报告；
9. Policy Gate 检查初始覆盖率和完整性。

初始化结果必须可重建、可审计、可作为后续增量更新的稳定起点。

### 4.2 增量流水线

增量阶段的目标是只重算受影响邻域。

流程：

1. Repository Scanner 从 Git 获取变更集；
2. Scope Manager 判断哪些工件受影响；
3. Language Adapter 重新解析变更文件及必要邻域；
4. Semantic Extractor 更新语义对象；
5. Fingerprint Engine 计算新指纹；
6. Drift Analyzer 对比新旧状态；
7. Mirror Generator 更新受影响镜像；
8. Index Builder 更新相关索引分片；
9. Policy Gate 决定是否放行。

增量模式 SHOULD 是默认路径。  
只有以下情况 MAY 触发全量重建：

- 语言适配器升级；
- 指纹算法升级；
- 范围配置重大变化；
- 模块规则变化；
- 索引损坏或无法证明一致性。

## 5. 控制面设计

### 5.1 配置项分类

控制面配置 SHOULD 至少包括：

1. `source_roots`
2. `exclude_patterns`
3. `module_rules`
4. `language_adapters`
5. `policies`
6. `routing`
7. `knowledge`
8. `performance`

### 5.2 配置设计原则

- 人可读；
- 可版本管理；
- 默认安全；
- 不依赖隐式全局状态；
- 变更可触发重建边界的判断。

### 5.3 模块规则设计原则

模块不是纯目录概念。  
一个健康的模块规则 SHOULD 同时满足：

- 业务边界稳定；
- 依赖关系相对集中；
- 对 Agent 任务路由有价值；
- 与代码所有权或维护责任接近。

---

## 6. 数据面设计

### 6.1 统一语义模型

ANRSM v1 建议建立最小统一语义模型，包含以下字段簇：

1. **身份信息**：路径、语言、模块归属、工件类别；
2. **符号信息**：导出符号、主要内部符号、入口点；
3. **依赖信息**：imports、calls、配置读取、外部接线；
4. **契约信息**：输入、输出、Schema、事件、路由；
5. **状态信息**：状态对象、生命周期、状态变化点；
6. **副作用信息**：I/O、网络、存储、日志、缓存、消息；
7. **风险信息**：异常路径、幂等性、鉴权、并发、回滚；
8. **置信信息**：解析等级、摘要等级、未决事项。

### 6.2 统一语义模型的意义

如果没有统一语义模型，后续所有能力都会失稳：

- 指纹无法稳定；
- 漂移难以分级；
- 路由索引缺少统一基础；
- 多语言适配器不可比较；
- 知识层很难保持一致。

## 7. 语言适配策略

### 7.1 适配优先级

推荐的事实提取优先级如下：

1. AST / 编译器 API；
2. LSP 或语义服务；
3. 结构化约定文件；
4. 稳定格式的静态解析；
5. 保守文本启发式。

越往后，置信度越低。

### 7.2 对 Prompt 的要求

Prompt MAY 用于：

- 把结构化事实压缩成可读镜像正文；
- 生成知识层；
- 辅助判断复杂影响面。

Prompt MUST NOT 直接取代：

- 符号提取；
- 指纹计算；
- 基础依赖识别；
- Freshness 判定；
- 漂移分级。

### 7.3 低置信度策略

当语言适配器无法稳定解析时，系统 MUST：

1. 输出 `confidence_band: low`；
2. 降低自动化放行力度；
3. 提示需要展开源码或人工复核；
4. 保持可追溯，而不是静默生成高自信 prose。

---

## 8. 生成策略

### 8.1 先事实，后表达

Mirror Generator SHOULD 按以下顺序工作：

1. 接收统一语义事实；
2. 用模板固定段落骨架；
3. 对每段应用压缩规则；
4. 保留关键约束、边界和副作用；
5. 附加展开提示；
6. 写入文档头。

### 8.2 稳定性优先于文采

镜像是工程工件，不是写作作品。  
因此生成策略 SHOULD 优先：

- 术语稳定；
- 句式稳定；
- 章节稳定；
- 可 diff；
- 少同义替换；
- 少无意义润色。

### 8.3 机器区块与人工区块

推荐将镜像正文分为：

- **机器受管区块**：由生成器完全维护；
- **人工补充区块**：允许人工追加上下文，但必须明确标记边界。

机器区块 MUST 可覆盖重写；人工区块 SHOULD 保持有限且有格式约束。

## 9. 索引设计

### 9.1 路由索引

职责：

- 从任务描述中召回候选模块、文件和符号；
- 支持 Agent 先读镜像再定向展开源码。

最小字段建议：

- `artifact_id`
- `module_id`
- `keywords`
- `exported_symbols`
- `contracts`
- `risk_tags`
- `adjacent_artifacts`

### 9.2 符号索引

职责：

- 追踪符号定义与引用；
- 支撑影响面分析；
- 为路由和漂移提供更精确粒度。

### 9.3 依赖索引

职责：

- 描述 import、调用、配置和事件邻接；
- 支持变更邻域扩张。

## 10. 漂移分析的架构位置

Drift Analyzer 位于“新事实生成之后、写回之前”的关键位置。  
其作用不是单纯检查“文件是否同时改了文档”，而是回答：

1. 当前源码是否引发语义变化；
2. 哪些镜像因此失效；
3. 失效是低风险还是阻塞级；
4. 是否需要扩展受影响邻域；
5. 是否需要升级到全量或更深层检查。

这也是 ANRSM 与普通 docs sync 工具的本质区别。

## 11. 策略门禁设计

Policy Gate SHOULD 根据执行场景支持多层策略：

### 11.1 pre-commit

目标：快速本地反馈。  
特点：

- 只检查变更集与直接邻域；
- 优先低延迟；
- 可配置保守放行阈值。

### 11.2 pre-push / CI

目标：提高一致性与阻断效果。  
特点：

- 可扩展影响面；
- 可跑更完整的索引更新；
- 对 `blocking` 级漂移必须失败。

### 11.3 main branch / release

目标：保证主干可信。  
特点：

- 策略最严格；
- 对 `unknown` 和 `invalid` 状态更保守；
- 必要时触发更深层校验。

## 12. 可观测性与诊断

一个可用的 ANRSM 系统必须能回答以下问题：

1. 为什么这个文件被纳入或排除？
2. 为什么这个镜像是 `stale`？
3. 哪个字段导致它被判定为 Material Change？
4. 为什么这个任务路由到了这些文件？
5. 为什么这次触发了全量重建？
6. 哪个语言适配器产出了低置信度？

因此系统 SHOULD 输出：

- 详细漂移报告；
- 受影响图；
- 失败码；
- 适配器诊断；
- 覆盖率与新鲜度统计。

## 13. 推荐部署形态

v1 推荐三种部署形态，从低到高依次是：

### 13.1 仓库内 CLI 模式

最简单，适合 PoC 和早期采用。  
优点：

- 易集成；
- 易调试；
- 与 Git/CI 兼容好。

### 13.2 CI 服务模式

适合团队级推广。  
优点：

- 强制一致性更好；
- 可以集中化缓存；
- 可积累指标。

### 13.3 守护进程或平台模式

适合后续平台化。  
但不应成为 v1 的首要投入方向。

## 14. 架构决策建议

### 决策 D1：先用文件系统作为镜像存储

理由：

- 可直接版本管理；
- 可 diff；
- 可 code review；
- 无外部存储依赖；
- 便于渐进式引入。

### 决策 D2：先做增量，不做全量实时

理由：

- 大多数仓库改动是局部的；
- 全量扫描会迅速拖垮体验；
- 增量才符合提交前和 CI 的实际节奏。

### 决策 D3：先锁定规范，再做体验层

理由：

- 没有稳定规范，任何 UI 或平台都会建立在松散语义上；
- 一旦规范漂移，后续迁移成本极高。

---

## 15. 一句话结论

ANRSM v1 的正确架构不是“一个会写文档的 Agent”，而是 **一个由范围控制、结构化提取、语义压缩、漂移分析、门禁治理和 Agent 契约共同组成的仓库语义基础设施**。
