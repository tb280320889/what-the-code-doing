# 11. 反模式、失败模式与决策记录指南

状态：Normative / Advisory  
版本：v1  
优先级：P0

---

## 1. 目的

本文件定义 ANRSM 在设计和落地中最常见的反模式、失败模式以及推荐的架构决策记录方式。

反模式文档的价值在于：  
**提前定义哪些路绝对不该走，避免团队在实现过程中用局部捷径破坏系统长期可维护性。**

---

## 2. 一级反模式

### 反模式 A1：把 ANRSM 当作文档生成器

表现：

- 团队只关心能否“批量出 Markdown”；
- 不做指纹、不做漂移、不做门禁；
- Agent 继续默认直扫源码。

问题本质：

- 没有治理闭环；
- 没有运行时契约；
- 没有新鲜度。

纠偏要求：

- 立刻把重点转回“镜像是否可验证、可同步、可阻断漂移”。

---

### 反模式 A2：Prompt 直接生成镜像事实

表现：

- 没有结构化提取；
- 让 LLM 直接看源码输出镜像；
- 结果写得像文档，但事实基础不稳定。

风险：

- 幻觉；
- 不可复现；
- 无法做可靠差异分析；
- 难以做自动门禁。

纠偏要求：

- 先建立结构化事实层；
- Prompt 只负责压缩、解释和重组。

---

### 反模式 A3：让镜像成为第二真相源

表现：

- 工程师习惯先改镜像，再决定要不要改代码；
- 人类知识层长期独立维护；
- 线上事实与镜像长期背离。

风险：

- 决策污染；
- Review 失真；
- Agent 被错误上下文误导。

纠偏要求：

- 强化“源码唯一真相源”；
- 只允许镜像做派生，不允许做主导。

---

### 反模式 A4：只做路径联动，不做语义漂移

表现：

- `src/**` 改了就要求 `mirror/**` 也改；
- 但并不检查镜像是否真的更新了关键语义；
- 格式化改动也频繁触发失败。

风险：

- 团队会把门禁视为噪声；
- 真正的 Material Change 可能仍然漏检。

纠偏要求：

- 引入语义指纹与分类；
- 用漂移报告驱动门禁，而不是只看文件是否同改。

---

### 反模式 A5：默认全量重建

表现：

- 每次提交都全量重扫全仓；
- 增量能力长期不上线；
- 本地与 CI 都变得缓慢难用。

风险：

- 团队绕过系统；
- 成本高；
- 规模扩展失败。

纠偏要求：

- 以差异驱动为默认路径；
- 全量重建只作为恢复或升级手段。

---

### 反模式 A6：镜像正文写成“漂亮空话”

表现：

- 段落很多，但没有责任边界、状态、副作用和风险；
- 全是“这个文件负责处理某某逻辑”之类低信息密度句子。

风险：

- Agent 和人类都无法真正决策；
- Token 节约了，但信息价值也消失了。

纠偏要求：

- 正文必须写关键契约、控制流、副作用、不变量和展开条件。

---

### 反模式 A7：忽略“何时必须展开源码”

表现：

- 团队假设镜像足够强大，可以替代具体实现阅读；
- Agent 在复杂变更上只看镜像不看源码。

风险：

- 精确修改失败；
- 影响面漏判；
- 高风险变更引入回归。

纠偏要求：

- 每个镜像明确给出展开条件；
- 高风险变更默认要求源码展开。

---

### 反模式 A8：没有低置信度机制

表现：

- 解析不稳定时仍输出高自信镜像；
- 没有 `confidence_band`；
- 门禁无法区分高信与低信工件。

风险：

- 系统表面正常，实际在关键时刻误导最严重。

纠偏要求：

- 显式输出置信度；
- 在策略层对低置信度做保守处理。

---

## 3. 二级反模式

### 反模式 B1：术语不固定

表现：

- 团队一会儿说 mirror，一会儿说 doc，一会儿说 note；
- knowledge 和 mirror 混用；
- drift 和 change 混用。

结果：

- 规范不可执行；
- 工具接口混乱；
- Agent prompt 难以稳定。

纠偏：

- 术语表必须先冻结；
- 代码、文档、CLI、报告统一命名。

### 反模式 B2：模块边界靠感觉定义

表现：

- 模块命名不稳定；
- 同一文件今天属于 A，明天属于 B；
- 模块与业务或维护边界脱节。

结果：

- 模块镜像没有长期价值；
- 路由索引精度下降；
- 知识层变得混乱。

纠偏：

- 模块边界要与业务责任、依赖关系和维护责任一致。

### 反模式 B3：把所有文件都镜像到同一强度

表现：

- 低价值样板代码和高风险核心文件使用相同镜像粒度；
- 系统在低价值区域浪费太多成本。

结果：

- 整体 ROI 下降；
- 维护成本过高。

纠偏：

- 建立工件分层和优先级；
- 高价值、高风险、高频变更区域优先。

### 反模式 B4：人类知识层提前平台化

表现：

- 在文件镜像和门禁没稳定前，先做漂亮 UI 和知识门户。

结果：

- 表层体验很好，但底层事实不稳定；
- 平台放大错误。

纠偏：

- 先把底层镜像和漂移治理打稳，再做知识层体验。

## 4. 常见失败模式

### F1：覆盖率高，但保真度低

症状：

- 大多数文件都有镜像；
- 但镜像看不出关键副作用和边界；
- Agent 仍然频繁误改。

根因：

- 过度追求覆盖；
- 摘要模板空洞；
- 结构化事实层薄弱。

修复：

- 提升高价值文件的语义抽取深度；
- 引入人工抽样审计。

### F2：门禁严格，但误报过高

症状：

- 团队频繁被阻塞；
- 大量失败来自 C0；
- 开发者开始绕过或关闭门禁。

根因：

- 语义指纹太敏感；
- 缺少 C0/C1 分类边界；
- 缺少保守豁免策略。

修复：

- 重新设计指纹排噪；
- 调整策略分级；
- 加入失败样本回归。

### F3：Agent 已接入，但行为未改变

症状：

- 系统里有镜像；
- 但 Agent 仍先扫源码，再偶尔看镜像。

根因：

- 路由索引无效；
- 镜像质量不够高；
- 运行时契约没有被硬编码到 Agent 工作流。

修复：

- 把读取顺序写进 Agent runtime；
- 把镜像读取变成默认步骤；
- 评估实际上下文消费路径。

### F4：知识层逐渐失真

症状：

- 架构文档越来越多；
- 但与真实代码越走越远。

根因：

- 知识层脱离镜像演化；
- 没有来源追踪；
- 没有更新策略。

修复：

- 强制知识层保留 `source_artifacts`；
- 定期从镜像重编译；
- 对关键知识文档引入 freshness。

### F5：系统只能在一个仓库一次性工作

症状：

- 第一次初始化可用；
- 后续升级、迁移、适配器变化就失控。

根因：

- 缺少版本化；
- 缺少重建策略；
- 缺少 schema 管理。

修复：

- 版本化协议、指纹和适配器；
- 建立升级和重建流程。

## 5. 设计评审时必须追问的问题

每次设计评审至少应追问：

1. 这项设计是否强化了“源码唯一真相源”？
2. 它是否提升了而不是削弱了漂移可观测性？
3. 它会不会让 Agent 又回到默认全仓读源码？
4. 它引入的新复杂度是否可审计、可重建？
5. 它是否把短期方便变成长期双真相源？

---

## 6. ADR（架构决策记录）建议

ANRSM 项目 SHOULD 使用 ADR 记录关键决策。  
原因：

- 术语和策略一旦漂移，系统会迅速失去一致性；
- 里程碑推进中会反复碰到“为什么当初这样设计”。

### 6.1 适合写 ADR 的主题

- 路径映射规则；
- 主语言优先级；
- 模块边界定义；
- 语义指纹算法；
- 低置信度策略；
- 全量重建触发条件；
- 是否纳入测试/配置/生成产物；
- 知识层生成策略。

### 6.2 ADR 模板建议

```markdown
# ADR-00X: <决策标题>

## 背景
问题是什么，为什么必须现在决定。

## 决策
最终采用什么方案。

## 备选方案
还有哪些方案被考虑过。

## 取舍
为什么放弃其他方案。

## 影响
对架构、流程、性能、团队协作有什么影响。

## 风险
该决策最可能导致的失败模式是什么。

## 复核条件
在什么情况下应重新打开此决策。
```

## 7. 审查反模式的操作方式

建议在每个里程碑结束时，用反模式文档做一次逆向审查：

1. 本阶段是否落入了任何一级反模式？
2. 是否出现二级反模式萌芽？
3. 是否有新的失败样本未进入回归集？
4. 是否需要新增 ADR 解释边界变化？

这样做的价值是：  
它能防止团队在工程推进中“逐渐偏离而不自知”。

---

## 8. 最危险的三个信号

如果项目出现以下三个信号，说明 ANRSM 已经开始偏离正确方向：

1. 团队开始直接手改镜像并以此代替源码事实；
2. Agent 依然默认大面积读源码，镜像只做装饰；
3. 门禁经常被关闭或绕过。

一旦出现这三个信号中的任意两个，项目必须停下来做一次规范回正。

---

## 9. 一句话结论

ANRSM 最常见的失败，不是技术做不到，而是团队把它悄悄做成了：  
**一套没有语义闭环、没有门禁、没有运行时契约的“高级文档生成器”。**
