# 07. 漂移检测、同步策略与 CI 门禁

状态：Normative  
版本：v1  
优先级：P0

---

## 1. 目的

本文件定义 ANRSM 的同步闭环：  
如何检测源码变化、如何判断镜像是否失效、如何基于漂移结果决定是否阻塞提交或合并。

ANRSM 是否真正成立，关键不在于“能生成镜像”，而在于：

1. 源码变了能否被感知；
2. 漂移能否被分类；
3. 同步能否自动或半自动发生；
4. 未同步的实质漂移能否被门禁阻断。

---

## 2. 为什么普通“源码改了文档也要改”规则不够

传统的 docs sync 规则通常是：

- 某个目录改了；
- 要求另一个目录也有改动；
- 否则失败。

这类规则的问题是：

1. 它只检查“有没有文件一起改”，不检查“改得对不对”；
2. 它不知道格式化和注释改动是不是值得更新镜像；
3. 它无法判断影响邻域；
4. 它不能输出工程上可用的漂移解释。

因此，ANRSM 的门禁必须建立在 **语义漂移分析** 上，而不是简单的路径联动。

---

## 3. 漂移检测的输入

Drift Analyzer 在 v1 中 SHOULD 使用以下输入：

1. Git 变更集；
2. 变更前后的源码指纹；
3. 变更前后的语义指纹；
4. 新旧索引；
5. 模块规则；
6. 策略配置；
7. 解析置信度。

---

## 4. 漂移检测的输出

漂移检测的输出 MUST 至少包含：

1. 变更文件列表；
2. 受影响镜像工件列表；
3. 每个工件的 `freshness_state` 建议；
4. 漂移级别 `drift_level`；
5. 实质变更分类；
6. 证据摘要；
7. 门禁建议。

---

## 5. 变更分类体系

为了降低误报和漏报，v1 推荐使用分级变更分类。

### 5.1 C0：Non-material Change

定义：不会改变镜像有效性的变化。  
示例：

- 空白、注释、格式化；
- import 顺序调整；
- 无影响的代码移动；
- 无外部可见影响的局部命名清理。

默认处理：

- 可以更新 `source_fingerprint`；
- `semantic_fingerprint` SHOULD 保持稳定；
- 不要求阻塞级镜像更新。

### 5.2 C1：Local Material Change

定义：本文件语义发生实质变化，但外部契约未明显改变。  
示例：

- 关键控制流变化；
- 本地状态管理变化；
- 副作用时机变化；
- 错误处理逻辑变化；
- 缓存策略变化。

默认处理：

- 文件镜像 MUST 更新；
- 模块镜像 MAY 更新；
- 若风险域受影响，SHOULD 升级检查邻域。

### 5.3 C2：Contract Material Change

定义：对外契约或可见边界变化。  
示例：

- 导出函数签名变化；
- API 路由或 Schema 变化；
- 事件名变化；
- 配置键变化；
- 权限要求变化。

默认处理：

- 文件镜像 MUST 更新；
- 模块镜像 SHOULD 更新；
- 依赖邻域 SHOULD 被扩展；
- 若未同步，CI SHOULD 失败。

### 5.4 C3：Systemic Material Change

定义：跨模块、跨边界或高风险系统行为变化。  
示例：

- 持久化模型变化；
- 事务边界变化；
- 鉴权链变化；
- 并发模型变化；
- 外部依赖接线变化；
- 发布/运维关键脚本变化。

默认处理：

- 相关文件镜像 MUST 更新；
- 模块镜像 MUST 更新；
- 知识层 MAY 被标记待更新；
- 主干策略下 SHOULD 视为阻塞级。

---

## 6. 漂移级别定义

### none

无漂移。  
条件：新旧语义状态一致，相关镜像仍然有效。

### low

有差异，但不构成阻塞。  
常见于：

- 局部低风险变化；
- 置信度下降但未达阻塞阈值；
- 辅助索引需要刷新。

### material

存在必须修复的镜像漂移。  
通常意味着文件镜像已不再可信。

### blocking

存在会影响主干可信度或 Agent 决策质量的漂移。  
在对应策略下，提交或合并 MUST 被阻止。

---

## 7. Git Diff 的使用原则

### 7.1 Git Diff 是候选集，不是真相

Git Diff SHOULD 用于缩小重新分析范围，但 MUST NOT 直接替代语义比较。  
原因：

- 同样的路径变更可能只是格式化；
- 某些语义变化会波及邻域，而邻域文件本身未改；
- 重命名、移动和生成文件会混淆简单路径匹配。

### 7.2 变更邻域扩展

当发生以下情况时，系统 SHOULD 扩展分析邻域：

1. 导出符号变化；
2. 契约变化；
3. 关键依赖边变化；
4. 高风险标签变化；
5. 模块入口文件变化；
6. 解析器置信度下降。

邻域扩展策略 MAY 采用：

- 直接 import 邻域；
- 反向引用邻域；
- 模块内高影响文件集合；
- 契约关联文件集合。

---

## 8. 同步策略

### 8.1 自动同步

在适配器和生成器足够成熟时，系统 SHOULD 自动更新受影响镜像。  
自动同步适用于：

- 文件镜像正文重写；
- 指纹与头字段刷新；
- 索引更新；
- 漂移报告生成。

### 8.2 半自动同步

当解析能力有限或知识层需要补充上下文时，系统 MAY 进入半自动模式：

1. 自动标记受影响工件；
2. 自动生成候选镜像；
3. 要求人或 Agent Reviewer 复核低置信度部分。

### 8.3 禁止的同步方式

系统 MUST NOT 仅通过“把 old mirror 的时间戳改新”来伪装同步完成。  
同步的判定必须建立在：

- 新的结构化事实；
- 新的语义指纹；
- 新的验证结果。

---

## 9. freshness_state 的判定规则

### 9.1 fresh

要求同时满足：

1. 当前源码可成功解析或以保守方式解释；
2. 当前语义指纹与镜像声明一致；
3. 当前正文与结构化事实一致；
4. 门禁未发现阻塞漂移。

### 9.2 stale

以下任一情况成立即可判定为 `stale`：

- 源码已改动但镜像未更新；
- 语义指纹变化但镜像头未刷新；
- 受影响邻域扩展后发现模块镜像已过时；
- 系统知道它应被更新但尚未完成更新。

### 9.3 invalid

以下情况应判定为 `invalid`：

- 文档头不可解析；
- 必填字段缺失；
- 字段值非法；
- `source_path` 不存在；
- 生成器失败写入；
- 正文结构损坏到无法被消费；
- 指纹自相矛盾。

### 9.4 unknown

以下情况可判定为 `unknown`：

- 解析器无法确定语义；
- 配置缺失；
- 增量状态损坏，但还未完整重建；
- 某些证据不足以得出结论。

---

## 10. 门禁层级设计

### 10.1 本地 pre-commit

目标：快速反馈。  
建议策略：

- 仅检查变更文件及直接邻域；
- 对 `blocking` 失败；
- 对 `material` 失败或警告，视项目节奏决定；
- 对 `unknown` 给出显式提示。

### 10.2 PR CI

目标：保证合并前可信。  
建议策略：

- 对 `material` 和 `blocking` 一律失败；
- 对 `unknown` 在高风险目录下失败；
- 允许低风险目录保守放行，但必须留下注释说明。

### 10.3 主干合并/发布

目标：保护主干与发布质量。  
建议策略：

- `blocking` MUST 失败；
- `material` MUST 失败；
- `invalid` MUST 失败；
- `unknown` SHOULD 失败，除非配置中存在显式例外。

---

## 11. 失败码建议

为便于自动化处理，建议输出标准失败码。

- `ANRSM-001`：受管文件缺少镜像
- `ANRSM-002`：镜像头不可解析
- `ANRSM-003`：必填字段缺失
- `ANRSM-004`：语义指纹不一致
- `ANRSM-005`：检测到 Material Change 但镜像未更新
- `ANRSM-006`：模块镜像失效
- `ANRSM-007`：索引失效或缺失
- `ANRSM-008`：解析器低置信度且策略要求阻塞
- `ANRSM-009`：配置错误导致无法判定新鲜度
- `ANRSM-010`：需要全量重建但未执行

失败码 SHOULD 伴随：

- 问题对象；
- 证据摘要；
- 修复建议。

---

## 12. 典型门禁规则矩阵

| 变更类型 | 文件镜像 | 模块镜像 | 索引 | 建议门禁 |
| --- | --- | --- | --- | --- |
| C0 Non-material | 可不更新正文 | 通常不更新 | 可局部刷新 | 默认放行 |
| C1 Local Material | 必须更新 | 视影响更新 | 应刷新 | 可阻塞 |
| C2 Contract Material | 必须更新 | 应更新 | 必须刷新 | 应阻塞 |
| C3 Systemic Material | 必须更新 | 必须更新 | 必须刷新 | 必须阻塞 |

注：表中“可不更新正文”不等于“完全不处理”。  
即使 C0，也 MAY 更新 `source_commit`、缓存或统计信息，但不应制造伪漂移。

---

## 13. 低置信度与保守策略

### 13.1 低置信度的本质

低置信度不是失败，但它意味着系统无法为某些判断承担完全信任。

### 13.2 推荐策略

当满足以下任一情况时，系统 SHOULD 采用保守策略：

- 高风险目录；
- 契约相关文件；
- 鉴权、事务、并发、迁移、部署脚本；
- 低置信度且发生 Material Change；
- 适配器升级后首次运行。

保守策略包括：

- 扩大邻域分析；
- 要求展开源码；
- 提高门禁级别；
- 要求人工复核。

---

## 14. 全量重建触发条件

下列情况 SHOULD 触发全量重建，而不是仅靠增量：

1. 语言适配器升级；
2. 语义指纹算法升级；
3. 模块规则变更；
4. 索引结构变更；
5. 基线损坏；
6. 大规模路径重命名；
7. Git 历史缺失导致差异不可用。

全量重建是恢复一致性的手段，不应成为默认路径。

---

## 15. 诊断信息要求

任何门禁失败都 SHOULD 输出足够诊断信息，至少包括：

1. 哪个源码文件变化触发了问题；
2. 哪个镜像工件失效；
3. 变化被归类为哪一类；
4. 证据是什么；
5. 需要执行什么修复操作。

坏诊断信息会导致团队把门禁视为噪声，从而绕过系统。

---

## 16. 对 Review 的建议

Code Review SHOULD 检查以下问题：

1. 这次代码改动是否真的改变了镜像所描述的语义；
2. 镜像更新是否只改了必要段落；
3. 是否误把 C0 识别成 C1/C2/C3；
4. 是否遗漏了受影响邻域；
5. 低置信度提示是否被合理对待。

---

## 17. 与 Agent 的关系

门禁系统不是和 Agent 对立，而是 Agent 工作流的一部分。  
一个合规的 Agent 应该把以下步骤视为默认义务：

1. 修改源码前读取镜像；
2. 修改后触发镜像更新；
3. 读取漂移报告；
4. 在失败时补足遗漏，而不是绕过门禁。

---

## 18. 一句话结论

ANRSM 的同步与门禁机制不是“检查有没有一起改 Markdown”，而是 **用源码语义变化去驱动镜像失效、更新、验证和阻断，从而把镜像真正纳入仓库治理闭环**。
