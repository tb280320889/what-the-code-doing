# 03. 系统边界、假设与非目标

状态：Normative  
版本：v1  
优先级：P0

---

## 1. 目的

本文件用来回答一个经常被忽略但会直接决定项目成败的问题：  
**ANRSM 到底覆盖什么，不覆盖什么；它依赖哪些前提；哪些目标不应被 v1 误背。**

边界不清会导致两类失败：

1. 目标膨胀，v1 过载；
2. 责任模糊，系统无法形成硬约束。

---

## 2. 范围边界

### 2.1 v1 的纳入范围

ANRSM v1 SHOULD 优先覆盖以下工件：

1. **一方业务源码**  
   前端、后端、移动端、共享库等核心业务代码。

2. **关键契约工件**  
   API 定义、Schema、事件定义、类型边界、协议声明。

3. **关键运行时配置**  
   影响行为、接线、部署或安全边界的配置。

4. **数据库迁移和持久化边界**  
   会改变数据模型或行为边界的迁移脚本与 ORM Schema。

5. **关键测试**  
   对外契约测试、集成测试、端到端测试或作为行为规范存在的测试。

6. **关键构建/发布/运维脚本**  
   当它们会影响发布、安全、环境差异或故障恢复时。

### 2.2 v1 的默认排除范围

以下工件 SHOULD 默认排除：

1. 第三方依赖目录；
2. 锁文件；
3. 编译输出目录；
4. 纯静态二进制资源；
5. 自动生成且不应直接编辑的中间产物；
6. 临时日志、缓存和本地工具状态文件。

### 2.3 需要特别策略判断的灰区工件

以下工件不应简单一刀切，需要由 Scope Manager 配置：

- 代码生成产物；
- 样板代码；
- 大型 JSON 配置；
- 文档内嵌脚本；
- 测试快照；
- 基础设施定义文件；
- 查询脚本和分析脚本。

原则是：  
**只有在该工件会显著影响开发任务路由、系统行为理解或风险识别时，才值得进入镜像范围。**

---

## 3. 真相源边界

### 3.1 真相源的组成

ANRSM 中的真相源包括：

- 可执行源码；
- 影响行为的受管配置；
- 决定对外契约的定义文件；
- 决定数据结构的迁移文件；
- 明确承载行为规范的测试；
- 关键发布与运维脚本。

### 3.2 不是真相源的内容

以下内容 MUST NOT 被当作真相源：

- 文件镜像；
- 模块镜像；
- 知识文档；
- PR 描述；
- Issue 讨论；
- Commit Message；
- 会议纪要；
- Wiki 页面；
- 口头知识。

这些内容可以辅助理解，但一旦与真相源冲突，必须以真相源为准。

---

## 4. 角色边界

### 4.1 ANRSM 负责什么

ANRSM v1 负责：

1. 为纳入范围的工件生成机器优先语义镜像；
2. 建立镜像与源码之间的确定性映射；
3. 检测语义漂移；
4. 通过门禁阻止未同步漂移流入主干；
5. 为 Agent 提供任务路由与最小充分上下文入口；
6. 为后续知识层生成提供统一语义基座。

### 4.2 ANRSM 不负责什么

ANRSM v1 不负责：

1. 替代编译、测试、Lint、静态分析；
2. 替代代码审查；
3. 保证所有自然语言总结都绝对完美；
4. 自动修复所有逻辑错误；
5. 对运行时动态行为做完整精确模拟；
6. 代替现有知识管理平台的全部功能；
7. 代替全文搜索；
8. 在无源码展开的前提下完成所有复杂修改。

---

## 5. 系统假设

### 5.1 仓库假设

v1 默认假设：

1. 仓库在 Git 管理之下；
2. 主要源码文件具备相对稳定的路径与扩展名；
3. 主要语言至少存在一种可用的解析策略；
4. 团队接受将镜像纳入版本管理；
5. 团队接受在提交或合并阶段增加同步门禁。

### 5.2 团队工作流假设

v1 假设团队具备以下最小工作流成熟度：

1. 提交与 PR 有明确的代码所有权或责任边界；
2. 团队能接受“源码改动触发镜像改动”的新规范；
3. Agent 的使用不是完全不可控的个人行为，而是可被规范化的协作方式；
4. 有能力维护最基本的 CI 流程。

### 5.3 技术假设

1. 至少一种主语言可做 AST 或等价结构化提取；
2. 对无法完全解析的语言，允许回退到保守策略，但 MUST 显式降低置信度；
3. 仓库中不存在大量无法识别边界的混沌脚本堆积，否则 v1 覆盖率会显著受限。

---

## 6. 非目标

为了防止目标膨胀，以下内容明确列为 v1 非目标。

### 6.1 非目标 N1：替代源码阅读

ANRSM 的目标不是让任何任务都只读镜像。  
它的目标是让大多数任务先通过镜像完成定位、规划和初步理解，再在必要处展开源码。

### 6.2 非目标 N2：一开始就支持任意语言

v1 SHOULD 先支持一到两种主语言，优先打通闭环。  
“支持所有语言”是平台化阶段目标，不应成为 v1 阻塞项。

### 6.3 非目标 N3：自动构建完美知识库

知识层是重要目标，但不是 v1 的最小阻塞项。  
v1 首先要保证文件镜像、漂移检测和门禁。

### 6.4 非目标 N4：让镜像脱离代码独立演进

如果镜像允许独立演进，它就会迅速成为第二真相源。  
这违背 ANRSM 的基本设计。

### 6.5 非目标 N5：用 LLM 直接发明事实

LLM 可以压缩、重写、解释，但不能在没有结构化约束和来源锚点的前提下“自由生成”核心语义事实。

---

## 7. 质量边界

### 7.1 可接受的不完美

以下情况在 v1 是可接受的，只要它们被显式标识：

- 某些语言仅有保守级提取；
- 某些复杂文件的摘要置信度较低；
- 某些低价值工件暂未纳入；
- 模块边界需要经历一轮重构后才能更清晰。

### 7.2 不可接受的状态

以下状态即使在 v1 也不可接受：

1. 镜像无来源、无指纹、无新鲜度；
2. 源码改了，但系统没有任何漂移信号；
3. Agent 可以长期跳过镜像检查；
4. 文档存在但门禁无法阻断过期镜像；
5. 生成器升级后全仓镜像不可重建；
6. 团队无法判断镜像是否可信。

---

## 8. 代码生成与镜像的边界

### 8.1 代码生成的基本原则

如果仓库存在代码生成链路，ANRSM SHOULD 优先镜像以下对象之一：

1. 生成器源头；
2. 模板；
3. Schema；
4. 生成规则。

而不是盲目镜像所有生成产物。

### 8.2 为什么不应默认镜像生成产物

原因有三：

1. 生成产物往往数量大、变化频繁、价值低；
2. 它们的真实语义通常受上游 Schema 或模板决定；
3. 直接镜像生成产物会制造无意义 Token 负担和噪声漂移。

### 8.3 例外条件

当生成产物本身被直接调试、直接引用或直接承载外部契约时，MAY 将其纳入范围，但应在配置中显式声明。

---

## 9. 测试的边界

### 9.1 为什么测试可能值得纳入

对于很多团队，测试不仅是验证工具，还是行为规格。  
因此：

- 契约测试；
- 集成测试；
- 端到端测试；
- 关键业务规则测试

都可能值得被镜像。

### 9.2 v1 的默认建议

v1 SHOULD 优先镜像以下测试：

1. 对外 API/事件/协议契约测试；
2. 关键业务规则测试；
3. 会显著影响任务路由的测试入口。

单元测试全量镜像通常不是 v1 必需能力。

---

## 10. 人类知识层的边界

### 10.1 人类知识层是什么

人类知识层是服务于理解、交接、排障、维护和培训的派生层。  
它不是仓库真实行为的唯一描述来源。

### 10.2 人类知识层不应做什么

人类知识层 MUST NOT：

- 自行定义新的行为真相；
- 绕过镜像直接描述与代码不一致的事实；
- 变成一个由个人长期手工维护但无人校验的孤立 Wiki。

### 10.3 v1 中的人类知识层定位

在 v1 中，人类知识层 SHOULD 作为模块镜像和结构化事实的二次编译物存在。  
若资源有限，知识层 MAY 推迟到里程碑后段。

---

## 11. Agent 边界

### 11.1 Agent 必须做什么

Agent 在合规模式下 MUST：

1. 先读镜像再决定是否展开源码；
2. 在修改源码后触发镜像更新；
3. 在镜像失效时显式说明失效原因或触发修复；
4. 不得把过期镜像当作高可信输入。

### 11.2 Agent 不应做什么

Agent SHOULD NOT：

1. 在没有镜像或镜像已知失效时直接大范围修改源码；
2. 只凭自然语言镜像生成复杂精确补丁；
3. 在未读受影响邻域的情况下跨模块盲改；
4. 修改源码后不更新镜像。

---

## 12. 仓库规模边界

v1 最适合以下量级：

- 单仓或单主仓；
- 主要语言数量有限；
- 代码规模从中小型到中大型；
- 变更频率高，但不是超大单体平台级别。

对于超大多仓、多语言、强动态元编程系统，v1 SHOULD 先从单域试点，而不是一次性全局铺开。

---

## 13. 时间边界

v1 的目标不是“一次初始化后永远正确”，而是：

1. 建立基线；
2. 能基于变更持续维护；
3. 在策略允许范围内快速感知失效；
4. 让过期状态可见、可阻断、可修复。

因此，“实时”在 v1 中应解释为：

- 提交前、本地验证前、PR 检查前或合并前的可接受同步粒度；
- 不要求毫秒级后台常驻分析。

---

## 14. 组织边界

ANRSM 不只是技术实现，还会改变协作契约。  
因此组织层面至少需要接受以下事实：

1. “代码变更不更新镜像”不再被视为正常行为；
2. 评审不仅检查代码，也检查镜像是否反映真实语义变化；
3. 低质量镜像会直接影响 Agent 和人类的后续效率；
4. 团队需要指定镜像策略和适配器的维护责任人。

如果组织不接受这些治理约束，ANRSM 很难长期有效。

---

## 15. 边界判断的决策准则

当团队不知道某类工件是否该纳入 ANRSM 时，可按以下顺序判断：

1. 它是否影响真实行为或行为边界？
2. 它是否能显著帮助 Agent 路由和规划？
3. 它是否会频繁变动且难以维护？
4. 是否存在更上游、更稳定的真相源可替代？
5. 将其纳入后，增量维护成本是否可接受？

只有在 1 和 2 至少满足其一，且 4 不成立时，纳入才有较高价值。

---

## 16. 一句话结论

ANRSM v1 的边界是：  
**用最少但关键的受管范围，建立源码到语义镜像的强同步闭环；它不是全能知识平台，也不是替代源码的魔法层。**
