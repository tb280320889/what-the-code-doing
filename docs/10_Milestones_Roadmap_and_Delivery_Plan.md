# 10. 里程碑、路线图与交付计划

状态：Planning  
版本：v1  
优先级：P0

---

## 1. 目的

本文件把 ANRSM 从概念和规范推进到可实施的阶段性工程计划。  
目标不是给出唯一排期，而是定义一个对中小型到中大型仓库都适用的 **最小风险递进路径**。

核心原则：

1. 先做闭环，再做平台；
2. 先做主语言，再做多语言；
3. 先做文件镜像，再做知识层；
4. 先确保门禁有效，再扩大依赖范围。

---

## 2. 里程碑总览

推荐采用六阶段路线：

- `M0` 概念验证与规范冻结
- `M1` 文件级镜像基线
- `M2` 增量同步与漂移门禁
- `M3` 路由索引与 Agent 集成
- `M4` 模块镜像与知识层编译
- `M5` 规模化、可观测性与多语言扩展

---

## 3. M0：概念验证与规范冻结

### 目标

把“我们要做什么”从想法变成稳定规范与试点边界。

### 必做事项

1. 冻结术语体系；
2. 冻结 v1 边界；
3. 选定试点仓库；
4. 选定第一主语言；
5. 选定镜像目录结构；
6. 选定文档头 schema；
7. 选定 Material Change 的初始分类。

### 交付物

- 设计母稿；
- 术语与规范语言；
- 路径映射规则；
- 初版配置文件；
- 试点仓库清单；
- 风险登记表。

### 退出条件

- 主要争议项有明确决策；
- 至少一个试点仓库和一条主链路可实现；
- 团队接受“源码改动必须触发镜像同步”。

### 风险

- 目标膨胀；
- 试点仓库选得过大或过乱；
- 未区分 v1 和长期愿景。

---

## 4. M1：文件级镜像基线

### 目标

建立从源码到文件镜像的第一条可运行闭环。

### 必做事项

1. 实现 Scope Manager；
2. 实现主语言适配器；
3. 实现统一语义模型最小子集；
4. 实现文件级镜像生成；
5. 实现头字段写入；
6. 生成基线指纹；
7. 生成基础索引雏形。

### 交付物

- `mirror/file/**`
- `mirror/index/symbol_index.json` 初版
- `mirror/index/dependency_index.json` 初版
- 基线生成 CLI
- 首轮覆盖率报告

### 退出条件

- 主目录文件镜像覆盖率达到目标；
- 高风险文件可生成有效镜像；
- 头字段完整；
- 生成过程可重复。

### 风险

- 主语言解析能力不足；
- 生成内容空洞；
- 路径映射不稳定。

---

## 5. M2：增量同步与漂移门禁

### 目标

让系统从“会生成”升级为“会维护、会阻断漂移”。

### 必做事项

1. 接入 Git diff；
2. 实现语义指纹；
3. 实现 C0/C1/C2/C3 分类；
4. 实现漂移报告；
5. 接入 pre-commit 或 CI 门禁；
6. 实现失败码和诊断输出。

### 交付物

- 增量更新 CLI
- `mirror/reports/drift_report.json`
- CI 工作流或 pre-commit hook
- 失败码说明文档
- 种子变更测试集

### 退出条件

- 常见 Material Change 可被识别；
- 未同步的 C2/C3 能阻塞；
- C0 误阻塞率可接受；
- 增量链路性能可用。

### 风险

- 误报过高导致团队绕过门禁；
- 漏报导致镜像失去信任；
- 增量邻域扩展不稳定。

---

## 6. M3：路由索引与 Agent 集成

### 目标

让 Agent 真正以镜像为默认入口，而不是让镜像只停留在文件层。

### 必做事项

1. 完善 `routing_index`；
2. 定义 Agent 读取顺序；
3. 定义 Expansion Read 触发条件；
4. 实现任务到模块/文件的基础路由；
5. 将镜像更新纳入 Agent 工作流；
6. 建立路由基准集。

### 交付物

- `mirror/index/routing_index.json`
- Agent 集成指南
- 路由基准测试结果
- Agent 运行时日志规范

### 退出条件

- 多数试点任务可先经镜像完成定位；
- Agent 默认不再全仓读源码；
- 修改后会自动更新镜像并通过门禁；
- Reviewer 能看到镜像同步结果。

### 风险

- 路由过宽；
- Agent 仍旧惯性跳过镜像；
- 镜像质量不足以支持定位。

---

## 7. M4：模块镜像与知识层编译

### 目标

把文件级能力提升为模块级理解和人类可用知识层。

### 必做事项

1. 定义模块规则；
2. 生成模块镜像；
3. 设计知识层模板；
4. 从镜像编译架构总览和功能边界文档；
5. 建立人工补充机制；
6. 为交接与排障生成最小知识集。

### 交付物

- `mirror/module/**`
- `knowledge/architecture/**`
- `knowledge/features/**`
- `knowledge/operations/**`
- 模块级验收报告

### 退出条件

- 模块级责任与边界表达清晰；
- 人类接手可以优先读知识层而非到处翻源码；
- 知识层来源可追溯；
- 未形成第二真相源。

### 风险

- 知识层开始脱离镜像独立维护；
- 模块边界定义混乱；
- 模块镜像只是文件摘要拼接。

---

## 8. M5：规模化、可观测性与多语言扩展

### 目标

把试点能力推进为更可运维、更可治理、更可扩展的平台级能力。

### 必做事项

1. 增加多语言适配器；
2. 引入缓存与性能优化；
3. 增强可观测性；
4. 引入失败样本库；
5. 完善策略分级；
6. 建立适配器升级和全量重建流程。

### 交付物

- 多语言支持矩阵
- 指标看板或报告
- 失败样本库
- 全量重建运维文档
- 版本迁移策略

### 退出条件

- 至少两种主要语言稳定支持；
- 全量与增量链路可运维；
- 误报/漏报持续受控；
- 团队对系统有持续信任。

### 风险

- 过早平台化；
- 复杂度上升快于价值释放；
- 适配器维护成本失控。

---

## 9. 各里程碑的依赖关系

```text
M0 -> M1 -> M2 -> M3 -> M4 -> M5
```

强依赖说明：

- 没有 M1，就不存在可验证镜像；
- 没有 M2，镜像无法进入治理闭环；
- 没有 M3，Agent 不会真正改变工作方式；
- 没有 M4，人类知识层无法共享同一语义基座；
- M5 只能在前四者稳定后才值得投入。

---

## 10. 推荐团队角色配置

### 架构负责人

- 冻结术语、边界和质量目标；
- 裁决模块规则和关键策略争议。

### 平台/基础设施工程师

- 实现适配器、生成器、门禁和索引；
- 维护性能与可观测性。

### 仓库领域工程师

- 提供模块边界知识；
- 审核镜像保真度；
- 提供高风险变更样本。

### Agent 工作流工程师

- 集成 ANRSM 读取顺序；
- 设计回退与写回逻辑；
- 建立任务路由基准。

---

## 11. 推荐实施顺序

如果资源有限，建议按以下具体顺序实施：

1. 选试点仓库与主语言；
2. 落地术语、路径映射和头 schema；
3. 做文件镜像生成；
4. 做语义指纹与漂移报告；
5. 接 CI 门禁；
6. 接 Agent 读取顺序；
7. 做模块镜像；
8. 做知识层；
9. 扩多语言；
10. 做平台化与指标系统。

这样做的原因是：  
它优先保证“价值闭环”，而不是“架构看上去很完整”。

---

## 12. 最小 PoC 建议

如果团队要用最短路径验证概念，PoC SHOULD 只包含：

1. 一个真实仓库；
2. 一个主语言；
3. 文件镜像生成；
4. 语义指纹；
5. 漂移报告；
6. 一个 CI 阻塞规则；
7. 一个 Agent 定位任务链路。

PoC 不应一开始就做：

- UI；
- 多语言；
- 知识平台；
- 分布式服务；
- 复杂权限系统。

---

## 13. 交付节奏建议

每个里程碑都应产生以下三类输出：

1. **规范层输出**  
   文档、ADR、schema、策略说明。

2. **系统层输出**  
   CLI、适配器、生成器、索引器、门禁链路。

3. **验证层输出**  
   指标、测试结果、失败样本、抽样评审记录。

这样可以避免“代码写了很多，但没有形成可治理能力”。

---

## 14. 里程碑退出评审问题

每个里程碑结束时，应回答以下问题：

1. 该阶段新增能力是否形成闭环？
2. 是否新增了新的不透明复杂度？
3. 是否开始偏离“源码唯一真相源”原则？
4. Agent 的默认行为是否真的被改变？
5. 下一阶段是否建立在稳定基础上？

只要其中有两项回答不清楚，就不应轻易进入下一阶段。

---

## 15. 一句话结论

ANRSM 的正确落地顺序，不是一次性造一个大而全平台，而是：  
**先把文件镜像、漂移门禁和 Agent 读取顺序这三个核心闭环打通，再逐步扩展到模块理解和人类知识层。**
