# 09. 验证、验收、测试与指标

状态：Normative  
版本：v1  
优先级：P0

---

## 1. 目的

本文件定义 ANRSM v1 的验证体系、测试层级、关键指标和验收标准。

ANRSM 不是一个“看上去挺合理”的系统。  
它必须能被验证、被基准测试、被审计，并且能在真实仓库工作流中稳定表现。

---

## 2. 验证维度

v1 SHOULD 至少验证以下五个维度：

1. **覆盖率（Coverage）**  
   纳入范围的源码中，有多少成功生成有效镜像。

2. **保真度（Fidelity）**  
   镜像是否保留了完成工程判断所需的关键语义。

3. **新鲜度（Freshness）**  
   镜像是否能在代码变化后及时进入正确状态并恢复为 `fresh`。

4. **可路由性（Routeability）**  
   系统是否能把任务正确映射到候选模块和文件集合。

5. **可阻断性（Gate Effectiveness）**  
   当发生未同步的 Material Change 时，门禁是否真的能阻止流入主干。

---

## 3. 测试层级

### 3.1 单元测试

应覆盖：

- 路径映射函数；
- 文档头解析与校验；
- 语义指纹计算；
- 状态机转换；
- 失败码输出；
- 配置解析；
- 单文件摘要生成规则。

### 3.2 适配器测试

应覆盖：

- AST 提取正确性；
- 导出符号识别；
- 依赖边识别；
- 契约识别；
- 低置信度回退行为。

### 3.3 集成测试

应覆盖：

- 初始化全链路；
- 增量更新链路；
- 漂移分析；
- 镜像写回；
- 索引更新；
- 门禁结果。

### 3.4 端到端测试

应覆盖真实仓库上的完整工作流：

1. 生成基线；
2. 修改源码；
3. 检测漂移；
4. 自动或半自动更新镜像；
5. 通过门禁；
6. Agent 使用镜像完成任务。

### 3.5 回归测试

每次变更以下对象时 MUST 补回归测试：

- 语义指纹算法；
- 语言适配器；
- 路由算法；
- 模块规则；
- 生成模板；
- 门禁策略。

---

## 4. 推荐指标

### 4.1 覆盖率指标

- `file_mirror_coverage`
- `module_mirror_coverage`
- `valid_mirror_ratio`

建议初始目标：

- P0 主语言与主目录覆盖率 >= 95%
- `invalid` 工件比例 <= 1%

### 4.2 新鲜度指标

- `fresh_ratio`
- `stale_ratio`
- `invalid_ratio`
- `mean_recovery_time_from_stale`

建议目标：

- 在常规增量链路下，变更相关工件应在一次更新流程后恢复为 `fresh`；
- 主干分支 `stale_ratio` SHOULD 接近 0。

### 4.3 漂移检测质量指标

- `material_change_recall`
- `material_change_precision`
- `false_block_rate`
- `missed_block_rate`

建议目标：

- 对种子用例的 Material Change 召回率 SHOULD 尽量接近 100%；
- 对 C0 误阻塞率 SHOULD 足够低，避免团队失去信任。

### 4.4 路由指标

- `top_k_module_recall`
- `top_k_file_recall`
- `route_precision_at_k`
- `mean_route_width`

建议目标：

- 常见任务的关键文件应能进入 Top-K 结果；
- `mean_route_width` 不应大到迫使 Agent 重新全仓扫描。

### 4.5 压缩指标

- `source_tokens / mirror_tokens`
- `mirror_token_budget_overrun_rate`

建议目标：

- 镜像相对于源码应有显著压缩收益；
- 压缩不能以丢失关键副作用和边界为代价。

---

## 5. 验证数据集设计

### 5.1 代表性样本

应选择覆盖以下情况的真实或准真实样本：

- 前端状态逻辑；
- 后端服务逻辑；
- API 契约；
- 数据库迁移；
- 配置接线；
- 测试入口；
- 高风险基础设施脚本。

### 5.2 种子变更集

验证漂移检测和门禁时，应预置种子变更集，至少包括：

1. 纯格式化改动；
2. 局部控制流变更；
3. 导出签名变更；
4. 配置键变更；
5. 事件名变更；
6. 数据模型变更；
7. 权限链路变更；
8. 路由文件移动。

这些种子样本能验证系统是否正确区分 C0/C1/C2/C3。

### 5.3 路由基准集

应建立一批真实任务描述，例如：

- 修复某错误提示；
- 调整鉴权逻辑；
- 修改某事件发送参数；
- 替换某缓存策略；
- 新增某 API 字段。

然后验证路由索引是否能召回相关模块和文件。

---

## 6. 验收标准

### 6.1 P0 验收标准

一个实现若要通过 v1 P0 验收，至少 MUST 满足：

1. 能对主语言主目录生成文件镜像；
2. 能为每个镜像生成标准头；
3. 能计算并使用语义指纹；
4. 能区分至少 C0/C1/C2 三类变化；
5. 能在变更后产出漂移报告；
6. 能在 PR 或等价场景下阻断未同步的 Material Change；
7. 能让 Agent 以索引和镜像为默认入口；
8. 能把 `freshness_state` 作为运行时信号暴露。

### 6.2 P1 验收标准

建议附加：

1. 模块镜像稳定生成；
2. 低置信度工件有明确复核队列；
3. 路由基准达到可接受精度；
4. 关键知识文档可从镜像自动编译。

---

## 7. 质量审计问题清单

上线前应回答以下审计问题：

1. 哪些文件仍无镜像，原因是什么？
2. 哪些高风险目录解析置信度仍低？
3. 门禁是否曾经被大量误阻塞？
4. 是否存在系统性漏检的 Material Change 类型？
5. Agent 是否真的读取了镜像，而不是继续直扫源码？
6. 知识层是否仍有脱离镜像独立维护的风险？

---

## 8. 人工评审要求

尽管 ANRSM 强调自动化，v1 仍应保留人工抽样评审：

### 8.1 抽样内容

- 高风险文件镜像；
- 低置信度镜像；
- 高频变更模块；
- 漂移误判样本；
- 路由失败样本。

### 8.2 抽样问题

1. 镜像是否遗漏了关键副作用？
2. 是否误把“功能描述”当成了“契约描述”？
3. 是否给出了明确的展开源码条件？
4. 是否存在空洞、模板化、没有判定价值的段落？
5. 是否存在术语不一致问题？

---

## 9. 发布门槛建议

在主仓正式启用门禁前，建议至少通过以下门槛：

1. 主目录镜像覆盖率达到目标；
2. 关键高风险目录误阻塞率可接受；
3. Agent 集成链路已跑通；
4. 漂移报告可读且可定位；
5. 维护者团队已理解失败码与修复方法；
6. 至少完成一轮真实项目试点。

---

## 10. 性能验收建议

v1 不必极端优化，但应保证在目标仓库规模下可用。

建议跟踪：

- 初始化耗时；
- 增量更新耗时；
- 单文件解析耗时；
- 索引更新时间；
- 门禁附加时间。

验收原则：

- 本地 pre-commit 不应慢到被团队系统性绕过；
- CI 不应慢到开发者倾向关闭门禁；
- 全量重建可慢，但必须可解释、可恢复。

---

## 11. 失败样本库

团队 SHOULD 建立失败样本库，记录：

- 漏检案例；
- 误检案例；
- 低置信度误导案例；
- 路由失败案例；
- 镜像质量差案例；
- 适配器退化案例。

每个样本 SHOULD 包含：

- 仓库快照或最小复现；
- 预期；
- 实际结果；
- 根因；
- 修复状态。

这是系统长期进化的关键资产。

---

## 12. 指标解释原则

指标不能脱离工程目标单独优化。  
例如：

- 过分追求压缩率会损害保真度；
- 过分追求召回率会扩大路由宽度；
- 过分追求零误报会增加漏报。

因此指标 MUST 组合解释，而不是单点最优。

---

## 13. 推荐的验收清单

### 功能性

- [ ] 文件镜像生成可用
- [ ] 模块镜像基础可用
- [ ] 索引可查询
- [ ] 漂移报告可生成
- [ ] 门禁可运行
- [ ] Agent 读取链路可运行

### 正确性

- [ ] C0 不被大量误阻塞
- [ ] C1/C2/C3 能被识别
- [ ] 高风险文件镜像无明显空洞
- [ ] `freshness_state` 可解释

### 可运维性

- [ ] 失败码清晰
- [ ] 日志可诊断
- [ ] 全量重建可恢复
- [ ] 配置可版本管理

### 可采纳性

- [ ] 本地体验可接受
- [ ] Reviewer 能读懂 diff
- [ ] 团队愿意持续使用而非绕过

---

## 14. 一句话结论

ANRSM 的验收标准不是“生成了一堆文档”，而是：  
**在真实变更和真实任务中，它能稳定生成、正确失效、正确更新、正确阻断，并实质性改变 Agent 的默认读取方式。**
