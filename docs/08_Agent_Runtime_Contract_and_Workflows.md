# 08. Agent 运行时契约与工作流

状态：Normative  
版本：v1  
优先级：P0

---

## 1. 目的

本文件定义 Agent 在 ANRSM 仓库中的默认读取顺序、任务路由方式、源码展开条件、写回义务和失败回退策略。

如果没有运行时契约，ANRSM 最终只会沦为“仓库里多了一层文档”。  
真正的价值在于：**Agent 的默认行为必须被镜像层重塑。**

---

## 2. 基本原则

### 原则 P1：先镜像，后源码

除非任务天然要求具体实现细节，Agent MUST 先读索引和镜像，再决定是否展开源码。

### 原则 P2：按需展开，不做默认全仓扫描

Agent SHOULD 只展开当前任务切片需要的源码文件。  
全仓扫描 MUST 被视为异常回退，而不是默认策略。

### 原则 P3：写源码即写镜像

当 Agent 修改受管源码后，MUST 触发对应镜像更新与验证。

### 原则 P4：低可信输入不得伪装成高可信

若镜像处于 `stale`、`invalid` 或 `unknown`，Agent MUST 降低信任等级，并在必要时展开源码或请求复核。

---

## 3. 标准读取顺序

v1 推荐 Agent 使用以下读取顺序：

1. 任务描述；
2. 路由索引；
3. 相关模块镜像；
4. 相关文件镜像；
5. 必要的相邻文件镜像；
6. 按需展开源码；
7. 测试、配置和契约文件；
8. 变更后重新读取漂移报告。

这个顺序体现了“最小充分上下文”原则。

---

## 4. 标准工作流

### 4.1 Phase 0：任务理解

输入：自然语言任务、Issue、PR 评论或缺陷描述。  
输出：任务切片假设，包括：

- 候选模块；
- 关键能力词；
- 风险域；
- 是否可能涉及契约变更。

### 4.2 Phase 1：任务路由

Agent MUST：

1. 查询 `routing_index`；
2. 召回候选模块和文件；
3. 读取前 N 个高相关模块镜像；
4. 生成初步受影响文件集合。

### 4.3 Phase 2：镜像理解

Agent MUST：

1. 读取受影响文件的 `file_mirror`；
2. 检查 `freshness_state`；
3. 理解责任、契约、状态、副作用和风险；
4. 判断是否已有足够信息制定修改计划。

### 4.4 Phase 3：源码展开

仅当出现以下任一条件时，Agent SHOULD 展开源码：

1. 需要精确修改控制流；
2. 需要判断具体实现细节；
3. 镜像显式提示必须展开；
4. 镜像新鲜度不足；
5. 涉及高风险边界；
6. 需要理解调用栈或测试断言。

### 4.5 Phase 4：实施修改

Agent 在修改源码时 SHOULD：

1. 限定修改边界；
2. 同步修改受影响测试；
3. 记录可能的影响面；
4. 尽量避免无关文件噪声改动。

### 4.6 Phase 5：写回与验证

Agent MUST：

1. 触发镜像更新；
2. 运行漂移检测；
3. 读取漂移报告；
4. 修复未同步工件；
5. 在门禁通过后才结束任务。

---

## 5. 任务路由协议

### 5.1 输入

任务路由至少接收：

- 自然语言任务文本；
- 可选的文件、模块或错误上下文；
- 当前仓库分支状态。

### 5.2 输出

路由结果 SHOULD 至少包含：

- 候选模块列表；
- 候选文件列表；
- 风险标签；
- 需要优先展开的文件；
- 可能受影响的测试；
- 置信度。

### 5.3 路由失败的定义

以下情况可视为路由失败：

1. 召回结果与任务明显无关；
2. 高影响文件未进入候选集；
3. 召回结果过宽，无法缩减任务切片；
4. 模块定位长期依赖人工猜测。

路由失败会直接导致 Agent 重新回到“大量读源码”的低效模式。

---

## 6. 上下文预算策略

### 6.1 预算优先级

对于多数工程任务，推荐的上下文预算顺序是：

1. 路由索引；
2. 模块镜像；
3. 文件镜像；
4. 关键源码片段；
5. 关键测试；
6. 其他邻域。

### 6.2 为什么不能默认读源码

默认全量读源码有三个问题：

1. Token 成本高；
2. 噪声多；
3. 重要边界被稀释。

ANRSM 的价值就在于把“先理解全仓”变成“先读结构化语义入口”。

### 6.3 何时允许扩大上下文

以下情况 MAY 扩大上下文：

- 跨模块变更；
- 大规模重构；
- 性能或并发问题；
- 漂移报告显示影响邻域广；
- 路由置信度低。

---

## 7. 对 Freshness State 的运行时解释

### 7.1 fresh

Agent 可以把 `fresh` 镜像作为默认高可信输入，但对高风险修改仍应按需展开源码。

### 7.2 stale

Agent MAY 使用 `stale` 镜像做粗定位，但 MUST NOT 仅依赖其做最终修改决策。  
通常应：

1. 展开源码；
2. 优先触发镜像更新；
3. 降低自动修改范围。

### 7.3 invalid

`invalid` 镜像不得作为行为判断依据。  
Agent SHOULD：

1. 重新生成镜像；
2. 若失败则回退到源码；
3. 将该工件标为修复优先项。

### 7.4 unknown

Agent 对 `unknown` 的处理 SHOULD 比 `stale` 更保守。  
原因是它不只是过期，而是“系统无法证明其可信度”。

---

## 8. 写回义务

### 8.1 写回触发条件

只要 Agent 修改了受管源码，以下写回义务 MUST 成立：

1. 更新对应文件镜像；
2. 必要时更新模块镜像；
3. 刷新相关索引；
4. 生成漂移报告；
5. 通过门禁。

### 8.2 写回的最小完成标准

以下条件必须同时满足，才算写回完成：

1. 变更文件不再触发 `material` 或 `blocking` 级漂移；
2. 相关镜像为 `fresh`；
3. 索引可用；
4. 失败码清零或在允许策略内。

### 8.3 不允许的结束方式

Agent MUST NOT 以以下方式结束任务：

- 代码改了，但镜像仍然 `stale`；
- 发现漂移后忽略不管；
- 仅更新时间戳不更新语义内容；
- 在 `blocking` 未消除时宣称任务完成。

---

## 9. 何时必须展开源码

即使镜像是 `fresh`，以下情况 Agent 仍 MUST 展开源码：

1. 需要精准修改算法、分支条件或异常路径；
2. 需要调整并发、事务、缓存或性能敏感逻辑；
3. 需要改动序列化、协议、迁移或安全逻辑；
4. 需要验证某段镜像描述是否遗漏具体实现约束；
5. 需要判断是否存在难以从镜像读出的副作用顺序。

### 9.1 何时可以只读镜像完成任务

以下任务 MAY 主要依赖镜像：

- 初步定位问题；
- 判断模块归属；
- 识别高风险入口；
- 估计影响范围；
- 制定阅读计划；
- 做交接和 Onboarding；
- 做初步 code review 预判。

---

## 10. 多 Agent 协作建议

v1 不要求复杂多 Agent 编排，但若采用多 Agent 模式，建议职责分离如下：

### Planner Agent

- 读取任务与路由索引；
- 输出候选模块、文件和计划。

### Executor Agent

- 展开必要源码；
- 修改代码和测试；
- 触发镜像更新。

### Reviewer Agent

- 检查漂移报告；
- 核对镜像是否反映真实变化；
- 识别遗漏邻域与风险。

关键要求：

- 三者 MUST 使用相同术语体系和相同镜像规范；
- 不得各自维护一套私有理解层。

---

## 11. 失败回退策略

当 ANRSM 运行时遇到问题，Agent SHOULD 按以下顺序回退：

1. 重新读取路由索引；
2. 重新生成或刷新相关镜像；
3. 展开直接相关源码；
4. 展开相邻调用方和测试；
5. 在必要时扩大范围到模块级源码邻域。

注意：  
回退是为了恢复任务可执行性，不是为了永久绕过 ANRSM。  
一旦任务完成，镜像状态仍必须被修复。

---

## 12. 提交说明建议

合规的 Agent 提交说明 SHOULD 包含：

1. 任务目标；
2. 修改的真相源工件；
3. 更新的镜像工件；
4. 漂移结果摘要；
5. 是否存在低置信度点；
6. 是否需要后续知识层更新。

这有助于 Reviewer 快速判断：

- 修改是否收敛；
- 镜像是否同步；
- 是否存在风险尾项。

---

## 13. Agent 的禁止行为

Agent MUST NOT：

1. 默认跳过镜像读取；
2. 在 `stale`/`invalid` 镜像基础上做大规模修改；
3. 修改源码后不更新镜像；
4. 手工篡改机器头字段来伪造 `fresh` 状态；
5. 在无验证的前提下用旧镜像指导跨模块重构；
6. 把知识层文档当作唯一输入而跳过镜像与源码。

---

## 14. 评估 Agent 是否“真正使用了 ANRSM”的标准

一个 Agent 只有在满足以下条件时，才算真正集成了 ANRSM：

1. 它的默认读取顺序以索引和镜像为起点；
2. 它会根据 `freshness_state` 调整行为；
3. 它会在修改后主动写回镜像；
4. 它会读取并处理漂移报告；
5. 它不会把镜像当作源码替代物。

如果一个 Agent 仍然习惯先扫源码，只把镜像当附带材料，那么它并未真正使用 ANRSM。

---

## 15. 一句话结论

ANRSM 的运行时契约，本质上是在强制 Agent 遵守一种新的仓库消费顺序：  
**先路由、再镜像、后源码、最后写回与验证。**
